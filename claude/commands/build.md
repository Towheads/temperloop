You are running the **build** command. Goal: drive every item in a `Plans/<date> <project> - <title>.md` plan note from `[ ]` to `[x] merged in #N` via one isolated worktree per item. Items inside a dependency level run unattended through agent → push → PR → CI-watch and park as `[m]` (merge-pending); a single batched merge gate fires at the end of each level before the next level starts. This is the **execution-lifecycle** stage of the funnel (`/triage` → `/assess --epic N` → `/build`); it owns the epic's runtime and **death** (close on last child), per [[Decisions/foundation - Triage stage and the logical-technical pipeline split]].

## Inputs

- `$1` — vault path to a plan note (e.g. `Plans/2026-05-16 stagefind - sweep follow-up.md`). Must conform to the plan-note schema (`~/.claude/plan-schema.md`) and have frontmatter `status: approved`.
- `--unattended` (optional) — auto-start mode, set when launched by the `assess` approval poll (the user approved and walked away). Skips the Step 1.5 "work all / subset / cancel" prompt and defaults to **work all pending items** (items the user pre-marked `[-]` in the doc stay skipped), and selects the Step 1.7 funnel-overlap **freeze** default (recorded, never silent). It does **not** bypass any downstream human gate — see the operating principle below.
- `--dry-run` (optional) — rehearsal mode: validate + parse, then print the full execution plan (issues that would be created, the level DAG, spike items, merge-gate count) with **zero mutation** — no status flip, no issue/epic creation, no claims, no worktrees. See Step 1.6.
- `--workflow` (optional) — selects the **per-level Workflow execution path** (`claude/workflows/build-level.mjs`, foundation #422) for Step 3's within-level loop instead of the conversational two-sweep loop. **Default OFF** — without it, the proven conversational orchestration (the two-sweep / `parallel()` fanout of Step 3, driven inline by this session) is unchanged. This is a **dual path, not a replacement**: both paths call the **same deterministic spine scripts** (`worktree.sh` / `pr.sh` / `ci-poll.sh` / `quality-gates.sh` / the board `claim.sh`) for every mechanical step, so they cannot diverge on *mechanics* — only on *orchestration* (who drives the 3a–3h loop: the conversational session, or the bounded Workflow process). Under `--workflow` the orchestrator's context stays pinned to one small `{parked, escalations}` object per dependency level regardless of how many items the level holds (the bounded-context win the Workflow re-homing buys, per [[Decisions/foundation - build workflow-spine invocation seam]]). The orchestrator still owns Step 4 (merge gate), ALL plan-note writeback, and escalation resolution under both paths — see the **Step 3 dual-path** note and the **orchestrator↔workflow boundary** below.

## Operating principles

- **Orchestrator owns the branch, push, PR, and merge.** Workers only edit + commit (in their own worktree); they never push.
- **Worker write-isolation is orchestrator-enforced, not model-discretion.** Leaving isolation to the model (the harness runs an `Agent { subagent_type: "claude" }` in the **parent cwd by default**) is **nondeterministic and has failed in practice** — workers stacked commits on one branch and leaked uncommitted edits into the parent tree. So the orchestrator **pre-creates a per-item worktree at a deterministic path** (`workflows/scripts/build/worktree.sh create <repo-root> <item.slug>` — path `<repo>.wt/<item.slug>`, branch `build/<item.slug>`, based on `origin/<default>`) and hands the worker its **explicit absolute worktree-root prefix** — the worker creates nothing and owns nothing about isolation (**Option A**). A **PreToolUse guard hook** (`claude/hooks/build-worktree-guard.sh`, matcher `Bash|Edit|Write|MultiEdit`, self-armed by the per-worktree `.build-guard` marker that `worktree.sh create` drops in the worktree root) then **structurally rejects** any file-tool write — or any destructive **Bash** verb (`rm`/`rmdir`/`mv`/`shred`/`truncate`/`dd of=`) — that resolves outside that root (**Option B**), so a stray parent-root path can no longer leak and worker Bash can no longer `rm -rf` its way out of the jail (F#932). The orchestrator stays in the **parent checkout** and owns the branch via **push-by-SHA** from the pre-created worktree (`git push origin "$(git -C <worktreePath> rev-parse HEAD)":refs/heads/<item.branch>` — the shared object store makes the worker's commit reachable). No `Agent { isolation: "worktree" }`. See Step 3 for the full flow and re-spawn handling. Failure history (#17 stacked commits, #10 parent-leak), the marker-vs-env-var arming (#171), the two registration surfaces (#72), and Options C (per-worker parent-clean assertion) / D (unimplemented upstream harness write-jail FR): [[Decisions/foundation - build worker write-isolation (worktree failure history + Option A-D)]].
- **Classifier-denial recovery is parent-side, never a retry (unattended runs).** When the auto-mode permission classifier denies a **worker or build-spine git operation** on an `--unattended`/funnel run — a `pr.sh` push/rebase, a force-push, or a merge/close of a PR the agent didn't author — do **NOT** retry it or force past the denial: a retry burns cycles and risks denial escalation (this recurred across the Epic C–F unattended runs). The orchestrator instead **completes the step itself, parent-side, from the worker's journal verdicts** — inspect the worktree state directly, **push by SHA** from the pre-created worktree (as 3f already does), **open the PR itself**, and **comment on (never close)** another agent's PR. The load-bearing de-obfuscation that keeps the classifier from denying the spine steps in the *first* place is the pre-resolved **`spineBinDir`** hand-off (Step 3 `spineBinDir` bullet — passing an absolute spine path instead of a nested `$(dirname "$(readlink -f …)")` command-substitution the classifier read as an obfuscated-command bypass, temperloop#72); this recovery playbook is the fallback for the residual denials de-obfuscation doesn't pre-empt. The root-cause defect for over-broad auto-mode denials is tracked separately (foundation#826). See [[Patterns/foundation - Classifier-denial recovery in unattended builds]].
- **Within a level, items run concurrently (independent by construction); merges stay gated.** Honor `depends-on`/`after` edges strictly for *level assignment* (see Step 1.4), but within a level use the two-sweep / `parallel()` fanout of Step 3 (gated by a width knob; `--serial` forces sequential). The merge gate (Step 4) remains the single human checkpoint.
- **Merge gate is batched per dependency level**, not per item. All items in a level run unattended through CI-green, then a single human gate at the end of the level approves the merge set. This lets the run proceed without constant monitoring.
- **Status sentinels go in-band on the plan note** so a future session can resume:
  - `[ ]` untouched
  - `[~]` in-progress (worker active, or PR opened but CI not green)
  - `[m]` merge-pending (PR open, CI green or no CI configured, parked for batch gate)
  - `[x]` merged
  - `[v]` verdict-captured (`kind: spike` — note written + issue routed; no PR)
  - `[-]` skipped
- **Worker self-verifies against `acceptance:` before returning `done`; the orchestrator then confirms *activation* — that a `class: A` item's built output is wired into the running path, not merely correct (Step 3e.6).**
- **Blocked / failed items stay synchronous.** A worker that returns `blocked` halts the level for an immediate clarification; a worker that returns `failed` surfaces retry/skip/abort. Only the merge gate is batched, because merges are the only step that has no further work to drive.
- **A design fork is a first-class halt that keeps implementation in the worker.** When a worker discovers mid-build that ≥2 viable designs exist and choosing between them changes the item's *contract or approach* (not merely a missing fact it could be told — that's `blocked`), it returns `design-fork` rather than guessing or quietly forcing the run into a hand-driven main-session build. The orchestrator surfaces the fork via `AskUserQuestion`, **captures the verdict durably** (a `notes:` stamp on the plan item, or a `Decisions/` note for an architectural call), and **re-spawns the same worker against the same worktree** with the verdict appended — so the design call is ratified by a human while the *build* stays delegated and the orchestrator stays lean. This is the seam that stops a design-laden epic from silently degrading into a context-blowing manual build across sessions. See [[Mistakes/foundation - build design-fork drops to manual, orchestrator context blowup]].
- **Branch names come from the plan's `branch:` field.** Do not derive — the plan is authoritative. The plan's branch values conform to [[Decisions/foundation - Branch naming convention]].
- **`--unattended` controls the *start*, not the *gates*.** When auto-started by the `assess` approval poll, build skips the initial subset prompt (Step 1.5, working all pending items) and selects the Step 1.7 funnel-overlap **freeze** default (both start-time defaults, both recorded — Step 1.5 to pending-decisions, Step 1.7 to `## Questions` + pending-decisions); it never changes which gate regime Step 4 selects. The downstream gates run in two distinct regimes:
  - **Merge gate (Step 4) is regime-selected, not unconditionally modal.** A clean, disjoint, independent set is **timed** — it auto-merges after the `BUILD_MERGE_GATE_WINDOW` window **unless an operator objects**, where absence of objection = consent (the walk-away case). A **structurally risky** set (Step 4b) is **modal** — it hard-blocks on an `AskUserQuestion` for explicit approval and is *never* timed-auto-merged.
  - **Blocked / design-fork / failed halts (3d) are always modal** — they issue an `AskUserQuestion` and wait for the user, in every run, with no timed variant.
  - **The keystone-spike review halt (4e-keystone) is never bypassed.** When a `keystone: true` spike captures its verdict this run (temperloop#526), the run **halts before dependent levels build** so the operator reviews the contract-reshaping verdict. It is `blocking-now` with **no safe default** and no timed variant — on `--unattended`/`FUNNEL_OPERATOR_ABSENT=1` it parks the plan (`status: executing`) and posts a verdict-review decision issue on the epic rather than auto-proceeding. Re-running `/build` after review is the operator's approval; the approval poll authorizes the *start*, never this mid-run gate.
  (The prior "merging into protected `main` is never automated" invariant is **deliberately superseded** by the timed gate — see [[Decisions/foundation - Timed merge-gate auto-merge (supersedes never-auto-merge-main)]].) See also [[Decisions/foundation - Approval-poll handoff for batch workflow]].
- **Non-blocking in-run decisions defer to the plan note's `## Questions` section, not an interrupting prompt.** A **`batch-at-gate`** decision (per `[[Context/foundation - AskUserQuestion severity taxonomy]]`) — a non-blocking choice that arises *during an active run* and carries an obvious **default** (Step 2.5 issue creation, Step 2.6 epic creation, the Step 0.5 reconcile auto-fix offer, a Step 4c per-PR conflict disposition, a Step 4b managed-merge EJECTED disposition) — does **not** issue an interrupting `AskUserQuestion` mid-level. It **appends one entry to the plan note's `## Questions` section** (the `## Questions` schema in `~/.claude/plan-schema.md`: question + bold `default:` + a one-line `auto-proceed:`), then proceeds on that default. The whole batch is surfaced as ONE question at the next **level merge gate** (Step 4), before merge consent is recorded — in the **timed** regime an entry still `[ ]` at window-elapse takes its default (same auto-proceed as the timed gate's absence-of-objection consent); in the **modal** regime the entries are shown alongside the merge `AskUserQuestion`. **An unanswered entry at gate resolution takes its default — never a silent stall** — which is what makes a non-blocking question safe on an unattended run instead of hanging it. This is **disjoint** from the `blocking-now` halts (3d blocked/design-fork/failed, 3a claim conflict, the 4b risky-set modal): those have *no* safe default and MUST keep interrupting, every run. See [[Context/foundation - AskUserQuestion severity taxonomy]] and `~/.claude/plan-schema.md` § Orchestrator-written `## Questions` section.
- **Every `blocking-now` human-gate ask routes through the `decision_sink_ask(...)` seam.** The orchestrator never calls `AskUserQuestion` directly at a gate; it calls the documented seam **`decision_sink_ask(question, options, severity)`** — *question* = the prompt text to surface, *options* = the offered choices (each a label + its trade-off/consequence, and which is the default/recommended one), *severity* = the taxonomy class (`blocking-now` for every gate enumerated below; per [[Context/foundation - AskUserQuestion severity taxonomy]]). The seam has **two wired backends, selected by operator-presence and severity:**
  - **Modal backend (default — operator-present runs):** `AskUserQuestion` with the inputs as given — byte-for-byte the same interactive prompt and result as before. This is the default for every run that does not explicitly set the operator-absent driver flag. An operator-present run is **never** affected by the async backend; its call-sites behave identically to before this backend was added.
  - **Async decision-issue backend (operator-absent + `blocking-now` only):** selected when the run carries the **operator-absent driver flag** (a `--unattended` flag or an equivalent `FUNNEL_OPERATOR_ABSENT=1` env signal set by the `funnel-tick` cron) **AND** `severity == blocking-now`. The backend: (1) posts a comment on the item's GH issue with the question text, the offered options, and the typed-reply grammar (fenced ` ```decision ``` ` block or `/choose <label>` / `/approve` shorthands, per `~/.claude/decision-queue-contract.md` § 3); (2) applies the `decision` label to the issue (`gh issue edit <n> -R "$REPO" --add-label decision`); (3) assigns the issue to the operator (`gh issue edit <n> -R "$REPO" --add-assignee "$ASSIGNEE"`, where `$OPERATOR` = the repo's configured operator handle read from `FUNNEL_OPERATOR` env, default `@example-operator`, and `$ASSIGNEE` is `$OPERATOR` with a leading `@` stripped — `ASSIGNEE="$OPERATOR"; [ "$ASSIGNEE" = "@me" ] || ASSIGNEE="${ASSIGNEE#@}"` — because GitHub's `replaceActorsForAssignable` cannot resolve an `@`-prefixed login like `@example-operator` (foundation #977; the special `@me` token is preserved, gh resolves it to the authenticated user)); (4) parks the plan item (`[?]` in the plan note) and stops processing this item for this tick. **No auto-default is ever taken** — the item stays parked until the operator replies, unassigns, and the drain step picks it up. The `decision` kind label and assignee-baton semantics follow `~/.claude/decision-queue-contract.md` exactly.
  - **Selection rule (load-bearing, non-negotiable):** `if operator_absent AND severity == blocking-now → async backend; else → modal backend`. The two batch-severity surfaces (`batch-at-gate` → `## Questions` with gate-elapse default; `batch-at-ritual` → `pending-decisions.md` + `check-in`) are **not routed through this seam** and are **not affected** — severity remains the routing axis, operator-presence is only the within-`blocking-now` tie-breaker. An operator-present run behaves **byte-for-byte identically** to before this backend was added.
  - **v1 exclusion — `design-fork` on an operator-absent run does NOT use the async backend.** A `design-fork` result holds a live worktree and requires a re-spawn against the same worktree after the operator answers; that continuation cannot survive a cron gap without persisting the worktree indefinitely. So: on an **operator-absent run**, a `design-fork` outcome **parks the item back to Ready** (status → Ready via board adapter), **discards the worktree** (removes the `.build-guard` marker and cleans the worktree path), stamps a `notes:` entry with the fork's `design_fork.decision` and `design_fork.options` for the next claim to see, and **stops** — no decision issue is opened, the fork is not handed to the async backend, and the item will be re-spawned fresh on the next claim. See the 3d bullet below for the concrete operator-absent design-fork path.
  **Load-bearing invariant: the seam carries ONLY the ask.** Every durable side-effect around a gate — the `## Design verdict` / `## User answers` append, the `escalated: true` stamp, the verdict-block writes, the post-announce objection scan, the deferred `## Questions` (4f) drain, the merge incantation — stays **OUTSIDE** the `decision_sink_ask(...)` call and in its **existing order** relative to the call. The seam is a pure ask: inputs in, the operator's choice out (modal) or void (async, parks and stops); it writes nothing and scans nothing. The gates that route through it: **3a** claim-conflict, **3d** blocked / failed (and design-fork on operator-present runs only — see v1 exclusion above), **3d-esc** continuation (per escalation kind), **4b** risky-set merge approval + the timed-path objection fallback. (**3e** pre-push review carries *no* human-gate ask — its blocking findings auto-loop to 3c — so there is nothing to route there. **Step 1.5**'s subset prompt is a `batch-at-ritual`-adjacent *start* question with a safe default, **not** a `blocking-now` gate, so it keeps its direct `AskUserQuestion` and does **not** route through the seam.) `/assess` Step 6 (approval poll) routes its one ask through the same seam.
- **Board integration is claim-first, and only when the project provides it.** When the shared board toolkit is present and this repo is a registered board (the Step 0 probe resolves the adapter `lib/board.sh` plus a board number — stageFind = project 3, foundation = project 4, owner `<org>`), build mirrors the plan onto the board: at start it ensures every worked item has a GH issue (Step 2.5 creates one when `gh_issue:` is missing), **claims** each item In Progress as the *first* per-item action — before the worker investigates, per [[Decisions/stageFind - Task workflow (board + WIP-3 gate)]] — **best-effort** clears the tmux claim marker when the item parks (the board claim is **held until Done** — 3h step 2, K#275 — so the marker clear is optional, never a required park step), and moves the item to **Done** on merge. A failed or contended claim **halts** that item (retry/skip/abort) — build never works an unclaimed or another session's item. All board reads/writes go through the adapter (`board_resolve` / `board_resolve_item` / `board_item_id` / `board_set_status` / `board_capture_item`), never a hardcoded project number or hand-resolved field/option id. **Single-item board ops use `board_resolve_item "$BOARD" <issue#>`, not `board_resolve`** — it resolves just the one issue's item via a targeted GraphQL lookup instead of pulling the whole-board `item-list --limit 200`, which drained the Projects-v2 budget in a per-item burst (GH #53). Reserve the full `board_resolve` for whole-board scans (the Step 0.5 reconcile). Repos with no adapter, or that aren't a registered board, run exactly as before; every claim/release/issue-creation step is skipped. See [[Decisions/foundation - build board integration]].
- **External gates lift by predicate, never by issue-closed-state inference.** An item gated on work *outside this plan* (a cross-plan leg, an unplanned upstream issue) carries its gate as prose in `notes:` ("don't start until #N lands") plus a machine-checkable `gate_check:` predicate on the **consumable, not the tracker** (`/assess` emits both; plan-schema rule 11 requires the predicate whenever the prose gate is present). **Any gate-lift or re-open decision — Step 3a's claim gate (3a-0) and the Step 0.5 reconcile re-open path — MUST run the `gate_check:` predicate and gate on its result; it must NEVER declare a gate lifted because the gated-on issue is closed.** *"Issue closed" ≠ "dependency consumable exists"* — an issue can close one half of its work while deferring the half this item needs (the ELT #492 false gate-lift: *"#380 CLOSED → roster gate lifted"* was wrong because #380 deferred product wiring; `configs/artists.toml` still listed 3 of 40 artists — costing the run's most expensive replan). Run the predicate against the consumable; infer nothing from the tracker. See 3a-0 and `~/.claude/plan-schema.md` § Optional `gate_check:` field.
- **5-hour quota gate — pause at the level boundary, auto-resume after reset.** After each level's merge gate, before advancing (Step 4e-quota), the run checks the remaining 5-hour usage quota via `quota-gate.sh` (which reads the snapshot `status-line.sh` persists to `~/.claude/rate-limits.json`). If the remaining quota is below `BUILD_QUOTA_PAUSE_PCT` (set in `workflows/scripts/build/build.config.sh`), the run **pauses at that clean boundary and auto-resumes after the window resets** — a background `sleep <wait_secs>` (the duration `quota-gate.sh` emits, = `reset_ts − now + buffer`), then re-check and continue — rather than getting cut off mid-merge in the next level. The signal **fails open**: a missing/stale/unreadable snapshot never pauses a run. This is a `blocking-now`-adjacent *self-throttle*, not a human gate — it interrupts nothing and asks nothing; it only defers the *next level's* work to after reset.
- **Epic lifecycle (board ON + plan has `epic:`): build owns runtime + death, not birth.** On the triaged path the epic already exists (`/triage` created it, `/assess` wrote `epic:` into the plan) and its children are already linked as sub-issues — so Step 2.6 is a **backstop**, creating an epic only for a plan that did *not* come through triage. At runtime build moves the **epic → In Progress** when it claims the *first* child (3a), and **closes the epic → Done** when its **last open sub-issue** closes (4d). Epic close is **data-driven** (open-children == 0), never "this plan finished" — so it is correct for partial-coverage plans and for children closed by triage-cull or by hand. See [[Decisions/foundation - build epic creation at run-start]] (epic creation now demoted to a backstop) and [[Decisions/foundation - Triage stage and the logical-technical pipeline split]].

## Step 0 — Validate

Run in parallel:

1. `git status --porcelain` on the parent checkout, then branch by operator presence. **Attended run:** if dirty, don't blanket-abort — offer the operator **stash → build → restore** (stash the parent's changes, run the build, restore them afterward), so their in-progress work is out of the way but never discarded. **Unattended run:** abort and record the item `failed` — never stash, reset, or otherwise touch the operator's own uncommitted work when no one is watching. Either way the orchestrator does not *reason about* the uncommitted changes; it either sets them aside intact (attended) or refuses to run (unattended).
2. `gh auth status` — needed for PR creation and merge.
3. `mcp__obsidian__get_vault_file` on `$1` — confirm plan note exists. (`$1` and other document paths cited throughout this file — `Plans/`, the priorities note (`Projects/<project>/Priorities.md`, falling back to the legacy `Priorities/<project>.md`), the pending-decisions surface (`Pipeline/pending decisions.md`, falling back to the legacy `Context/pipeline - pending decisions.md`) — path fallback convention, `claude/commands/check-in.md` — are relative to **the knowledge store root**, resolved per `workflows/scripts/lib/knowledge_store.contract.md`. The agent-plane transport stays on Obsidian MCP tools per that contract's Obsidian-mode note.)
4. Read project `CLAUDE.md`; extract `## Branch policy` and `## Subagents` sections verbatim for passing to workers. Also extract the `## PR verification surface` rule — it lives in the global `~/.claude/CLAUDE.md` (source: `claude/CLAUDE.md`), with a project `CLAUDE.md` taking precedence if it defines its own.
5. **Board-integration probe.** Canonical predicate: [[Decisions/foundation - Project capability probes]] — a project is **board-wired** iff the shared board adapter resolves **and** the repo maps to a registered board number. The operational resolution mechanics follow; every board step goes through the adapter, never a hardcoded project number or a hand-resolved field/option id.
   - Set `BOARD_LIB` = the first of `scripts/lib/board.sh` (a consuming repo like stageFind, which vendors the toolkit) or `workflows/scripts/board/lib/board.sh` (foundation, the toolkit's home) that exists. Set the claim/release entrypoints `CLAIM`/`RELEASE` = the `claim.sh`/`release.sh` next to `BOARD_LIB` (i.e. `$(dirname "$BOARD_LIB")/../claim.sh`), or the `claim`/`release` commands on `PATH` (put there by `make install-board` in the foundation fleet — a kernel checkout has no such target and resolves the scripts next to `BOARD_LIB` as above).
   - **Adapter found** → resolve the board number from the adapter's own registry, so this works on any registered board with no hardcode:
     `source "$BOARD_LIB"; repo="$(gh repo view --json nameWithOwner -q .nameWithOwner)"; BOARD=""; for b in $(board_registered_boards); do [ "$(board_repo "$b")" = "$repo" ] && BOARD="$b"; done`
     (the candidate list is the adapter's own `board_registered_boards` — the single source of truth for the registered set: stageFind=3, foundation=4, ssmobile=5, subsetwiki=6, temperloop kernel=7 (issues-only), plus any `boards.conf`-registered board. Onboarding a board needs no edit here — it flows from `board_repo()`'s map. Mirrors `/triage` Step 3's identical reverse-lookup. Iterating a hardcoded `3 4 5 6` literal here is exactly what dropped board 7 and left `/build` board-OFF on the kernel's own tracker — temperloop#352.)
   - **`BOARD` resolved** → board integration is **ON**. **The `project`-scope check is backend-conditional** (`board_backend "$BOARD"`): a **Projects-v2 board** (`board_backend "$BOARD"` ≠ `issues` — boards 3-6) makes its board edits through the Projects-v2 GraphQL API, which needs the `project` gh scope — verify `gh auth status` lists it (tell the user to run `gh auth refresh -s project` and stop if missing). An **issues-only board** (`board_backend "$BOARD"` = `issues` — the temperloop kernel tracker, board 7) drives Status / claim-stamp / Done / mirror entirely through **plain-REST label writes + issue-close + the sub-issues API** (`ISSUES-ONLY-BACKEND.md`), all covered by the default `repo` scope — so **skip the `project`-scope check** for it. Requiring `project` here would wrongly halt a board-7 run whose token carries only `repo`, even though every board step is backend-agnostic and works without it (temperloop#391). Record `BOARD_LIB`, `BOARD`, `CLAIM`, `RELEASE`; **`source "$BOARD_LIB"` at the top of every board bash block** to get `board_resolve` / `board_resolve_item` / `board_item_id` / `board_set_status` / `board_capture_item` / `board_stamp` / `board_backend` and the `BOARD_OPT_*` constants. Pass `--board "$BOARD"` to `CLAIM` so the claim targets the right board on any repo (it defaults to 3).
   - **No adapter, or no registered board matches this repo** → board integration is **OFF**: **board integration unavailable — claim/mirror steps skipped** (2.5, 2.6, the 3a claim, the 3h release, the 4d Done move + epic-close). The run behaves exactly as the pre-board version.
6. **Source the batch-pipeline config + resolve the quota gate (best-effort).** `source workflows/scripts/build/build.config.sh` (bare repo-relative, the same convention the spine scripts use — `worktree.sh`/`pr.sh` etc.) — the worked example of the kernel's Step-0 config-sourcing convention (`~/.claude/CLAUDE.md` § Prose-resident knob convention). This pulls the central `BUILD_*` defaults into scope — the merge-gate window, the queue timeout, and the **5-hour quota-gate** knobs (Step 4e-quota) — with any pre-set env value still overriding. Then resolve the gate script and **guard its existence**: `quotaGate="workflows/scripts/build/quota-gate.sh"; [ -x "$quotaGate" ] || quotaGate=""`. If the config/script isn't found (a consuming repo that doesn't vendor the toolkit), proceed: the prose keeps its inline `${VAR:-default}` fallbacks, and an empty `$quotaGate` makes Step 4e-quota a no-op (proceed) — the **optional** quota gate must never be the thing that breaks or stalls a run.
7. **Build-spine staleness check (read-only, no-op when not a symlink).** A consuming repo's `workflows/` directory is often a dev-local symlink into a separate foundation checkout — whichever branch that checkout is on becomes the live build-spine for this run. If the spine is stale (foundation on a feature branch or behind `origin/main`), every spine script (`worktree.sh`, `pr.sh`, `ci-poll.sh`, etc.) may be missing merged fixes, with no visible signal. This check makes staleness loud. **It is strictly read-only and must never mutate, fetch into, or check out the foundation checkout.**

   ```bash
   # Resolve the symlink target, then walk up to find the foundation repo root.
   spineLink="workflows"
   if [ -L "$spineLink" ]; then
     spineTarget="$(readlink -f "$spineLink")"   # absolute, canonical
     # Walk up from the symlink target to find a git root that is NOT the current repo.
     foundationRoot=""
     probe="$spineTarget"
     currentRoot="$(git rev-parse --show-toplevel 2>/dev/null || true)"
     while [ "$probe" != "/" ]; do
       if [ -d "$probe/.git" ] && [ "$probe" != "$currentRoot" ]; then
         foundationRoot="$probe"
         break
       fi
       probe="$(dirname "$probe")"
     done
     if [ -n "$foundationRoot" ]; then
       spineBranch="$(git -C "$foundationRoot" symbolic-ref --short HEAD 2>/dev/null || echo "(detached)")"
       # Count commits behind without fetching — compare against the already-known remote ref.
       spineBehind="$(git -C "$foundationRoot" rev-list --count HEAD..origin/main 2>/dev/null || echo "unknown")"
       if [ "$spineBranch" != "main" ] || [ "$spineBehind" != "0" ]; then
         echo "WARNING: build-spine symlink (workflows/ -> $foundationRoot) is STALE"
         echo "  branch:        $spineBranch  (expected: main)"
         echo "  commits behind origin/main: $spineBehind"
         echo "  Spine scripts may be missing merged fixes. Switch foundation to main and pull before building."
       fi
     fi
   fi
   # If workflows/ is not a symlink, or the symlink target is within the current repo,
   # the check is a no-op — the spine is the repo itself (e.g. running inside foundation).
   ```

   When no warning fires, proceed silently. This check never aborts the run — it is advisory only.

**Merge incantation (canonical).** `gh pr merge <pr> --merge --delete-branch` — `--merge` preserves the repo's `--no-ff` history convention; `--delete-branch` cleans the remote. This is the single source of truth for the merge strategy *on the MANAGED backend* (the 4a `gate.sh backend` verdict): `gate.sh managed-merge` runs exactly this incantation internally — **not** `--auto` — after establishing mergeability itself (its `--strict` arm folds an `update-branch` + SHA-pinned CI re-poll in front of it; `--non-strict` merges directly). The batch merge gate (Step 4b, "Merge — MANAGED backend") references it.

A repo with GitHub's **native merge queue** (the 4a `gate.sh backend` probe returning `NATIVE`) uses a **different, smaller** incantation: **`gh pr merge <pr> --auto`** — no `--queue`, no `--merge`, no `--delete-branch`. Rationale (verified on **gh 2.93.0**): `--queue` **does not exist** as a flag (it errors `unknown flag`; the gh 2.93 `pr merge` flags are `--auto`/`--disable-auto`/`--admin`/`--merge`/`--rebase`/`--squash`/`--delete-branch`), and on a queue-required branch the **queue owns the merge strategy** (the ruleset's `merge_method`, so `--merge` is ignored) and the **branch lifecycle** (so `--delete-branch` is rejected). `--auto` arms auto-merge, which enqueues the PR once its required checks pass — the build case, where a parked PR may not yet be green. So the NATIVE path changes *both* when GitHub lands it *and* drops the strategy flags (the queue supplies them).

If any check fails, surface in one line and stop.

## Step 0a — Tick-start plan-approval drain (operator-absent / cron runs only)

**Runs only when the operator-absent flag is active** (`FUNNEL_OPERATOR_ABSENT=1` env signal or equivalent cron signal) **AND no plan-note path `$1` was given** — i.e. the tick is deciding what to work, not resuming a specific plan. Skip this step entirely on operator-present runs and on any run where `$1` is already set. **This is the epic-carried plan-approval handoff that replaces the `ScheduleWakeup` poll for cron ticks.** The `assess` Step 6 async path posts the poll offer on the plan's epic issue and stops; this step picks it up at the next tick.

**Why this exists (re-homed, not inherited):** The `ScheduleWakeup` poll in `assess` Step 6 runs inside a single live session — it cannot span cron-tick gaps. On an operator-absent run the async backend posts the approval offer on the epic issue and stops (no `ScheduleWakeup` is armed). The drain step below is what makes the approval-poll work across ticks: it runs at tick start, finds answered approval issues, and drives each approved plan into `/build --unattended`. This is an **additive, cron-path-only drain** — it does not modify the operator-present behavior of `assess` Step 6 in any way.

**Procedure:**

1. **Query answered plan-approval decision issues.** Run:
   ```sh
   gh issue list -R "$REPO" \
     --label decision \
     --state open \
     --assignee "" \
     --json number,title,body,comments
   ```
   Filter the results to those whose body contains the magic marker line `plan-approval-poll: [[Plans/` (the marker the async backend stamps into the issue comment body — see `assess` Step 6, async path: the comment body must include this marker followed by the vault path of the plan note, e.g. `plan-approval-poll: [[Plans/2026-06-20 foundation - my plan]]`). A decision issue without this marker is a different kind of decision (e.g. a blocked/failed item) and is **not** drained here — leave it in the queue for the appropriate drain step.

2. **For each matched issue, apply the contention pre-check.** Re-read the issue's current assignee count (`gh issue view "$N" -R "$REPO" --json assignees --jq '.assignees | length'`). If non-zero, another tick already re-assigned it — skip this tick.

3. **Parse the operator's reply** (the most recent comment on the issue, per `claude/decision-queue-contract.md` § 3):
   - A `decision` block with `chosen: approve` (or `/approve` shorthand) → approved.
   - A `decision` block with `chosen: skip` (or `/choose skip`) → declined; the plan stays draft.
   - A `/hold #N` → defer; leave the issue in the queue.
   - Any other comment or parse miss → follow the contract's parse-miss rule (re-assign to operator with a "couldn't parse" comment; skip this tick).

4. **Apply the answer:**
   - **Approved:** (a) Remove the `decision` label — `gh issue edit "$N" -R "$REPO" --remove-label decision`. (b) Post a confirmation comment on the issue: "Decision applied: plan approved. Invoking `/build --unattended`." (c) Extract the vault path from the marker line (`plan-approval-poll: [[Plans/<path>]]` → `Plans/<path>`). (d) Invoke `/build Plans/<path> --unattended`. **The `/build` invocation replaces the `ScheduleWakeup` poll's role — it is the continuation of the `assess` Step 6 approval-poll handoff, fully re-homed onto the decision-issue backend.**
   - **Declined (skip):** Remove the `decision` label; post a confirmation comment: "Decision applied: plan not approved — skipping." The plan note remains `status: draft`.
   - **Deferred (/hold #N):** Leave the issue as-is (label + no assignee); skip for this tick.

5. **One plan per tick-start drain.** If multiple answered plan-approval issues are found, process them in issue-number order, one per tick. This prevents a single tick from launching unbounded concurrent builds (the funnel's `FUNNEL_DRIVE_CONCURRENCY` governor bounds per-tick drive concurrency, and the per-item claim-first lock governs concurrency once items are claimed). Log which issues were found and which was processed.

## Step 0.5 — Reconcile state (resume only)

Runs only when the plan's frontmatter `status` is `executing` at start (i.e. a resume). Cross-checks the four state stores that can diverge under a crash — plan-note sentinels, git/remote (worktrees, branches, open PRs), the board, and the workflow journal — and surfaces drift **before** parsing levels. **Report-first, fix-on-confirm:** never silently mutate outward state.

1. **Worktrees.** `git worktree list` — flag `<repo>.wt/<slug>` worktrees (on `build/<slug>` branches) not owned by an in-flight (`[~]`) item (offer pruning via `workflows/scripts/build/worktree.sh prune <repo-root>` — removes only `<repo>.wt/*` worktrees whose branch is merged into `origin/<default>`, marker included, skipping dirty worktrees unless `--force`, one structured outcome per worktree; each path maps back to a plan item by slug).
2. **PRs vs sentinels.** `gh pr list --state open --json number,headRefName` cross-mapped to plan `branch:`/`pr:` values — flag open PRs with no `[m]`/`[~]` item, and `[~]`/`[m]` items whose `pr:` PR is closed or gone. **Speculative-worker sentinel:** a `[~]` item carrying `  - speculative: true` and **no `pr:`** is a *held speculative worker* (worktree built, awaiting the 3f base-currency check), **not** a stuck worker to reset — its worktree holds real build work that a reset would discard. Route it to the **3f step-0.5 base-currency check** (rebase-then-reverify, or discard-and-respawn on conflict), not a re-spawn. Its worktree (flagged in item 1) is therefore *expected*, not orphaned — do not offer to prune it.
3. **Board vs sentinels (board ON).** Read the board via the adapter (`source "$BOARD_LIB"; board_resolve "$BOARD"`, then inspect `BOARD_ITEMS_JSON` — each item's `.status` and `.["host/Session"]`) — flag In-Progress cards for `[x]` items (should be Done) and Done cards for non-terminal items. **Self-claim recovery:** an item In Progress under *this host* but a *prior* `Host/Session`, with no live tmux marker, is a reclaimable dead-session self-claim — re-claim it rather than treating it as a foreign conflict (cf. 3a). **Epic drift:** if the plan has `epic:`, flag an epic that is closed while a non-terminal child remains (re-open it), or open with zero open children (the 4d close was missed — close it on confirm).
4. **Workflow journal vs sentinels (intra-level recovery).** The Workflow runtime journals each `agent()` call as `agent-<id>.jsonl` in the transcript directory; a crash mid-level leaves that journal as the only record of intra-level progress the plan note had not yet been updated to reflect. Consult it after the three stores above, using the following **authority ordering**:

   | Priority | Store | Scope of truth |
   |---|---|---|
   | 1 (highest) | **Plan-note sentinels** (`[ ]`/`[~]`/`[m]`/`[x]`/`[v]`/`[-]` + `pr:`/`pushed_sha:`) | Durable truth **at level boundaries** — written by the orchestrator on each state transition; survives any crash. |
   | 2 | **Workflow journal** (`agent-<id>.jsonl` in the transcript dir) | Truth for **intra-level** progress of the **currently-running level only** — recovers facts (e.g. a `pr:` number) the orchestrator had not yet stamped to the plan note before a crash. |
   | 3 (lowest) | **Git + board** | External cross-check (stores 1–3 above). |

   **Invariant — journal NEVER overrides `[x]`/`[-]`:** a human merge (`[x]`) or human skip (`[-]`) sentinel on the plan note outranks any intra-level journal entry. A journal record showing a PR was opened or pushed does NOT let you flip an `[x]` back to `[~]` — that transition is one-way and human-owned. This is the authority ordering's load-bearing guarantee.

   **Crash-quantization.** A crash between levels lands in a clean reconcile state: no intra-level journal progress to recover (the prior level's sentinel writes are durable, the new level's journal is empty). A crash mid-level leaves the currently-`[~]` level's journal with per-`agent()` entries the orchestrator can mine. Net effect: recovery either sees nothing intra-level (clean) or sees the structured journal for exactly one active level (recoverable).

   **Concrete reconcile step.** On resume, if the currently-`[~]` level has a workflow journal (the transcript directory is the Workflow runtime's per-run transcript dir, surfaced in the `Workflow` launch result as `Transcript dir: …` and stable across a `resumeFromRunId`; the `agent-<id>.jsonl` files live under it):
   - Read the journal entries for any `pr.sh open` executor-agent calls that returned `{outcome:"PR_OPENED", pr_number, url}` (per the `runSpine` contract — each spine call is its own `agent()` entry, so PR-open is a discrete journalled outcome, not buried mid-sequence).
   - For each such entry whose item **does not yet carry `pr:` in the plan note** — and whose sentinel is still `[~]` (not `[x]`/`[-]`) — **recover the `pr:` pointer**: stamp it via the deterministic spine writer `workflows/scripts/build/plan.sh writeback <plan> --slug <slug> --sentinel '[~]' --pr <N>` (the same sub-line writer 3f uses — no `patch_vault_file` em-dash-heading misfire on the item's sub-lines). This lets Step 1.4's `[~]`-with-`pr:` path re-attach to the open PR instead of re-spawning a worker.
   - Similarly recover `pushed_sha:` from a `pr.sh push` journal entry (`{outcome:"PUSHED", sha}`) for any `[~]` item that has no `pushed_sha:` yet — add `--pushed-sha <SHA>` to the same `plan.sh writeback` call. This anchors the CI re-poll to the correct SHA.
   - **Do NOT touch any `[x]`/`[-]` item** — the invariant above is absolute; the journal cannot reverse a terminal sentinel.
   - Include the journal recovery (items recovered, pointers stamped) in the Step 6 summary.

5. Emit one reconciliation report; the offer to auto-fix the **safe** divergences (adopt an orphan PR's number, prune a dead worktree, move a stale board card, close an all-children-closed epic) is a **`batch-at-gate`** decision — **append it to the plan note's `## Questions` section** rather than issuing an interrupting `AskUserQuestion`, then proceed on its default. Entry: `` `step: 0.5` — Auto-fix the safe reconcile divergences (<the specific safe fixes>)? **default: don't auto-fix (report only)** ``, with an `auto-proceed:` sub-line (unanswered → leave the divergences as reported; nothing is mutated). The batch surfaces at the next level merge gate (Step 4). Never auto-close issues or rewrite human merge decisions — the conservative default (report only) is what keeps a deferred reconcile from silently mutating outward state on an unattended run. **A gate-lift / re-open decision here is bound by the gate-predicate rule (operating principles, 3a-0):** if reconcile considers re-opening or un-parking an externally-gated item whose `gate_check:` reads **false**, do NOT treat the gated-on issue being closed as evidence the gate lifted — *"issue closed" ≠ "dependency consumable exists"* (the ELT #492 trap). Re-evaluate the predicate; gate on its result, not the tracker's state. Echo the result in the Step 6 summary.

2. Enforce the validation rules from `~/.claude/plan-schema.md`:
   - Frontmatter `status` must equal `approved`.
     - If `draft`: tell the user to promote it and stop.
     - If `executing`: ask whether to resume.
     - If `done` / `abandoned`: ask whether to re-open.
   - Every item must have a `slug:` and a non-empty `acceptance:` block.
   - Every item's `slug` is unique within the plan and matches `[a-z0-9-]+` (≤40 chars).

   - Every `after:` slug (logical-ordering edge) must exist in this plan; the union of `depends-on` + `after` edges must be acyclic.
   - No item's `acceptance:` block still contains the placeholder line `- (no acceptance criteria derivable from source — fill in during review)` — a remaining placeholder is a validation FAILURE (fill it in before approving).

(A `gate_check:` is optional on an item with no external gate.)
   - Every `source:` target must resolve. A **vault wikilink** source (`[[path#heading]]`) is checked via `mcp__obsidian__get_vault_file` on the path part (heading anchor isn't strictly verifiable).    - The level list is the unit of batching: all items in level *k* park as `[m]` and pass through a single merge gate before level *k+1* starts.
   - Items already `[x]` (merged) **or `[v]` (spike verdict-captured)** are terminal and skipped (resuming a partial run) — their level is not re-processed and their kind-fork does not re-run. This is load-bearing for the keystone-spike gate (temperloop#526): a keystone spike already `[v]` at run start is excluded here, so its level is never re-entered on a resumed RUN 2 and the 4e-keystone halt cannot double-fire.
   - Items `[~]` are surfaced: if the item carries a `pr:` sub-line (3f recorded it), **re-attach to that open PR** (re-watch CI / re-check mergeability) instead of re-spawning a worker. **Else if it carries `speculative: true` and no `pr:`** — it is a *held speculative worker* (its worktree has real build work, holding before the 3f currency check); **do NOT reset/re-spawn** (that discards the build). Resume it at the **3f step-0.5 base-currency check**: `BASE_CURRENT` → promote to push-by-SHA; `BASE_STALE` → rebase-then-reverify (push the rebased SHA, re-run 3e.5, re-watch CI) or discard-and-respawn on conflict. **Else if it carries `escalated: true` and no `pr:`** — it is an *escalated item awaiting continuation* (its worktree is intact, a verdict block was appended in 3d-esc step 3, and it is waiting for a re-spawned worker to drive it to `[m]`); **do NOT reset/re-spawn** (that discards the worktree and the human verdict). This case must be distinguished from both the `speculative: true` held-speculative case and the plain `[~]`+no-`pr:`+no-sentinel "stuck worker → ask resume/reset" default. Otherwise (no `pr:`, no `speculative:` sentinel, no `escalated:` sentinel) ask whether to resume or reset.
   - Items `[m]` from a prior session are **revalidated before** rolling into the gate: `gh pr view <pr> --json state,mergeable,mergeStateStatus,statusCheckRollup`. `state != OPEN` → surface (re-open / mark `[-]` / re-run); `statusCheckRollup` not `SUCCESS` (and not `PENDING`) → demote to `[~]` and re-enter 3g — **except an item carrying a `  - no_ci: true` sub-line** (3h stamps this at park time for a `NO_CI`-outcome item, temperloop#605): with zero check-runs ever configured, its `statusCheckRollup` reads neither `SUCCESS` nor `PENDING` **by construction**, which is that item's normal, stable `[m]` state — not a demotion signal. Leave it `[m]` and do **NOT** re-enter 3g (there is nothing for 3g to poll; re-entering would spin until the grace window re-fires `NO_CI` again, for no benefit). 5. Show the parsed level list + per-level item ordering.

   [ -x "$overlapCheck" ] || overlapCheck=""   # consuming-repo fallback, like the quota gate
   [ -n "$overlapCheck" ] && "$overlapCheck" --board "$BOARD" <file...>
   ```

   - Append a `## Funnel freeze` section to the plan note via `vault_append` (reliable EOF op): timestamp, who/what decided (operator choice vs unattended default), the lever used (`schedule note enabled: no`), and the standing obligation line `un-freeze owed at epic close (4d-funnel)`. This section is the durable marker 4d-funnel keys off.

## Step 2.5 — Mirror plan items onto the board (board integration ON only)

Skip this whole step when board integration is OFF. Goal: every item that will be worked carries a `gh_issue:`, so it can be claimed (3a) and `Closes #N`'d (3f). **Idempotent** — items that already have `gh_issue:` are left untouched, so a resumed run creates nothing twice. (A plan triaged the **normal** way — `/triage` made native sub-issues, `/assess --epic` prefilled each item's `gh_issue:` from them — arrives with every `gh_issue:` already set, so this step is normally a no-op for it. **Two triaged variants are the exception** and arrive with `gh_issue:` *unset* by design — split items, and a **Contract-derived epic-decomposition plan** (foundation #526; a pre-designed epic decomposed from its `## Contract` body, every item `gh_issue:`/`split_from:` unset with `epic:` in frontmatter). Both are the *expected* set here — minted below, then linked under the existing epic in Step 2.6.)

1. Collect the **pending** items to be worked this run (state `[ ]`; not user-`[-]`-skipped) that have **no** `gh_issue:` field. On the triaged path this set is normally empty, with **two deliberate exceptions:** (i) **split items** — `/assess` leaves `gh_issue:` unset and stamps `split_from: #N` whenever it splits one coarse sub-issue #N into multiple items, precisely so this step mints a **fresh issue per item** and the one-item↔one-issue invariant holds (#73); and (ii) **Contract-derived items** (epic-decomposition mode, foundation #526) — `/assess` leaves `gh_issue:` **and** `split_from:` unset on every item of a pre-designed epic it decomposed from the epic's `## Contract`, so this step mints a fresh issue per item, which Step 2.6 then links under the **existing** epic (there is no coarse parent to close — unlike split items). Both kinds are expected here; mint them like any other.
2. If that set is empty, skip to Step 3. Otherwise the creation gate is a **`batch-at-gate`** decision — **append it to the plan note's `## Questions` section** instead of interrupting the level, then proceed on the default (**create**). Entry: `` `step: 2.5` — Create N tracking issues for the unmilestoned items (<slug list>)? **default: create** `` with an `auto-proceed:` sub-line (unanswered → create the N issues; this is the same disposition `--unattended` takes). Choosing **Skip creation** at the gate leaves those items untracked (they run normally but are never claimed and emit no `Closes`); **Cancel the run** is also offered at the gate. So this step still **creates the issues now** (proceeding on the default) so the items can be claimed and worked this level — the deferred question only lets the operator *retract* that at the gate; an unattended run never stalls here. *(Resume idempotency is unchanged: 2.5 probes for an already-created issue before creating, so a default-create that is later retracted at the gate is the operator's explicit choice, not a duplicate.)*
   - **Unattended (`--unattended`):** proceed and create them — the approved plan is the authorization; no `## Questions` entry is appended (there is no operator to surface it to at the gate).
3. For each item in the set:
   - **Probe before create (crash-safe idempotency), corpus-rg-first.** Search for an issue a prior crashed run may already have created, matching on the exact `Tracked in plan: [[Plans/<date> <project> - <title>#<slug>]]` back-link string (unique per item) — **not** the title (titles collide across plans). Resolve the marker-probe helper once per run (mirrors 2.6/2.5's other single-resolve lib seams): `PROBE_LIB="$(git rev-parse --show-toplevel 2>/dev/null || echo .)/workflows/scripts/lib/issue-marker-probe.sh"`; if it exists, `source "$(dirname "$PROBE_LIB")/../board/lib/cache.sh"; source "$(dirname "$PROBE_LIB")/knowledge_store.sh"; source "$(dirname "$PROBE_LIB")/issue-corpus.sh"; source "$PROBE_LIB"` and probe with `issue_marker_probe "$repo" "Tracked in plan: [[Plans/<date> <project> - <title>#<slug>]]"` — it searches the rendered issue corpus first (zero `gh` calls when fresh) and falls back to the identical live `gh issue list --search "<slug> in:body" --state all --json number,title,body` this step always used whenever the corpus is absent or stale-beyond-limit. **Graceful degrade:** if `PROBE_LIB` doesn't exist in this checkout, use that direct `gh issue list` call exactly as before this item. If a match exists (`jq -r '.[0].number // empty'` off the returned array), **adopt** its number and skip creation.
   - Otherwise create it: `gh issue create --title "<item.title>" --body "<body>"`, where `<body>` = the item's one-line scope, the `acceptance:` bullets, `Derived from: <item.source>`, and `Tracked in plan: [[Plans/<date> <project> - <title>#<slug>]]`. Apply the repo's default issue label if project `CLAUDE.md` names one. **If the item carries `split_from: #N`,** add a `Split from #N` line to the body (human lineage back to the coarse sub-issue) — the operative close-out of #N happens in Step 2.6.
   - **Write order — durable pointer before board mutation:** immediately patch `  - gh_issue: <N>` into the item via `mcp__obsidian__patch_vault_file` — and **in the same patch, when the item carries `split_from: #N`, remove the `split_from:` line**, so the item ends with `gh_issue:` set and `split_from:` gone (the "1:1 restored" end-state: e.g. `split_from: #40` / no `gh_issue:`  →  `gh_issue: #57` / no `split_from:`). This atomic `split_from:`→`gh_issue:` swap is what keeps `plan-schema.md` **rule 10**'s mutual-exclusion invariant always true, so a resumed run's Step-1 re-validation of rule 10 passes instead of failing on an item that carries both. **Then** place it on the board through the adapter — `source "$BOARD_LIB"; board_capture_item "$BOARD" "<issue-url>" <N>` (rides the board's auto-add workflow and lands it in **Backlog**, falling back to an explicit add only if auto-add never fires — GH #53; dedups by URL). Landing the vault pointer first shrinks the duplicate window to just the board-add. New items land in Backlog — the per-item 3a claim moves them to In Progress. This is what makes resume idempotent and lights up the `Closes #N` path in 3f.
4. Record the created `(slug → issue#)` set for the Step 6 summary.

## Step 2.6 — Ensure the epic + sub-issue linkage (board integration ON only) — backstop on the triaged path

Skip when board integration is OFF. Goal: the plan is tracked as one unit by a parent **epic**, and every per-item issue is a child of it. **Idempotent on the plan's `epic:` frontmatter field**, exactly as 2.5 is idempotent on per-item `gh_issue:` — a resumed run creates nothing twice. See [[Decisions/foundation - build epic creation at run-start]].

**On the triaged path this step is a verification no-op.** `/triage` owns epic **birth**: it creates the epic and links survivors as native sub-issues, and `/assess --epic` writes `epic: <N>` into the plan frontmatter. So a triaged plan arrives with `epic:` set and its children already linked — there is nothing to create. Epic **creation here is now a backstop**, firing only for a plan that did *not* come through triage (no `epic:` frontmatter — e.g. a hand-written plan). Child-linkage stays as an idempotent safety net (it skips already-linked children, which is all of them on the triaged path).

1. **Ensure the epic exists (backstop).** If the plan frontmatter has no `epic:` (the non-triaged path):
   - The epic-creation confirmation is a **`batch-at-gate`** decision — **append it to the plan note's `## Questions` section** rather than interrupting, then proceed on the default (**create**). Entry: `` `step: 2.6` — Create a parent epic for this plan? **default: create** `` with an `auto-proceed:` sub-line (unanswered → create the epic and link the children; same disposition `--unattended` takes). **Skip — no epic this run** / **Cancel** are offered at the gate. As with 2.5, the step proceeds on the default *now* (creating the epic so children can be linked and the 4d epic-close can converge); the deferred question only lets the operator retract it at the gate. **Unattended:** create it (the approved plan authorizes); no `## Questions` entry is appended.
   - `gh issue create --title "Epic: <plan title>" --body "<body>"`, where `<body>` = the plan's `## Summary` plus a task-list of every item (`- [ ] <title> (#<gh_issue or TBD>)`). Apply an `epic` label if the repo defines one. Place it on the board through the adapter (`source "$BOARD_LIB"; board_capture_item "$BOARD" "<epic-url>" <epic-N>` — auto-add aware, explicit add as fallback; GH #53).
   - **Patch `epic: <N>` into the plan frontmatter immediately** (`mcp__obsidian__patch_vault_file`) — this is what makes the step idempotent across resumes.
   - If `epic:` is already present (triaged path), **adopt it** — skip creation entirely; just run the linkage check below.
2. **Link each per-item issue as a child of the epic.** For every worked item with a `gh_issue:` not yet linked: attach via the REST sub-issues API — `gh api -X POST repos/<owner>/<repo>/issues/<epic>/sub_issues -F sub_issue_id=<child-db-id>` (this needs the child's **node/database id**, not its number; resolve with `gh api repos/<owner>/<repo>/issues/<n> --jq '.id'`). Skip children already linked (the common case on the triaged path).
   - **Linkage failure warns-and-continues** (does NOT halt) — the typed parent link is cosmetic relative to closure; `Closes #N` (3f) + the 4d board→Done move + the 4d epic-close still work without it. Note the warning in the Step 6 summary.
   - **Fallback when the sub-issues API is unavailable:** append `- [ ] #<child>` to the epic body's task-list and prepend `Part of #<epic>` to the child body — GitHub renders that as a tracked relationship, and the 4d epic-close fallback can read the task-list.
   - **Split items** (no `split_from:` parent yet linked as a leaf): link each minted leaf as a **direct child of the epic** exactly like any other item — *not* nested under its coarse `split_from` parent. The 4d epic-close counts the epic's one-level open children, so the leaves must sit at that level for it to converge.
3. **Close out each coarse `split_from` parent (board ON, split items present).** For every distinct `#N` appearing as a `split_from:` value among this run's worked items: once **all** of #N's split leaves have been minted (Step 2.5) and linked as epic children (step 2 above), retire #N so the data-driven 4d epic-close doesn't hang on a bucket issue no item will ever close. **Idempotent — skip #N if already closed.**
   - **De-link #N from the epic** (best-effort, cosmetic): `gh api -X DELETE repos/<owner>/<repo>/issues/<epic>/sub_issue -f sub_issue_id=<N-db-id>` (resolve the db id with `gh api repos/<owner>/<repo>/issues/<N> --jq '.id'`). Warn-and-continue on failure.
   - **Close #N** (load-bearing — this is what 4d's open-children count reads): `gh issue close <N> -R <repo> --comment "Split into <space-separated #leaf list> by /build; tracked by those sub-issues."` and move its board item to a terminal state via the adapter (`board_resolve_item "$BOARD" <N>; board_set_status "$(board_item_id <N>)" "$BOARD_OPT_DONE"`). (Same unchecked-return family as the 3a epic stamp, foundation#1220 — if `board_set_status` exits non-zero, warn naming #N and continue: the `gh issue close` already fired the close→Done cascade, and `reconcile.sh` backstops the card, so this is warn-and-continue, not a halt.)
   - **Never close #N while any of its leaves is unminted** — that would reintroduce the premature-close failure #73 fixes. Gate on all leaves having a `gh_issue:` first.
3. Record `(epic #, created vs adopted, linked children)` for the Step 6 summary.

## Step 3 — For each dependency level, process items then run the batch merge gate

For level *k* in the level list, process every item through 3a–3h, then run Step 4 (batch merge gate) before moving to level *k+1*.

### Step 3 is dual-path: conversational (default) vs `--workflow`

**Branch on the `--workflow` flag once, per level, here** — the two paths run the *same* 3a–3h mechanics over the *same* spine scripts; they differ only in *who drives the loop*:

- **`--workflow` OFF (default — the conversational path).** This session drives the within-level loop inline: the two-sweep / `parallel()` fanout described in **Within-level execution** below, running each item's 3a–3h directly and parsing each worker's verdict JSON (§3c) by hand. This is the proven path — the 3a–3h prose below **is** that path, executed conversationally.

- **`--workflow` ON (the per-level Workflow path).** Invoke the per-level Workflow once for **this level** via the Workflow tool — by **`scriptPath`** to the deployed script (`"$HOME/.claude/workflows/build-level.mjs"`, foundation #422), **NOT** by `name:` (the tool's `name:` resolves built-ins ONLY — deep-research/code-review — never deployed user workflows; #437). `args` is delivered to the script as a JSON **string** (the `.mjs` parses it itself). Pass:
  ```
  args = { repoRoot, planLink, board, items, verdicts, ownerRepo, claimCmd, spineBinDir, onlySlugs }
  ```
  where `items` is this level's **full** item array (each `{ slug, branch, title, kind, ghIssue, alsoCloses, model, acceptance, source, scope, notes, dependsOn }`). How each arg is resolved:
  - **`dependsOn`** — the **dep-merge precondition** input (#108), an array of `{ slug, sha }`: for each of this item's `depends-on` targets, the dep's **merged head sha** (read from that dep's plan-note `pushed_sha:` sub-line — a `depends-on` target is by construction in a **prior**, already-merged level, so its `pushed_sha:` is recorded). Empty/omitted for level-0 items and items whose only cross-item edges are `after:` (logical-order, not merge). `build-level.mjs`'s 3b-0 gate runs `worktree.sh deps-merged` on these shas **before** `worktree.sh create`: the worktree is created only once **every** dep sha is an ancestor of `origin/<default>` (the depended-on PR has MERGED), so the worker builds and self-verifies against merged dependency code — not a pre-merge base. If a dep has not merged, the item escalates `dep-not-merged` (no worktree, no worker) rather than silently building on stale code. The orchestrator's level ordering (Step 4 merges level *k* before this call for *k+1*) is the primary guarantee; `dependsOn` is the mechanical backstop against a resume race / partial merge / ordering bug. The 3f unconditional rebase (#525) does **not** replace it — that only repairs the branch *textually* at push, too late for the worker's own build/verify.
  - **`repoRoot`** / **`planLink`** / **`board`** — from the run context (the parent checkout's top-level path, the plan note's vault link, the logical board number or OFF).
  - **`ownerRepo`** — `"owner/repo"` for `ci-poll.sh` / `gh` ops; resolved by the **Step 0 probe** (`gh repo view --json nameWithOwner -q .nameWithOwner`). **The workflow has no shell to derive it**, so the orchestrator MUST pass it — without it every CI poll gets `''` and resolves to `ERROR`.
  - **`claimCmd`** — the absolute path to the board **`claim.sh`** entrypoint, resolved by the **Step 0 `CLAIM` probe**; consumed by 3a (board-ON claim). Absent → 3a falls back to a bare `claim.sh` on `PATH`.
  - **`spineBinDir`** — the **already-resolved absolute path** to the build-spine script directory (`worktree.sh`/`pr.sh`/`ci-poll.sh` live here). Resolve it in the orchestrator's **own** shell (Step 0): `spineBinDir="$(cd workflows/scripts/build 2>/dev/null && pwd)"` — a plain `cd`+`pwd` that follows a `workflows/` → foundation symlink to the real path. Passing it lets `build-level.mjs` emit **plain absolute spine paths** (`'<dir>/pr.sh' push …`) instead of resolving them **inline** with a nested `$(dirname "$(readlink -f …)")` command-substitution — which the auto-mode safety classifier read (together with the runSpine "run exactly this command" instruction) as an obfuscated-command bypass and **denied** on `--unattended`/funnel runs, blocking every push/worktree spine step and then crashing `driveItem` on the null result (temperloop#72). **Absent → `build-level.mjs` falls back to its in-bash `spineBin` resolver** (the #560 repo-local→foundation chain), so this is a classifier de-trip optimization, **not** a hard dependency — an older/consuming-repo invocation that omits it still runs, just without the de-obfuscation.
  - **`verdicts`** — the escalation-continuation map. **Empty/absent on a fresh level**; on a 3d-esc continuation, keyed by slug as `{ [slug]: { kind, verdict_section } }` (the full appended markdown block — see 3d-esc step 4). The workflow reads it **only** for slugs in `onlySlugs`.
  - **`onlySlugs`** — the continuation filter. **Absent on a fresh level** (drive everything); on a 3d-esc continuation, set to the array of still-unresolved slugs to re-drive (siblings already parked are left untouched). A slug in `onlySlugs` is driven in **continuation mode** — 3a (claim) and 3b (worktree create) are **skipped** and the captured verdict is injected at 3c (see 3d-esc step 4).

  The Workflow drives **every active** item's 3a–3h via the `runSpine(cmd)` bridge (each spine call is one `agent({schema})` running exactly one command and returning its closed-outcome JSON) plus an `agent({schema})` worker per item, expresses the level as a `parallel()` over `driveItem` (= the `pipeline()` that owns intra-level concurrency), and **returns only** a small `{ parked, escalations }` summary:
  ```
  { parked:      [{ slug, pr, pushed_sha, acceptance_results }],
    escalations: [{ slug, kind, payload }] }
  ```
  The Workflow **NEVER writes the plan note** and **NEVER merges** — it returns *what to write*; the orchestrator does the writeback (serialized, single-writer) and runs Step 4. The orchestrator then partitions the return: `parked` items get their 3h sentinel writes + worktree removal + claim release; `escalations` items re-enter the **3d-esc** continuation loop (below). See the **orchestrator↔workflow boundary** statement under 3d-esc.

**The 3a–3h prose below is retained verbatim** — it is (a) the **spec the Workflow implements** (`build-level.mjs`'s `driveItem` is the as-built encoding of 3a–3h; read its header DESIGN NOTES) and (b) the **conversational fallback** run inline when `--workflow` is OFF. Do **not** delete it.

### Per-item processing (3a–3h)

**Branch on `kind` first.** A `kind: spike` item has no PR — its deliverable is a Context/Decision note + a routed follow-up issue. For a spike: run 3a (claim, mark `[~]`), then spawn a **read-only** worker whose return reports `{ verdict_note_path, routed_issue }` instead of commits; verify the note exists (`mcp__obsidian__get_vault_file`) and the issue is routed (`gh issue view`), then **skip 3b–3h entirely** and mark the item `[v]` (verdict-captured). On board-ON projects, move its issue to Done on verdict-capture (cf. 4d) and close it with a comment linking the note — no PR `Closes` — then **run the 4d epic-close check** (a spike closing can be the epic's last open child); if 4d-epic closes the epic here, run 4d-retro immediately after (same trigger/guard). A **routine** spike (`keystone:` absent) runs **fully autonomously** — no operator gate — even when its verdict reopens a previously-locked decision; it produces findings, not merges, and the downstream merge gates stay intact.

**Keystone-spike review gate (`keystone: true` — temperloop#526).** A spike whose plan item carries `keystone: true` (§ plan-schema `keystone:`) is a *keystone* spike — its verdict **reshapes downstream contracts/scope**, so its findings must **not** flow into dependent builds unreviewed. Capture its `[v]` verdict exactly as above (note + routed issue, Done-move, 4d/4d-retro epic checks all run identically — the marker changes nothing about verdict capture). The halt itself is **not** raised here in the kind-fork — it is raised at the level boundary by **Step 4e-keystone**, which is **plan-note-driven and path-agnostic** (it re-reads the plan note and keys off the `keystone: true` + `[v]` sentinels the kind-fork just wrote, so it works identically whether the kind-fork ran conversationally or inside `build-level.mjs`, and needs no in-memory flag to cross the orchestrator↔workflow boundary). Do not halt *inside* the kind-fork: let the spike's whole level finish its per-item processing and its no-op merge gate first, so a level that happens to pair the keystone spike with an independent sibling still completes that sibling before the boundary halt (in the isolated-level authoring pattern the level is spike-only, so this is moot). The halt is `blocking-now` (no safe default) and is **NOT bypassed by `--unattended`** — see 4e-keystone.

A `kind: code` item (the default) runs the full 3a–3h below.

### Within-level execution — two-sweep, optionally parallel

Items in a level are independent by construction (no merge edge between them — that is what put them in one level), so the orchestrator does **not** block on each item's CI before starting the next. Two modes:

- **Two-sweep (default for ≥2 items).** *Spawn-push sweep:* run 3a–3f for every item (claim, worker, push-by-SHA, open PR), kicking off the **REST CI poll (3g)** as a background job per PR and recording the `pr:` pointer (3f). *Collect sweep:* join each background CI poll, then run 3g (failure handling) + 3h (park). This overlaps the long CI waits across items instead of serializing them. If interrupted between sweeps, `[~]` items with `pr:` re-attach to their open PRs on resume (Step 1.4) rather than re-spawning.
- **Parallel fanout (≥3 items; the width knob).** Express each item's **kind-appropriate** pipeline as a `parallel()` over the level's items on the Workflow substrate — `code` items run claim → worker → push → PR → CI-watch → park; `spike` items run their read-only kind-fork to `[v]` (no push/PR/CI). Three coordination rules are load-bearing: (1) **all plan-note patches are serialized through the orchestrator** — workers never patch the note (concurrent `patch_vault_file` on one file races); (2) **blocked/failed items are collected into one batched `AskUserQuestion` at the join** and do not halt siblings; (3) the parent-tree **leak-check (3h step 0) runs once at the join**, not per item — if dirty, bisect by each branch's committed paths. A `--serial` escape hatch forces one-at-a-time for debugging.

Below the width threshold (single-item levels), process the one item through 3a–3h directly.

**Under `--workflow`, this within-level concurrency is owned by the Workflow, not this section.** When `--workflow` is set (Step 3 dual-path above), `build-level.mjs` expresses the level as a `pipeline()` = `parallel(activeItems.map(driveItem))` and owns the intra-level fanout itself — so the two-sweep / `parallel()` orchestration described here is **superseded by the workflow's `pipeline()`** for that run. The two coordination guarantees still hold identically (and are the workflow's responsibility under `--workflow`): (1) **the orchestrator is the single plan-note writer** — the workflow returns `{parked, escalations}` and writes nothing, so concurrent `patch_vault_file` races cannot occur; (2) **blocked/failed/design-fork items are collected, not halted inline** — they ride back as `escalations[]` and are batched at the level boundary (3d-esc) rather than interrupting siblings. The text in this section remains the **conversational-path** (default, `--workflow` OFF) behavior, driven inline by this session.

##### Speculative next-level execution (opt-in optimization — overlap build with merge latency)

This is an **opt-in** optimization layered on top of the default sequential level flow — it changes *when* a worker spawns, never the one-worktree-per-item isolation or the per-level merge gate. With it off, levels run strictly sequentially as above; with it on, the orchestrator overlaps the next level's build with the current level's merge-wait. Governed by the ratified [[Decisions/foundation - Speculative next-level execution worktree-isolation contract]] — implement exactly that contract.

> **NON-GOAL under `--workflow` (v1).** Speculative cross-level overlap is **out of scope for the `--workflow` path** in v1. The per-level Workflow model (`build-level.mjs` drives exactly one level and returns `{parked, escalations}`) **quantizes the level boundary** — the orchestrator invokes the workflow per level and reasons only over its summary, with no in-process handle to pre-spawn the *next* level's workers during this level's merge-wait. So this entire speculative section is **conversational-path-only** (it applies when `--workflow` is OFF). Under `--workflow`, levels run strictly sequentially with no speculative pre-spawn; lifting this is deferred past v1.

- **What may overlap.** When level *k* reaches its CI-poll / merge-wait (all its items pushed and in 3g, or parked at the Step 4 gate awaiting merge), the orchestrator **MAY pre-spawn level *k+1* workers** rather than waiting for level *k* to fully merge — running their 3a–3e build (claim, worktree, worker, review) against `origin/<default>` while level *k* is still in flight. This recovers the merge-latency window the default flow leaves idle.
- **Eligibility — what may be speculated.** ONLY level *k+1* items with **(a) no `depends-on` into the in-flight level *k*** AND **(b) no *unsatisfied* `after:` edge into level *k*** — i.e. every level-*k* item the candidate points to via `after:` has already reached a **terminal state** (`[x]` merged, `[-]` skipped, or `[v]` verdict-captured), per the `after:`-satisfaction rule in Step 1.4. A `depends-on` item is **NEVER** speculated — it needs the dep's *merged* code, which by definition isn't on `origin/<default>` until level *k*'s gate lands. An item with an **unsatisfied `after:` into level *k*** is likewise **NEVER** speculated — its logical-ordering predecessor hasn't yet reached a terminal state, so spawning it would violate the ordering the `after:` edge encodes. **Note:** during level-*k*'s merge-wait, most level-*k* `after:`-targets are still in-flight (`[~]`/`[m]`), not terminal — so this naturally excludes most `after:`-dependent items from speculation; a candidate qualifies via (b) only when *all* of its level-*k* `after:`-targets happen to be terminal already (e.g. a spike that captured a `[v]` verdict before the level's merge gate). **Eligibility is necessary but NOT sufficient:** two items can be edge-independent (no `depends-on`/`after` between them) yet edit **overlapping lines** of the same file — so eligibility gates *spawning*, never *pushing*. The sufficient half is the base-currency check in 3f (step 0.5 below), enforced before any speculative worker's PR is promoted.
- **Isolation invariants hold identically (#17, #10).** Each speculative worker still gets its **own** orchestrator-pre-created worktree on its **own** `build/<slug>` branch (3b) — speculation changes *when* the worker spawns, not the one-worktree-per-item isolation that #17 fixed (no stacked commits on one branch). The write-jail guard (armed by the worktree's own `.build-guard` marker, dropped at 3b creation) + worktree-qualified paths (3c, #10) hold for a speculative worker exactly as for a sequential one (no parent-tree leak).
- **Promotion is gated on currency, not spawn.** A speculative worker that finishes its build does **not** push immediately. It holds at the boundary before 3f until level *k* has merged and its base can be checked for currency (3f step 0.5) — only then is it rebased/respawned as needed and promoted to a real PR. See 4e for how the in-flight speculative set is handed into the next level.
- **Crash-safe sentinel — stamp `speculative: true` (load-bearing for resume).** A speculative worker sits as `[~]` with a worktree but **no `pr:` yet** (it holds before 3f). That state is indistinguishable from a *stuck non-speculative worker* (`[~]`, no `pr:`) — and the Step 1.4 default for such an item is to *reset/re-spawn*, which would **silently discard the speculative worker's real build work**. To make the two distinguishable across a crash, **when the speculative worker is claimed and its worktree is created (3a/3b), stamp a `  - speculative: true` sub-line on the plan item** — same durable-sentinel pattern as `pr:`/`pushed_sha:`, written via `mcp__obsidian__patch_vault_file`. Step 0.5 reconcile and Step 1.4 resume both key off this sentinel (below) to route a `[~]` + `speculative: true` + no-`pr:` item to the **3f step-0.5 base-currency check** (rebase/respawn as needed) instead of a discard-and-re-spawn. **Clear the sentinel** (remove the sub-line) once the item promotes to a real PR (when `pr:` is recorded in 3f step 3) or merges — past that point it is an ordinary `[~]`/`[m]` item and the `pr:` pointer governs resume.

#### 3a. Claim the item (claim-first), then mark `[~]`

**3a-0. External-gate predicate check (gate-lift by predicate, NEVER by inference) — runs FIRST, before the claim proceeds.** If this item carries a `gate_check:` field (the external-gate predicate — set by `/assess` for an item whose `notes:` hold a cross-plan "don't start until #N lands" gate; see `~/.claude/plan-schema.md` § Optional `gate_check:` field), **evaluate the predicate now and gate the claim on it**:

1. **Run the predicate against the consumable, not the tracker.** `gate_check:` is a command or file-check on the **dependency consumable** (an artifact / file / schema / count) — e.g. `gate_check: "configs/artists.toml lists >=40 artists"`. Evaluate it directly (run the command / check the file). **PASS** → the gate is genuinely lifted; proceed to the claim (3a item 1 below). **FAIL** → the consumable does not yet exist; **do NOT claim or spawn** — park the item (leave `[ ]`, append a one-line `- external gate not yet lifted: <predicate> read false <date>` note) and surface it to the user (skip-for-now / abort), exactly as a `blocked` halt. The gate re-evaluates on the next run.
2. **NEVER infer gate-lift from issue-closed-state.** Do **not** substitute "the gated-on issue #N is closed" for the predicate — **"issue closed" ≠ "dependency consumable exists."** An issue can close one half of its work (data capture) while explicitly deferring the half this item needs (product wiring), so a closed tracker is **not** evidence the consumable exists. This is the **ELT #492 false gate-lift**: #492's gate rode `notes:` as prose, the orchestrator declared *"#380 CLOSED → roster gate lifted"*, but #380 had deferred product wiring (`configs/artists.toml` still listed 3 of 40 artists) — #492 was re-opened on that false premise, re-parked, two prereqs filed (#519/#520), and the leg re-decomposed: the costliest replan of the run. The `gate_check:` predicate exists precisely so the gate-lift decision reads the consumable, not the tracker — run it, don't infer.
3. An item with an external gate in `notes:` but **no `gate_check:`** failed plan-schema **rule 11** (Step 1 validation) and should not have reached here; if one slips through, treat the missing predicate as a halt (surface it) rather than inferring lift from issue state.

(Most items carry no `gate_check:` — this sub-step is a no-op for them. It binds only items with an external/cross-plan gate.)

**Board integration ON** — claiming is the next per-item action (after 3a-0), before the worker spawns (claim-first — canonical: stageFind project `CLAUDE.md` § Task workflow → "Claim first"):

1. **Contention pre-check.** Read this issue's current board state via the adapter (`source "$BOARD_LIB"; board_resolve_item "$BOARD" <gh_issue>`, then read its `.status` and `.["host/Session"]` from `BOARD_ITEMS_JSON` — the single resolved item). If `.status` is already **In Progress** under a *different* `Host/Session` than this run, treat it as a **claim conflict** — halt the item via `decision_sink_ask(<the conflict + the owning Host/Session>, [retry, skip `[-]`, abort], blocking-now)` (operator-present → modal `AskUserQuestion` surfacing the identical retry / skip / abort prompt; operator-absent → async decision-issue posted on this item's GH issue, item parks, tick stops for this item). Don't steal another session's claim. (`claim.sh` itself would silently overwrite, so this pre-check is what honors the lock.)
2. **Claim.** Run `"$CLAIM" <gh_issue> --board "$BOARD"` (the entrypoint resolved by the Step 0 probe) — it sets In Progress, stamps this run's `Host/Session` (`$CLAUDE_CODE_SESSION_ID`), and sets the tmux claim marker in the orchestrator window. If it exits non-zero (missing `project` scope, issue not on the board, network) → **halt the item** (retry / skip / abort); do **not** spawn the worker on an unclaimed item.
   - An item with **no** `gh_issue:` (creation was skipped in 2.5) can't be claimed — note that and proceed unclaimed.
3. **Epic → In Progress + claim-stamp (once per run, if the plan has `epic:`).** When claiming the **first** child of this run, move the epic's board item to **In Progress** if it isn't already **and stamp it with this run's `Host/Session`** — the epic is the lock unit for the *run*, so it must carry a real ownership stamp, not a bare status flip. Via the adapter: `source "$BOARD_LIB"; board_resolve_item "$BOARD" <epic#>; eid="$(board_item_id <epic#>)"; board_set_status "$eid" "$BOARD_OPT_INPROGRESS"; board_stamp "$eid" "$BOARD_FIELD_HOSTSESSION" "<run-stamp>"` — where `<run-stamp>` is the **same `host:sess` value `claim.sh` writes** (`host="${SUBSET_HOST_LABEL:-${STAGEFIND_HOST_LABEL:-$(hostname -s)}}"`, `sess="${CLAUDE_CODE_SESSION_ID:-manual}"`, `stamp="${host}:${sess:0:8}"`); read it back from any child this run just claimed if convenient. Idempotent and cheap — re-stamping an epic this run already owns is a no-op, so do it on the first claim — but **the once-per-run skip latches on success only**: check both adapter calls' exit codes (`board_set_status` / `board_stamp` return non-zero on real failure — e.g. the API-rate-limit exhaustion that also kills claims, foundation#1220). If either exits non-zero → **warn loudly, naming the epic and the failure** (`WARNING: epic #<epic#> In-Progress stamp failed: <error> — will retry at next child claim`), do **not** latch, and **retry the full status+stamp pair at the next child claim** this run makes; skip thereafter only once both calls have succeeded. A run that ends with the stamp never landed **must report the unstamped epic in the Step 6 summary** (the Board line's `epic stamp:` slot) rather than ending clean — a silently-unstamped epic is exactly the unattributable-owner state this stamp exists to prevent, and it also means Step 5's park-back will (correctly) decline to park an epic that doesn't carry this run's stamp. **Why stamp, not just status:** without it, "which run owns this epic" is unknowable, and a crashed run leaves the epic In Progress with no attributable owner (observed: an epic kept a 3-days-dead session's stamp because nothing re-stamped it). The epic's Done transition is the data-driven 4d check; the epic's **park-back** is the new Step 5 step (both below). (Rollup of the epic's own `Status` beyond In-Progress/Done is deferred — see the triage decision's "Open / deferred".)

Then patch the item's checkbox to `[~]`. Lets `git log` and the vault agree on what's in-flight. (Board OFF: just patch `[~]`.)

#### 3b-0. Dep-merge precondition gate (before create — #108)

**A `depends-on` edge requires its target be `[x]` MERGED before this item's worker starts** — the worker must build and self-verify against the dependency's *merged* code, not a pre-merge base. Level ordering (Step 4 merges level *k* before this call for level *k+1*) is the primary guarantee, but it is enforced only by orchestrator sequencing; this gate makes it **mechanical**. For each `depends-on` target of the item, read that dep's **merged head sha** from its plan-note `pushed_sha:` sub-line (the dep sits in a prior, already-merged level, so `pushed_sha:` is recorded) and, **before** calling `worktree.sh create`, verify it has landed:

```
workflows/scripts/build/worktree.sh deps-merged <repo-root> <dep-sha,dep-sha,...>
# → {"outcome":"DEPS_MERGED"}  (every sha is an ancestor of origin/<default> — all deps merged)
#   {"outcome":"DEPS_UNMERGED","unmerged":[...]}  (≥1 dep has NOT merged)
# The script fetches origin/<default> first (like create), then tests ancestry. An unknown/
# unfetched sha reads UNMERGED (conservative — never a false green). Branch on .outcome.
```

Only on `DEPS_MERGED` proceed to 3b (`worktree.sh create`). On `DEPS_UNMERGED`, **do NOT create the worktree and do NOT spawn a worker** — escalate `dep-not-merged` (a resume race, a partial merge, or an ordering bug left a dep unlanded; surface it rather than build against a stale base). Items with no `depends-on` (level-0) or only `after:` edges (logical-order, not merge) skip this gate entirely — `dependsOn` is empty. **The 3f unconditional rebase (#525) is not a substitute:** it repairs the branch *textually* at push, which is too late for the worker's own build/self-verify against merged code. (Under `--workflow`, `build-level.mjs`'s `driveItem` runs this gate from `item.dependsOn`; see the `dependsOn` arg above.)

#### 3b. Pre-create the per-item worktree (orchestrator-owned, deterministic path)

The orchestrator — **not** the worker — creates one dedicated worktree per item via the deterministic-spine script, at a **deterministic path** so 3f (push-by-SHA) and 3h (cleanup) reference it without depending on anything the worker reports:

**`repo:` honor point — resolve `<repo-root>` before calling `create`.** By default `<repo-root>` is the plan's own checkout (the orchestrator's parent tree). When the item carries `repo:` (`~/.claude/plan-schema.md` § Optional `repo:` field — the kernel-vs-overlay case), resolve `<repo-root>` to a **local checkout of `repo:` instead** — e.g. a sibling checkout at a conventional path, or wherever the operator's environment keeps it — and pass *that* root to `worktree.sh create`, so the item's worktree, branch, and every commit land against the target repo, never the plan's own. **If no local checkout of `repo:` exists, this is a hard block** — surface it via `decision_sink_ask` (missing checkout path + the item) rather than silently falling back to the plan's own repo, which would build the fix in the wrong place. This is a no-op for the common case (`repo:` absent — everything below runs against the plan's own repo exactly as before).

```
workflows/scripts/build/worktree.sh create <repo-root> <item.slug>
# → {"outcome":"CREATED","path":"<repo>.wt/<item.slug>","branch":"build/<item.slug>","base":"origin/<default>"}
# worktreePath = <repo>.wt/<item.slug>; worktreeBranch = build/<item.slug> (throwaway local
# branch; the plan's branch: is honored at PUSH time). Closed outcome set: CREATED, or
# {"outcome":"ERROR",...} + non-zero exit — branch on .outcome, never parse prose.
```

- The script branches it from `origin/<default>` so it starts clean from `main` (a `depends-on` item's base is `main` *after* that dep merged at the prior level's gate — see 4e; the **3b-0 gate above** enforces that "after that dep merged" precondition mechanically before this create runs).
- **`create` drops a `.build-guard` marker file in the new worktree root** — this is what arms the write-jail guard (3c) for any worker running in that worktree. Per-worktree state, so concurrent sessions on one host arm independently; the orchestrator exports no env var. `worktree.sh remove`/`prune` clean the marker up with the worktree.
- **`create` also self-heals a legacy-tracked `.build-verification.md` (#529).** The verification-surface artifact is meant to be a dev-local, uncommitted file (kept out via `info/exclude`), but in consuming repos that committed it before the exclude existed, `info/exclude` is powerless and every item re-commits it → a multi-item level's serial-merge conflicts on it. So `create` untracks it as its **own** `chore: untrack …` commit on the fresh branch. **Expect that one extra deletion commit** on the first build branch in any such repo — it's intentional spine hygiene, not the worker's doing; all level branches make the identical removal (merges delete-vs-delete cleanly), and once that repo's `main` is clean the step is a silent no-op.
- The worktree is created in the **parent checkout's** working tree but is a separate checkout dir; the orchestrator itself stays in the parent checkout. Two parallel level-0 workers each get their **own** pre-created worktree on their **own** `worktreeBranch`, so they **cannot** stack commits on one branch or commit onto the parent's branch (the #17 failure mode).
- The plan's `branch:` name is honored at **push** time (3f), not worktree-creation; remote-branch-name collisions are handled there.
- If the deterministic path already exists from a prior aborted run (Step 0.5 didn't prune it), `create` force-removes and re-adds it — the path is a pure function of the slug, so re-deriving it is always safe.
- **Speculative spawn (opt-in optimization on):** when this worktree is being created for a **speculatively pre-spawned** next-level item (per "Speculative next-level execution"), the orchestrator also stamps a `  - speculative: true` sub-line on the plan item now — via `mcp__obsidian__patch_vault_file`, alongside the `[~]` flip — so a crash between here and the 3f currency check leaves the item legibly distinguishable from a stuck non-speculative `[~]` worker (Step 0.5 / Step 1.4 key off it). The sentinel is cleared when the item promotes to a real PR (3f step 3 records `pr:`) or merges.

#### 3c. Spawn the implementation agent

`Agent { subagent_type: "claude" }` — no `isolation` param; when the item carries a `model:` field, pass it through to the spawn (`Agent { subagent_type: "claude", model: <item.model> }`), and when it's absent pass nothing (the worker inherits the session model — see **Model tiering** below). The worker runs against the **`worktreePath` the orchestrator pre-created in 3b** (a deterministic, orchestrator-owned value — *not* something the worker reports back). The write-jail guard is **already armed** for this worker: `worktree.sh create` (3b) dropped a `.build-guard` marker in the worktree root, and the worker's PreToolUse hook self-arms from that marker (rejecting any write that resolves outside that root — see Option B below); the orchestrator exports nothing. Tell the worker its Bash cwd is `<worktreePath>` and to do all work there.

**Model tiering — tier by verification, not difficulty.** The item's optional `model:` field (`~/.claude/plan-schema.md` § Optional `model:` field, stamped by `/assess` Step 2) is advisory routing for this spawn. The principle, for this seat and any future seat: a seat takes a cheaper model only when a **mechanical gate** downstream — CI (3g), the parent-side acceptance gate (3e.5), the item's acceptance bullets, the structured return contract — catches a weaker model's mistakes; a seat keeps the top model when its output *is* the gate (judgment nothing downstream checks). Route new seats by that test, never by how hard the task looks.

- **Escalate on retry — a retry never stays on the cheap model.** Any re-spawn of this item after a **`failed` return** (3d) or a **CI failure** on its branch runs on the **top tier** — omit `model:` so the re-spawn inherits the session model — regardless of the item's stamped `model:`. The first failure is evidence the cheap tier was insufficient for this item. Re-spawns that are *not* failures — a `blocked` answer round-trip or a `design-fork` verdict — keep the stamped tier.
- **Seat tiers (current policy).** Worker seats: per the stamp (`size: S/M` + `kind: code` → `sonnet`; `kind: spike` or `size: L` → absent, inherit). Review seats: `requirements-auditor`, `workflow-reviewer`, and `docs-reviewer` → `sonnet` (set in their `claude/agents/` frontmatter — their findings are advisory inputs the orchestrator and human filter). Explore-style locate-and-report fan-outs → `haiku` (their structured returns are checked by the caller; the built-in Explore agent has no def file in this repo, so route it at spawn time with `model: haiku` — and set the frontmatter wherever such defs exist downstream, e.g. a consuming repo's `.claude/agents/`, as deploy-time follow-through). `architecture-reviewer` explicitly **keeps the session model** (`model: inherit` in its def) — its boundary calls are judgment whose output *is* the gate. The orchestrator/assess/triage seats (the `/build`, `/assess`, and `/triage` sessions themselves) always keep the session model — they are sessions, not agent defs, and their judgment is what everything else gates on.
- **Seat scoping — nested review delegation (#225 residual).** A worker MAY delegate a focused review to a **nested** agent for context control (keeping a bulky read-and-judge pass out of its own window) where the harness allows. Known constraint: subagent contexts have historically been unable to dispatch further subagents (observed empirically on stageFind PR #38), so this is an **aspiration gated on harness capability**, not a mandated step — a worker that can't nest does the review inline.

The prompt includes:

- Branch policy from project CLAUDE.md.
- **Effective engineering principles (summary form) — resolved once per run, point-of-use, never cached across runs.** Resolve the **kernel set** (`claude/engineering-principles.md`'s "The seven principles" — always applies, even with nothing else declared) merged with the item's project's own extension — its `## Principles` section in `Projects/<project>/Priorities.md`, falling back to the legacy `Priorities/<project>.md` (path fallback convention, `claude/commands/check-in.md`), read via `mcp__obsidian-builtin__vault_read` — **per `claude/engineering-principles.md`'s own § Merge semantics (the single statement site; implement it here, do not restate or re-derive it):** extend by default, or narrow per an explicit `mode: replace`, named exclusion(s), or `none` the section declares — dedupe/override handling (a duplicate collapsing to one entry with project phrasing winning, a genuine contradiction named as a kernel-principle override) is per that section too, not restated here. An absent project section (file/read/section missing) leaves the kernel set alone, project slot empty; a *present but empty or malformed* section resolves the same way plus a degradation notice. **Resolve this once per distinct `(repo, project)` pair encountered this run** — most runs have exactly one, so this is one resolution for the whole run; an item carrying its own `repo:` (cross-repo) resolves its own pair once, reused for every later item sharing it, never re-derived per item and never persisted into a future `/build` invocation. Embed the resolved effective set in the worker's prompt **in summary form** — a compact numbered list of the principle names/one-line criteria in force, each tagged `[kernel]` or `[project]` — this is the **same merged set** 3e hands the pre-push reviewer for this item's `(repo, project)` pair (§ 3e reuses this resolution rather than re-deriving it). Tell the worker to weigh its own choices against this set and, if its own summary cites a principle-shaped concern, to name the principle and its origin.
- The full item block: title, slug, scope, source wikilink, files, acceptance, notes.
- **Source-doc excerpt:** read the `source:` target (path + heading for a vault source, or `gh issue view <n>` for an issue-ref source) and embed the relevant section verbatim, so the worker doesn't have to chase the reference itself.
- **Worktree-qualified file paths — load-bearing, do NOT skip.** Every repo file path you hand the worker (in `files:`, the scope, examples, verification commands) MUST be written under the pre-created `worktreePath` (`<worktreePath>/evals/scorer.py`), never the bare parent-repo-root path (`/…/<repo>/evals/scorer.py`). The worker's `Edit`/`Write` echo the path you give them; a bare parent-root absolute path resolves against the parent checkout and **leaks an uncommitted write into the orchestrator's tree** even though the worker's Bash cwd is the worktree (the #10 parent-leak failure — history in [[Decisions/foundation - build worker write-isolation (worktree failure history + Option A-D)]]). Because the orchestrator now **pre-creates** the worktree (3b), you **know the exact `worktreePath` up front** — prefix every handed path with it, and additionally tell the worker: "make ALL Edit/Write/path references either relative to your Bash cwd (the worktree root) or absolute under `<worktreePath>/…`; never write to the parent `/…/<repo>` checkout." Relative paths from the worker's Bash cwd are safest. **This prose instruction is now backstopped structurally:** with the `.build-guard` marker in place (dropped by 3b's `worktree.sh create`), the worker's PreToolUse guard *rejects* an out-of-worktree write outright (Option B) — so a path-discipline slip fails loudly at the worker instead of silently leaking into the parent (the prose alone was the discretion this item removed).
- **Guard enablement (Option B), worker-side — load-bearing.** The PreToolUse hook `build-worktree-guard.sh` (matcher `Bash|Edit|Write|MultiEdit`) is **inert by default** and **self-arms from a per-worktree marker file**: it enforces iff the tool cwd's `git rev-parse --show-toplevel` contains a `.build-guard` marker AND that toplevel sits under a `<repo>.wt/` directory (the deterministic-path convention from 3b). `worktree.sh create` drops the marker; `remove`/`prune` clean it up with the worktree. **The orchestrator exports nothing** — the marker replaces a prior env-var arming (`BUILD_WORKTREE_GUARD`) that was never armable for Agent-tool workers (why: [[Decisions/foundation - build worker write-isolation (worktree failure history + Option A-D)]] §#171). The marker is per-worktree state, so N concurrent build sessions on one host (a shared cron/deploy host) arm independently and correctly. A marker found **outside** the `.wt` convention (a stale or hand-copied marker) makes the hook warn on stderr and fail **open**; a toplevel with no marker leaves the hook inert — so a normal interactive session, even one with the hook installed via the foundation fleet's `make install` (a kernel checkout installs via `workflows/scripts/install-claude-md.sh` / `install-settings.sh`), is unaffected. **Hook *registration* is two distinct surfaces (#72):** (1) for **foundation-repo** runs, foundation's `claude/settings.json` — symlinked to the user-global `~/.claude/settings.json` by `make install-claude` — registers it with an absolute command path; (2) for a **consuming repo** (e.g. stageFind, whose build runs may execute on a different host), the guard is **vendored into that repo** at `.claude/hooks/build-worktree-guard.sh` and registered in the repo's **project-level** `.claude/settings.json` with a repo-relative command path (path-portable across machines). foundation is the source of truth; `make sync-stagefind-hooks` copies the banner-stamped hook (the settings.json registration is a one-time per-repo edit). **A consuming repo with no project-level registration runs the write-jail un-armed** — build then relies on Option A (pre-created worktrees) + Option C (per-worker parent-clean assertion) alone, which is weaker; verify the registration exists before trusting Option B there. The guard denies any `Edit`/`Write`/`MultiEdit` whose resolved absolute path is outside the marker's worktree toplevel — and, on the **Bash** arm (F#932), any destructive filesystem verb (`rm`/`rmdir`/`mv`/`shred`/`truncate`/`dd of=`) whose target it cannot prove stays inside the worktree, including a non-literal target (`$`-expansion, `$(…)` substitution, or glob) it cannot resolve statically — with an allow-list for `/tmp`, `$TMPDIR`, and gitignored source copies (e.g. a `.env` copied in from the parent). **Matcher-registration caveat:** the Bash arm only fires where the hook's settings.json matcher includes `Bash`; a consuming repo still registered with the old `Edit|Write|MultiEdit` matcher runs the file-tool jail but leaves worker Bash un-jailed until its matcher is updated.
- Instruction to commit on the current branch but NOT push and NOT open a PR.
- **No issue-closing keywords in commit messages — load-bearing, do NOT skip.** Tell the worker: its commit messages and commit titles MUST NOT contain any GitHub issue-closing keyword (`Closes`/`Fixes`/`Resolves`/`Close`/`Fix`/`Resolve` followed by `#<N>`). Use a plain descriptive message only. **Why:** GitHub scans the messages of all commits merged to the **default branch** for closing keywords, not just the PR body — so a worker's stray `Closes #153` in a commit message auto-closes that issue on merge, even when the PR body omits it (observed: commit `ec8d5fd` carried a `Closes #153` that auto-closed an issue whose *other half* of the work wasn't done). Issue linkage is the **orchestrator's** job and belongs in exactly one place — the PR body's `Closes #<gh_issue>` line (3f) — never a worker commit message. This is backstopped by the orchestrator's pre-push scan in 3f (the prompt prohibition alone was the discretion this rule removed).
- **Self-verification requirement:** before returning `done`, the worker must explicitly check each acceptance bullet and report which passed with evidence.
- **Design-fork escalation — do NOT guess on a contract-changing decision.** If, while implementing, you discover that ≥2 viable designs exist and choosing between them would change this item's *contract or approach* — its interface, schema shape, the meaning of an acceptance bullet, or a no-behavior-change guarantee (e.g. a byte-exactness trap, or an acceptance criterion that assumes a mechanism that turns out not to exist) — do **not** pick one and build it, and do **not** file it as `blocked`. Return `status: "design-fork"` with the `design_fork` object (decision, viable options + trade-offs, your recommendation, the evidence that surfaced it). The orchestrator gets a human verdict and re-spawns you against this same worktree with it appended. Rule of thumb: a *missing fact you could simply be told* is `blocked`; a *design choice a human must own* is `design-fork`.
- **Verification surface (per the project rule) — write it to a FILE, return only the path (#418 inflow-cut).** Assemble a verification-surface markdown block appropriate to the change type — eval/artifact → before/after numbers; scorer/prompt → per-fixture diffs with examples; config → corrected-vs-prior side by side; refactor → explicit no-behavior-change claim + the proving diff/tests. **Write that block to `<worktreePath>/.build-verification.md`** (in the worker's own worktree) and return its path as `verification_surface_path` in the verdict — do **not** inline the block in the verdict JSON. This keeps the large surface OUT of the orchestrator's context: it never round-trips through the worker's return message; `pr.sh open` reads the file directly at 3f (`--verification-surface-file`). Quote the `## PR verification surface` rule (extracted in Step 0 item 4 — global `~/.claude/CLAUDE.md`, or the project `CLAUDE.md` if it overrides) into the prompt so the worker meets it. (Back-compat: an inline `verification_surface` field is still accepted as a fallback — `pr.sh` resolves the surface by precedence file-flag → `verification_surface_path` → inline → recap — but the file path is preferred.)
- **No long-running background work — foreground the gate, never `run_in_background`.** The worker runs **every** verification command — the `quality-gates.sh` acceptance gate above all, plus evals, builds, sweeps — in the **foreground**. NEVER launch one with `run_in_background: true`: a subagent's backgrounded process is **reaped when the worker yields**, and there is no `SendMessage` to resume it — a backgrounded gate/eval silently dies. **A turn that ends while awaiting a background-task notification is the #1219 bug, not a valid return** — the notification never reaches a subagent, so the worker hangs and yields no verdict object at all. If one command would exceed the ~10-min `Bash` foreground cap (e.g. a mega-fixture eval), the worker **narrows/splits it, or returns `blocked`/`failed`** and lets the orchestrator run that verification parent-side — it never backgrounds-and-waits.
- **Gitignored files are absent from the worktree** (e.g. `.env`). If a tool needs one, source/copy it from the parent checkout (`set -a; . <repo>/.env; set +a`). (The eval harness loads repo-root `.env` itself per stageFind#203, but other tools may not.)
- **Return contract — a compact structured verdict, as the worker's FINAL message.** End the turn with exactly the fenced JSON block below and **nothing after it** — no prose transcript, no narration of the build. This object *is* the worker's report: the orchestrator drives **every** state transition in 3d (and the PR body in 3f) from these fields alone, so a tight, fully-specified verdict keeps the orchestrator reasoning over a small object instead of ingesting a transcript — which is what bounds orchestrator-context growth across a long run. Return the smallest object the status requires: populate the fields marked **required when status=…** for the returned status, and omit the rest.

  **Field ownership — what the *worker* sets vs what the *orchestrator* writes:**
  - The worker owns **only** the fields below. It does **not** set `branch`, `pr`, or `pushed_sha` — the orchestrator owns the branch (3b), and records `pr:`/`pushed_sha:` itself on push (3f). The worker surfaces `commits` (the sha array it produced); the orchestrator pushes those and writes the PR pointer.
  - The `status` enum is **closed** at exactly the four variants below, and each maps 1:1 to a handling branch in 3d — `done` → 3d (all-passing → 3e; any `passed:false` → blocked re-loop), `blocked` → questions loop, `design-fork` → verdict loop, `failed` → retry/skip/abort. Never return a status outside this set; if you can't pick one, you're in `blocked` (missing fact) or `design-fork` (design choice a human must own).

  ````
  ```json
  {
    "status": "done" | "blocked" | "design-fork" | "failed",   // required ALWAYS — closed enum, 1:1 with 3d branches
    "summary": "<1-3 sentence change summary>",          // required when status=done
    "acceptance_results": [                              // required when status=done — one entry per acceptance bullet
      { "criterion": "<verbatim bullet>", "passed": true, "evidence": "<file:line or test name>" }
    ],
    "commits": ["<sha>", "..."],                         // required when status=done — worker-produced shas; orchestrator pushes them (3f)
    "verification_surface_path": "<path>",               // required when status=done — path to <worktreePath>/.build-verification.md (the surface block, written to a FILE so it stays OUT of orchestrator context); pr.sh open reads it via --verification-surface-file (3f). Legacy inline "verification_surface":"<markdown>" still accepted as a fallback.
    "questions": ["...", "..."],                         // required when status=blocked — the facts you need to proceed
    "design_fork": {                                     // required when status=design-fork
      "decision": "<the design question whose answer changes the item's contract/approach>",
      "options": [ { "label": "<option>", "tradeoff": "<what it costs / implies>" } ],
      "recommendation": "<which option you'd pick + one-line why>",
      "evidence": "<file:line / test / constraint that surfaced the fork>"
    },
    "failure_reason": "..."                              // required when status=failed — why it can't be completed as specified
  }
  ```
  ````

  Fields the worker **never** sets (orchestrator-owned, listed here so the boundary is explicit): `branch` (3b), `pr` / `pushed_sha` (recorded in 3f). The shape above is **contract-stable** — the orchestrator's 3d match keys on `status` plus the per-status fields, so a worker emitting the prior shape (no `required when` annotations — those are documentation, not new keys) parses identically. The `//` comments inside the block aren't valid JSON anyway — any parser strips them; they exist for the worker-as-reader, not the orchestrator-as-parser — so the annotations are purely additive.

  **Under `--workflow`, this object is the `agent({schema})` JSON schema — machine-validated by the runtime, not hand-parsed.** The same closed contract above is encoded as **`WORKER_VERDICT_SCHEMA`** in `claude/workflows/build-level.mjs` (the schema passed to every worker `agent({schema})` call): the runtime *validates the worker's return against it* and hands `driveItem` a typed object rather than free text, so a malformed verdict fails structurally at the workflow boundary instead of mis-parsing. The conversational path (`--workflow` OFF) parses **this same JSON by hand** from the worker's final message — same shape, same `status` enum, same per-status required fields; the only difference is *who* validates it (the runtime under `--workflow`, the orchestrator's own read otherwise). `WORKER_VERDICT_SCHEMA` is the single as-built source — keep this prose contract and that schema in lockstep (a new field/status added to one must be added to the other).

#### 3d. Handle the agent's return (synchronous — halts the level)

Branch **solely on the worker's returned verdict object** (3c) — its `status` plus the per-status fields — never on prose it emitted. The four branches below are the **closed, 1:1 image** of the 3c `status` enum (`done` / `blocked` / `design-fork` / `failed`): every enum variant has exactly one branch here, and every branch consumes exactly one variant's required field(s). Keep them in lockstep — adding a status to one without the other is the half-specified-contract bug this gate exists to prevent.

**First, the verdict object must EXIST (#1219).** Branching *solely on the verdict object* presumes there **is** one. If the worker's final message carries **no parseable verdict JSON with a `status` in the closed enum** — the common cause is a worker that backgrounded a gate/eval (a `run_in_background: true` `quality-gates.sh`) and ended its turn awaiting a notification that never arrives (§3c **No long-running background work**) — do **NOT** wait on it and do **NOT** invent a status. Treat it as an **incomplete return**: re-spawn the same worker **once** against the same `worktreePath`, **on the item's stamped `model:` tier** (this is a mis-applied *instruction*, not evidence the tier was too weak, so the escalate-on-retry top-tier bump does **not** apply to this first re-spawn), with an explicit instruction to *run the gate in the FOREGROUND and end the turn with the fenced verdict JSON and nothing after it*. If the second return is still non-conforming, handle it as **`failed`** with a synthesized `failure_reason` of `"worker returned no parseable verdict after a foreground-instructed re-spawn (#1219)"` (there is no verdict object to source one from) — i.e. enter the `failed` branch below; its own retry/skip/abort re-spawns then follow the normal escalate-on-retry top-tier rule. This is a guard on the **presence** of a valid object, not a branch on emitted prose — so the "branch solely on the verdict object" invariant is preserved. (Under `--workflow` this case cannot arise: `agent({schema: WORKER_VERDICT_SCHEMA})` validates every return, and `driveItem` **retries the main worker once on a `null` verdict, then escalates** — `build-level.mjs`:587–601 — the same retry-once-then-`failed` shape as this guard; the read-only spike path escalates a `null` immediately with no retry — `build-level.mjs`:477–480. Keep this guard and those two blocks in lockstep per the schema↔prose mandate above.)

Each re-spawn is a **fresh** `Agent`, re-run against the **same pre-created `worktreePath`** from 3b (its `.build-guard` marker persists with the worktree, so the guard stays armed — no re-arming step) — there is no `SendMessage` to continue the prior one, but the worktree persists between spawns (the orchestrator only removes it in 3h or on abort). A re-spawn that must build on already-committed work is told to `git fetch origin <item.branch> && git reset --hard FETCH_HEAD` first (only meaningful once 3f has pushed that branch); pre-push re-spawns just start fresh from the worktree's `origin/<default>` base. Re-spawn model tier follows the 3c **escalate-on-retry** rule: after a `failed` return (or a CI failure), re-spawn on the top tier (omit `model:`) regardless of the item's stamped `model:`; `blocked`/`design-fork` re-spawns keep the stamped tier.

- **`blocked`** — surface the questions via `decision_sink_ask(<the worker's `questions[]`>, <one choice per question / freeform answer>, blocking-now)` (operator-present → modal `AskUserQuestion`, the identical prompt as before; operator-absent → async decision-issue posted on this item's GH issue, item parks, tick stops for this item — the drain step translates the operator's typed reply into a `## User answers` block, and the next tick's re-spawn picks it up as if the modal path had answered), then re-invoke `Agent` with the original prompt plus a `## User answers` section. Loop until status changes.
- **`design-fork`** — the worker hit a contract-changing design decision (see the 3c escalation rule). **Branch on operator-presence:**
  - **Operator-present:** surface via `decision_sink_ask(<`design_fork.decision`>, <one option per `design_fork.options` entry (label + trade-off), with the worker's `recommendation` as the first/recommended option>, blocking-now)` — modal `AskUserQuestion`, the operator sees the identical option set. **Then, AFTER the ask returns** (the seam carries only the ask), **capture the verdict durably *before* re-spawning** so a resume or re-spawn never re-litigates it: a one-line stamp on the plan item's `notes:` for a local call, or — for an architectural / load-bearing fork (a schema, a module seam, a public contract) — a `Decisions/` note per the global decision-capture rule, linked from the item. Then re-invoke `Agent` against the **same `worktreePath`** (the `.build-guard` marker is still in place, so the guard stays armed) with the original prompt plus a `## Design verdict` section. Loop until status changes. The item stays `[~]` throughout — a design fork halts the *level* (like `blocked`) but keeps the *build* delegated to the worker; this is what prevents the drop-to-manual context blowup ([[Mistakes/foundation - build design-fork drops to manual, orchestrator context blowup]]). (Order preserved: `decision_sink_ask` → durable verdict capture → re-spawn.)
  - **Operator-absent (v1 exclusion — async backend does NOT apply here):** a live worktree cannot be held across a cron gap. Do NOT open a decision issue. Instead: (1) stamp the plan item's `notes:` field with the fork's decision question and offered options (so the next claim has the context it needs); (2) move the item back to **Ready** via the board adapter (`board_set_status "$(board_item_id <n>)" "$BOARD_OPT_READY"`); (3) remove the `.build-guard` marker from the worktree and discard the worktree; (4) stop processing this item for this tick. The item will be claimed fresh on the next operator-present run or claim, which will see the `notes:` stamp and surface the fork via the modal path. Log: `design-fork on operator-absent run — parked back to Ready (worktree discarded); verdict stamped in notes: for re-spawn.`
- **`failed`** — surface via `decision_sink_ask(<`failure_reason`>, [retry, skip this item, abort the run], blocking-now)` (operator-present → modal `AskUserQuestion`, the identical prompt as before; operator-absent → async decision-issue posted on this item's GH issue with the failure reason and the three options, item parks, tick stops for this item):
  - **Retry** (re-spawn with failure context)
  - **Skip this item** — discard the worker's worktree (`workflows/scripts/build/worktree.sh remove <repo-root> <item.slug>` — removes worktree, branch, and marker), mark `[-]` on plan, move on to the next item.
  - **Abort the run** — clean up before yielding (per *park, don't abandon*): remove every worktree captured this run (`workflows/scripts/build/worktree.sh remove <repo-root> <slug>` per item — worktree, branch, and marker), run `"$RELEASE"`, and leave any already-pushed PRs as `[m]`/`[~]` with their `pr:` recorded. The level merge gate does not fire on aborted runs. Do **not** leave worktrees + claims dangling — that strands board items In Progress under a dead session and orphans locked worktrees (Step 0.5 reconciles these on the next resume).
- **`done` with any `passed: false`** — treat as `blocked`; ask user whether to accept partial completion or send the worker back.
- **`done` with all passing** — proceed to 3e.

#### 3d-esc. Escalation-resume continuation loop (`--workflow` path)

When the level runs via the **`--workflow` flag** (the `build-level.mjs` Workflow — item #426), the per-item driver (`driveItem`) returns `escalations: [{slug, kind, payload}]` instead of halting inline (the item level escalates rather than the parent session). At the **level boundary**, after `parallel()` collects all results, the orchestrator handles the batched escalation set as follows. This is the seam that keeps the orchestrator context bounded to one small `{parked, escalations}` object per level even when some items need a human round-trip.

**Orchestrator↔workflow boundary (state it explicitly).** Under `--workflow`, responsibilities split on a hard line:
- **The Workflow (`build-level.mjs`) owns 3a–3h for one level** — claim, worktree, worker spawn, review, acceptance gate, push-by-SHA, PR open, CI poll, park — driven via the `runSpine` bridge + `agent({schema})` worker, with `worktree.sh`-created worktrees and the `.build-guard` write-jail (NOT `isolation:'worktree'`). It **returns** `{parked, escalations}` and **NEVER writes the plan note** and **NEVER merges** — escalated items keep their worktrees intact for the orchestrator to re-drive; the workflow removes no worktrees.
- **The conversational orchestrator (this session) retains, under `--workflow`:** (1) **Step 4 (the merge gate) in full** — regime detection, the timed/modal approval, the NATIVE/MANAGED backend merge paths, the deferred `## Questions` drain (4f), and the 4d Done-moves / epic-close; (2) **ALL plan-note writeback, serialized as a single writer** — the `[~]`/`[m]`/`[x]`/`[v]`/`[-]` sentinel flips, `pr:`/`pushed_sha:` sub-lines, the `## Run status` dashboard, and the 3d-esc verdict blocks — the workflow returns *what* to write and the orchestrator writes it, one `vault_patch` per item in sequence (concurrent writes race); and (3) **escalation resolution** — the 3d-esc continuation loop below (surface the batched escalations, capture verdicts durably, re-invoke the workflow scoped to `onlySlugs`).

This split is what bounds orchestrator context: the workflow absorbs the per-item spine/worker churn, the orchestrator reasons only over the `{parked, escalations}` summary plus its own merge + writeback + escalation duties.

**Re-derivation inputs (checkable, no live workflow state required).** A continuation is fully re-derivable from two durable sources only:

1. **Plan-note level sentinels + `pr:` sub-lines** — which items are `[~]` (worktree intact, no PR yet), `[m]` (worktree removed, PR parked), which hold `speculative: true` (held before the 3f currency check), and which hold **`escalated: true`** (worktree intact, verdict block appended, awaiting continuation — the discriminator stamped in step 3 below). These sentinels survive a crash and are the authoritative status. A `[~]`+no-`pr:`+`escalated: true` item unambiguously identifies an escalated item awaiting continuation (vs. `speculative: true` for a held speculative worker, and no sentinel at all for a stuck worker).
2. **The appended verdicts map** — the `## User answers` / `## Design verdict` blocks appended to the plan note per the durable-capture step below. These carry the human's choice for each escalated slug.

No in-memory workflow state is needed. A resumed orchestrator reconstructs the continuation by: reading the plan note, finding `[~]` items with **`escalated: true` and no `pr:`** (the durable sentinel stamped in step 3 below — see "Stamp `escalated: true`"), cross-referencing the verdicts map for their slugs, and re-invoking the workflow with those slugs as `onlySlugs`. The `escalated: true` sentinel (stamped in step 3 when the verdict block is appended) is the unambiguous discriminator: a `[~]`+no-`pr:`+no-`escalated:` item is a stuck non-speculative worker (ask resume/reset); a `[~]`+no-`pr:`+`speculative: true` item is a held speculative worker (3f currency check); a `[~]`+no-`pr:`+`escalated: true` item is an escalated item awaiting continuation (re-enter 3d-esc). The `pr:` sentinel (written in 3f step 3) is what distinguishes "worktree intact, escalated" from "PR already open, just watching CI" — never conflate the two.

**Step-by-step loop:**

1. **Partition the level result.** From the workflow's return `{parked, escalations}`:
   - `parked` items — already drove to `[m]`; write their plan-note sentinels now (`[m]` + `pr:` + `pushed_sha:` sub-lines, per 3h step 1), remove their worktrees (3h step 3), and release their claim markers (3h step 2). **Serialize these sentinel writes — one `mcp__obsidian-builtin__vault_patch` per item in sequence** (concurrent `patch_vault_file` on one note races and silently drops a write, per the within-level coordination rule). These siblings are **done for this level boundary** and must not be re-driven by any continuation.
   - `escalations` items — **leave their worktrees intact** (the workflow guarantees this); their plan-note sentinels stay `[~]` (no `pr:` sub-line, since 3f never ran for them). They re-enter the loop below.

2. **Surface as one batched `decision_sink_ask(...)` per escalation kind.** Group by `kind` for clarity; within a group, one option-set per slug. Operator-present → modal `AskUserQuestion`, each `decision_sink_ask(...)` here surfaces the **identical batched modal prompt** as before. Operator-absent → async decision-issue per slug (one issue per escalated item with the kind + payload, `decision` label, operator assigned, item parks) — except `design-fork` slugs on an operator-absent run: those follow the v1 exclusion (park to Ready, worktree discarded, `notes:` stamped) instead of posting a decision issue. The seam carries only the ask; the durable verdict capture (step 3) and the `escalated: true` stamp happen **after** it returns (modal) or after the drain step translates the reply (async), never inside the seam call. **Serialization order (preserved, load-bearing — per [[Decisions/foundation - Autonomous funnel driver + GitHub decision queue]] CORRECTION):** for each slug the `decision_sink_ask` (this step) **precedes** the verdict-block append (step 3) **precedes** the `escalated: true` stamp (step 3). Present `payload` verbatim (the worker's `design_fork` or `questions` object, depending on `kind`):
   - `design-fork` — surface via `decision_sink_ask(<`design_fork.decision`>, <options from `design_fork.options`, the worker's `recommendation` pre-selected>, blocking-now)`. One question per slug (the design decisions are item-specific and must not be merged across slugs — a human must ratify each separately).
   - `blocked` — surface the `questions[]` array via `decision_sink_ask(<the `questions[]`>, <freeform / per-question answer>, blocking-now)`; one ask per slug (or a single multi-slug batch when the questions are identical and an answer applies to all — judge by the payload).
   - `failed` — surface `failure_reason` via `decision_sink_ask(<`failure_reason`>, [retry, skip, abort], blocking-now)`. A **skip** here marks the item `[-]` and prunes its worktree; an **abort** cleans up as in 3d-failed above.
   - `ci-failed`, `acceptance-incomplete`, `acceptance-gate-failed`, `push-rejected`, `closing-keyword`, and all other non-enumerated kinds — **generic escalation path**: render the raw `payload` as a text block with the slug and kind as a heading, then ask the user how to proceed via `decision_sink_ask(<the rendered payload>, [retry, skip, abort], blocking-now)`. A non-enumerated kind must **never produce a silent stall** — unknown payload shape escalates visibly rather than dropping the item.

3. **Capture the verdict durably before re-spawning.** For each escalated slug:
   - **`design-fork`** — append a `## Design verdict — <slug>` section to the plan note via `mcp__obsidian-builtin__vault_append` (**not** `mcp__obsidian__patch_vault_file` — the em-dash in the heading causes that tool to misfire; use the builtin-REST server):
     ```markdown
     ## Design verdict — <slug>
     Decision: <the decision text from design_fork.decision>
     Chosen: <the label of the selected option>
     Rationale: <one sentence from the user's reply or the recommendation>
     ```
     If the fork is architectural / load-bearing (a schema, a module seam, a public contract), **also** file a `Decisions/` vault note per the global decision-capture rule. Either way, the plan-note block is what the continuation reads — the vault note is the long-lived record.
   - **`blocked`** — append a `## User answers — <slug>` section via `mcp__obsidian-builtin__vault_append` (**not** `mcp__obsidian__patch_vault_file` — same em-dash heading caveat):
     ```markdown
     ## User answers — <slug>
     <verbatim answers to each question[], one per line>
     ```
   - **Other kinds** — append a `## Escalation resolution — <slug>` section (also via `mcp__obsidian-builtin__vault_append`) with the user's choice (retry / skip / action taken).
   
   **Stamp `escalated: true` (load-bearing for resume).** After appending the verdict block for a slug, **immediately** stamp `  - escalated: true` on that plan item via `mcp__obsidian-builtin__vault_patch` (mirroring the `speculative: true` precedent exactly — same durable-sentinel pattern, same tool). This sentinel is what Step 0.5 reconcile and Step 1.4 resume key off to route a `[~]`+no-`pr:` item to the 3d-esc continuation (re-enter with the `onlySlugs` filter) instead of the "stuck worker → ask resume/reset" default. Without it, a crash between appending the verdict block and re-spawning the worker leaves the item indistinguishable from a stuck non-speculative worker, and the reset path discards the worktree plus the human verdict. **Clear the sentinel** (remove the sub-line, via `mcp__obsidian-builtin__vault_patch`) when the continuation drives the item to `[m]` (parked) or `[-]` (skipped) — same lifecycle as `speculative: true`.
   
   These appended blocks are the **only** durable escalation state beyond the `escalated: true` sentinel. The continuation workflow reads the plan note, finds them, and injects the relevant block as the `## Design verdict` / `## User answers` section in the worker's re-spawn prompt (step 4 below). This is what makes the re-derivation checkable: if the block is present and `escalated: true` is stamped, the verdict was captured; if either is absent, the escalation is unresolved.

4. **Invoke the scoped continuation.** Re-invoke the **same Workflow** (`build-level.mjs`) with `args.onlySlugs = [<slug>, ...]` containing only the still-unresolved slugs (those whose worktrees are intact and for which a verdict block was just appended). Pass the **full `args.items` array** unchanged (the workflow's `onlySlugs` filter selects the active subset) plus a `verdicts` map:
   ```json
   {
     "verdicts": {
       "<slug>": { "kind": "<escalation-kind>", "verdict_section": "<the FULL appended markdown block — heading + body>" }
     },
     "onlySlugs": ["<slug>", "..."]
   }
   ```
   `verdict_section` is the **full markdown block** the orchestrator appended to the plan note for this slug — the `## Design verdict — <slug>` (or `## User answers — <slug>`) heading **and its body** — not merely the heading. The workflow injects it **verbatim** into the re-spawned worker's prompt, so the value must be self-contained (the worker has no access to the plan note). The workflow's `driveItem` for each continued slug re-spawns the worker with the original prompt **plus** that block appended as an extra `## Design verdict` / `## User answers` section (read from `args.verdicts[slug].verdict_section`), so the worker sees the human's decision instead of re-forking. The re-spawn does **not** re-run 3a (claim already held — `driveItem` skips it on a slug in `onlySlugs`) or 3b (**the existing worktree is reused** — `driveItem` skips `worktree.sh create`, whose force-remove-and-re-add would otherwise discard the escalated build; the `.build-guard` marker is still in place, so the guard stays armed without re-arming). It resumes at **3c** (worker re-spawn), continuing forward through 3d–3h as normal. Re-spawn model tier follows the 3c **escalate-on-retry** rule: `failed` → top tier; `blocked`/`design-fork` → keep the stamped tier.

5. **Loop until done.** Collect the continuation's `{parked, escalations}`. Write parked siblings' sentinels and remove their worktrees (step 1). If `escalations` is non-empty, loop back to step 2 (a re-spawned worker may discover a further fork or remain blocked). The loop terminates when `escalations` is empty (all items either parked or skipped/aborted). A stuck loop (the same slug escalates on consecutive iterations with no change in payload) is surfaced after 3 identical rounds — prompt the user to skip or abort the slug rather than looping indefinitely.

**Acceptance invariants (falsifiable):**
- A design-fork/blocked/failed escalation **re-drives its slug to parked** via a verdict-seeded continuation; siblings that are already `[m]` are left untouched (their worktrees were removed in step 1 and they carry `pr:` on the plan note — the `onlySlugs` filter excludes them).
- The continuation is **re-derivable from {plan-note level sentinels + `pr:`} + {the appended verdicts map} ALONE** — the exact inputs are: (a) every `[~]` item in this level with no `pr:` sub-line **and `escalated: true`** (= worktree intact, verdict captured, awaiting continuation — the sentinel stamped in step 3); (b) the `## User answers — <slug>` / `## Design verdict — <slug>` / `## Escalation resolution — <slug>` sections appended to the plan note. The `escalated: true` sentinel is the unambiguous discriminator — a `[~]`+no-`pr:` item without it is a stuck non-speculative worker (ask resume/reset), not an escalated item. No live workflow state is needed; these two durable sources reconstruct a continuation fully after a crash.
- Any non-enumerated terminal kind (not `design-fork`, `blocked`, `failed`) **surfaces as a generic escalation** with its raw `payload` rendered as a text block — it never stalls silently.

#### 3e. Optional pre-push review

If project CLAUDE.md `## Subagents` lists a review subagent matching the item's `review:` override, the item's change *kind* (`architectural` → `architecture-reviewer`), the extension/path-glob axis — `workflows/scripts/config/reviewer-routing.tsv` is the **single source of truth for that axis** (ADR 0008, `docs/adr/0008-reviewer-routing-tsv-extension-axis-scope.md`; consult the tsv for the full extension/glob → reviewer list, which includes the docs-prose glob) — or the prose fallback of that axis — **any other stranger-facing prose `*.md` not matched by a `reviewer-routing.tsv` row → `docs-reviewer`** (a non-extension catch-all ADR 0008 keeps prose-resident rather than a tsv row, which would over-match) — **except** a workflow spec under `claude/commands/*.md`, which always routes to `workflow-reviewer` regardless of any tsv row or the prose fallback (temperloop#1007; invariants, not style) — invoke as read-only pass. A change matching more than one axis (a `review:` override, the change kind, a tsv route, the prose `*.md` fallback, or the `claude/commands/*.md` exception) runs each matching reviewer. `docs-reviewer` is advisory only — its findings are a review pass like this step's other reviewers, never a `checks` gate entry (`scripts/quality-gates.sh` / `.github/workflows/ci.yml` name no LLM-judgment gate, by design: non-reproducible judgment doesn't belong in the deterministic set). **When invoking, pass the effective (merged) principle set** — the resolution 3c already computed for this item's `(repo, project)` pair (§ 3c "Effective engineering principles") — to the reviewer as **additional evaluation criteria** ("judge the diff against these too — cite the principle's name and origin, `kernel` or `project`, in any finding"). **Reuse that resolution; do not re-resolve it here** — the kernel set always applies, so there is no "no declared principles" branch left to fall into; an item's project slot being empty, or the section being empty/malformed, is a *resolved outcome* already carried forward from 3c; state it once, in the Step 6 summary's `principles:` line, never as per-item noise in this hot loop. Blocking issues loop back to 3c with the review feedback as context. If a matching reviewer is named but unavailable (not declared / not installed — capability-probe predicate), state the skip for the item rather than silently passing the gate, using the two-form degradation line (`claude/message-schema.md` § Degradation notice): bare `skipped — <agent> unavailable`, or — when the reviewer ships as source at `claude/agents/<agent>.md` (flat, e.g. `architecture-reviewer`/`docs-reviewer`/`requirements-auditor`/`workflow-reviewer`, or under `reviewers/`, e.g. the per-language reviewers) but isn't installed — the remedy-bearing `skipped — <agent> available as source; run workflows/scripts/install/project-agents.sh to enable`. **Mandatory for a `claude/commands/*.md` diff (temperloop#1007):** the `workflow-reviewer` pass is *required*, not worker-discretion — for any item whose diff touches a workflow spec under `claude/commands/*.md` it MUST either run or emit the legible skip notice per the two-form rule above (`skipped — workflow-reviewer unavailable`, or — since `workflow-reviewer` ships as source at `claude/agents/workflow-reviewer.md` — the remedy-bearing `skipped — workflow-reviewer available as source; run workflows/scripts/install/project-agents.sh to enable` when it isn't installed), and is **never** omitted or self-run informally. Tally each command-doc item's reviewer outcome (`ran` / `skipped — unavailable`) for the Step 6 summary, so coverage is legible at session end. The cross-PR coverage rate over merged command-doc PRs is reported out-of-band by `workflows/scripts/workflow-reviewer-coverage.sh` (a reporting rollup, **not** a `checks` gate — LLM-judgment gates stay out of the deterministic set by design).

#### 3e.5. Parent-side acceptance gate (no-op if the project declares no gates)

Before pushing, the orchestrator independently verifies the worker's claims rather than trusting the return JSON. Run the project's **single source of truth for the static gate set — `scripts/quality-gates.sh`** (the same script CI and CLAUDE.md § Dev workflow invoke, so the acceptance gate can't drift from a hand-maintained inline copy; it runs the gates **bare** and **repo-wide**, **not** path-scoped) against the worker's commit, in a throwaway checkout of the worker SHA (or the worker's worktree, before 3h removes it). A path-scoped green that masks a repo-wide failure is the PR #309 silent-red class this gate exists to catch. **No-op on a project that ships no `scripts/quality-gates.sh`** (e.g. a no-CI repo, or foundation itself).

- **Fail** → loop back to 3c with the gate output as context; do **not** push a known-red branch.
- **Pass** → proceed to 3f.
- **`set -o pipefail` / `${PIPESTATUS[0]}` is mandatory whenever the gate is piped (temperloop#68).** The Pass/Fail verdict is the gate's **own exit code** — so it must not be swallowed. If you capture the gate's output through a filter or `tee` (e.g. `scripts/quality-gates.sh 2>&1 | tee /tmp/qg.log`), a bare pipe's exit status reflects the **last** stage (`tee`'s `0`), **masking a red gate**: the gate degrades to a silent no-op and a locally-red branch sails through to push, leaving CI (3g) as the only catch — the exact PR #309 silent-red class this gate exists to pre-empt. So either run the gate **un-piped** and branch on its exit directly, or `set -o pipefail` first (or read `${PIPESTATUS[0]}`, the gate's own exit, not `$?`) before deciding Pass/Fail. The `--workflow` path's §3e.5 invocation in `claude/workflows/build-level.mjs` already bakes this in (`set -o pipefail` prefixes the gate command); a hand-run conversational-path gate must do the same. Do **not** hand-edit either invocation to pipe the gate without carrying pipefail through — that silently reintroduces the swallow.
- **The gate runs hermetically — the pipeline's own `build.config.sh` value-knobs are scrubbed first (temperloop#1241).** The `--workflow` §3e.5 invocation in `claude/workflows/build-level.mjs` `unset`s the knob set `workflows/scripts/build/build-config-knobs.sh` prints (SSOT-derived from `build.config.sh`'s own `: "${NAME:=…}"` declarations) *before* running the gate, so the suite runs at **tracked defaults** — matching what CI's `checks` job sees — instead of inheriting the funnel-drive session's ~40 exported knobs. Without the scrub those exports win the config-precedence rung (`env > machine-conf > repo-local > tracked-default`) and false-**Fail** the precedence tests (`test_config.sh` / `test_stranger_config.sh` / `test_funnel_cron.sh`) on a change CI passes green — the #1241 merge-tier blocker. The two config-**file** resolvers `BUILD_CONFIG_MACHINE` / `BUILD_CONFIG_LOCAL` are deliberately **not** scrubbed: a file-based machine-local leak is a distinct mechanism (temperloop#1055) an env scrub can't fix. A hand-run conversational-path gate is not exposed today (its fresh Bash calls don't inherit the funnel exports), but must preserve this hermeticity if that ever changes — mirror the scrub, exactly as with pipefail above.
- Require the worker's `acceptance_results` evidence to be **machine-checkable** (exact test node id / command + exit code), not prose — tighten the 3c prompt accordingly.
- Scope to the fast **static** gates here; the slow pytest/eval still runs in CI (3g) where the project has it. No-op on projects that declare no gates (e.g. a no-CI repo).

#### 3e.6. Class-A activation gate — confirm the built thing is actually wired in (no-op unless the item declares `activation: class: A`)

"Merged + CI-green" proves the artifact is *correct*, not that it is *reachable* — a new runner never registered, a flag left off, a rollup computed but never rendered all sail through 3e.5 (see `Decisions/temperloop - Activation-completeness contract`). This gate closes the **synchronous / in-repo** class of that gap; the cross-repo (class B) and soak (class C) classes are ledger-discharged and out of scope here (temperloop#317).

Applies **only** when the item carries an `activation:` block with `class: A` (plan-schema § activation). No-op otherwise — an item with no activation block, or a `class: B`/`C` block, skips this step.

- **Explicit predicate** — if the block carries a `proof:` predicate, run it against the worker's worktree (the same throwaway checkout 3e.5 used, before 3h removes it), from the worktree root. `proof:` reads **false until the built code is genuinely reachable** — `grep -q GeminiRunner evals/runners/__init__.py` (registered), `test "$(cfg get feature.x.enabled)" = true` (flag flipped), a one-line drive that exits non-zero when the panel is absent. Exit 0 = activated.
- **No predicate → drive it** — a `class: A` block with no `proof:` means "activation is real but not expressible as a static check": invoke the **`/verify` skill** on the item's touched flow (`files:`) — it exercises the change end-to-end and observes behavior, which is exactly the activation signal. `/verify` exists precisely for this and is otherwise never invoked by the pipeline.
- **Fail** → loop back to 3c with the activation output as context; do **not** push a branch whose feature is dormant. The worker's fix is to add the missing wiring (register / flip / render), never to weaken the predicate.
- **Pass** → proceed to 3f.
- **Legible degradation (per § Legible agent-gate degradation).** If the block declares `class: A` with no `proof:` **and** `/verify` is unavailable in this environment (capability-probe false), do not silently pass — state `skipped — /verify unavailable` for the item and tally it in the Step 6 summary, so a dormant-but-merged feature is visible at session end rather than masked.
- **`set -o pipefail` / `${PIPESTATUS[0]}`** applies here exactly as in 3e.5 whenever the `proof:` predicate is piped — the Pass/Fail verdict is the predicate's own exit code and must not be swallowed by a `tee`.

#### 3f. Push and open the PR

From the **parent checkout** (not a worktree — the worker's commit is in the shared object store; for an item carrying `repo:`, this is the **same cross-repo `<repo-root>` resolved in 3b**, not the plan's own checkout — the worktree shares its object store, so push/PR-open must target the same repo the worktree was created against), via the deterministic-spine script `workflows/scripts/build/pr.sh` (epic #253: each mechanical step below is a pure function of observable machine state with a closed outcome set, so its mechanics live in code — the orchestrator invokes the subcommand and branches on the structured `.outcome`; the judgment-shaped halves — rewording an offending commit, the `BASE_STALE` rebase/conflict handling, branch-collision triage — stay orchestrator-driven here):
0. **Closing-keyword pre-push backstop (enforce, don't merely prompt).** Before pushing, scan the worker's commit message(s) for issue-closing keywords — the prose prohibition in 3c is backstopped here so a slip is caught structurally:
   `workflows/scripts/build/pr.sh scan <worktreePath>`
   → `SCAN_CLEAN` (proceed) or `SCAN_BLOCKED` + the offending lines (non-zero exit; the script greps the `origin/<default>..HEAD` commit messages for the closing-keyword pattern). `SCAN_BLOCKED` means a worker commit message carries a closing keyword (the `ec8d5fd` failure mode — GitHub auto-closes the issue on default-branch merge regardless of the PR body). Do **not** push such a branch as-is: either re-spawn the worker against the same `worktreePath` (the `.build-guard` marker is still in place, so the guard stays armed) instructed to amend/reword the offending message to a plain description and re-commit (no push), or strip it locally (`git -C <worktreePath> commit --amend`/`rebase --reword`) — then re-run the scan until `SCAN_CLEAN` before the push-by-SHA below. Issue linkage belongs in **one place only** — the orchestrator's PR-body `Closes #<gh_issue>` (step 2 below), never a worker commit.
0.25. **Unconditional rebase onto fresh `origin/<default>` — the stale-base guard (#525).** Independent of speculation: **every** worker branched off `origin/<default>` at the start of its run, and on a fast-moving default a long run lets the default advance mid-build — so by push/PR-open time the worker's base may be stale and a straight push lands a PR whose cumulative diff **reverts whatever merged in between** (the W49 PR#82 / W52 PR#83 stale-base reversions). Before the scan/push, `build-level.mjs` (3f) runs `workflows/scripts/build/pr.sh rebase <worktreePath>` to fetch the default fresh and replay the worker's commits onto its tip (a no-op when already current):
   - **`REBASED`** → the base is now current; the PR diff carries ONLY the worker's own changes. Proceed.
   - **`REBASE_CONFLICT`** → pr.sh has already `git rebase --abort`ed (worktree left clean, the worker's commits intact, **never** a silent revert). build-level escalates `rebase-conflict` for a human to resolve — it does **not** push the stale SHA. This is the always-on counterpart to the speculative-only `base-check`/`BASE_STALE` flow below (which stays for the speculative-promotion path); the rebase here makes the guard unconditional so a non-speculative long worker on a fast-moving default never reverts intervening merges.
0.5. **Speculative-base currency check — load-bearing (speculative workers only; the sufficient half of the speculation contract).** When this worker was **speculatively pre-spawned** during the prior level's merge-wait (per "Speculative next-level execution" in the within-level section), its worktree was branched from `origin/<default>` *before* that level merged — so its base may now be **stale** (the prior level advanced `main`). Skip this step for a non-speculative worker (its base is current by construction — 3b branched it from the freshly-advanced `origin/<default>` per 4e). For a speculative worker, **before the push-by-SHA below**, check base currency against the now-advanced default tip:
   `workflows/scripts/build/pr.sh base-check <worktreePath>`
   → `BASE_CURRENT` / `BASE_STALE` (the script fetches `origin/<default>` and compares `merge-base(HEAD, origin/<default>)` against the advanced tip). The check is mechanical; what follows a `BASE_STALE` — the rebase and its conflict handling — stays orchestrator-driven:
   - **`BASE_CURRENT`** (merge-base == advanced tip) → the base is up to date; proceed to push-by-SHA (step 1).
   - **`BASE_STALE`** (merge-base behind tip) → do **NOT** push the stale SHA — pushing a stale speculative base **silently drops the merged level-k changes in overlapping regions, and CI on the speculative head won't catch it** (the failure this check exists to prevent). Bring it current first:
     - **Rebase-then-reverify (preferred) — push BEFORE watching CI:** `git -C <worktreePath> rebase origin/<default>`. The rebase changed the diff, so the prior green is stale and must be re-earned — but **CI check-runs only exist after a push**, and an **unpushed** SHA is not the same case 3g's `NO_CI` outcome covers (temperloop#605): the remote has never heard of an unpushed local SHA at all, so `ci-poll.sh` querying `commits/<sha>/check-runs` against it 404s and resolves to a hard `ERROR` (after `gh_retry`'s bounded retry budget) — **never** `NO_CI` and never a green. `NO_CI` fires *only* for a SHA the remote genuinely has (a real, pushed commit) that happens to carry zero check-runs configured against it (e.g. this repo has no CI at all). This is exactly why push-before-watch stays mandatory post-#605: watching the unpushed local SHA would not let an un-reviewed diff quietly wave through the merge gate as a benign no-CI skip (the 404→`ERROR` path stops that outright) — but it would still be polling a SHA that doesn't exist anywhere but this worktree, which is the wrong thing to watch regardless. So order the steps so CI is always watched on a **pushed** head, never an unpushed local SHA. On a clean rebase:
       1. **Force-push the rebased SHA** to the plan branch: `workflows/scripts/build/pr.sh push <worktreePath> <item.branch> --force`. (Force is required — the rebase rewrote history relative to any prior push of this branch.)
       2. **Re-run the 3e.5 parent-side acceptance gate** against the rebased SHA — this is the local static gate (no network), so it may run before or after the push; either is fine.
       3. **Watch CI (3g) on the pushed head** — the just-pushed rebased SHA, never the unpushed local SHA.
       Only after both the 3e.5 gate passes *and* CI re-confirms green on the pushed head does the speculative worker promote (it is already pushed; record the `pr:`/`pushed_sha:` pointer per step 3 — no second push-by-SHA in step 1 is needed for this path, the force-push above already placed the branch).
     - **Discard-and-respawn (fallback on rebase conflict):** if the rebase conflicts, remove the worktree (`workflows/scripts/build/worktree.sh remove <repo-root> <item.slug>`), **re-create it from fresh `origin/<default>`** per 3b (`worktree.sh create` — re-drops the marker), and **re-run the worker** (3c) against the current base — then proceed normally through 3d–3e.5 and this step again (now `BASE_CURRENT`).
   - **Promote to a real PR only after the currency check passes.** This is the sufficient half the eligibility predicate ("no `depends-on`") is *not* — it catches the overlapping-but-independent-lines case eligibility can't see. See [[Decisions/foundation - Speculative next-level execution worktree-isolation contract]].
1. Push the worker's commit to the plan branch **by SHA**:
   `workflows/scripts/build/pr.sh push <worktreePath> <item.branch>`
   → `PUSHED {sha, branch}` — pushes the worktree's `HEAD` by SHA (`<worktreePath>` is the **deterministic path the orchestrator pre-created in 3b** (`<repo>.wt/<item.slug>`), not a value the worker reports; its `HEAD` is the worker's top commit). This honors the plan's `branch:` name regardless of the worktree's throwaway `build/<item.slug>` local branch.
   - **Remote-branch collision** → `PUSH_REJECTED {error}` + non-zero exit: if it's a stale branch from a prior aborted run, re-run with `--force` after a quick check; otherwise surface to the user (pick a new branch name → patch the plan's `branch:`, or abort).
2. Open the PR — write the worker's verdict JSON (the 3d return object) to a temp file and run:
   `workflows/scripts/build/pr.sh open --repo <repo-root> --branch <item.branch> --title "<item.title>" --verdict <verdict.json> [--gh-issue <N>] [--also-closes <M,M2,…>] [--verification-surface-file <worktreePath>/.build-verification.md] --plan-link "Plans/<date> <project> - <title>#<item.slug>" --source "<item.source>"`
   Pass `--verification-surface-file <worktreePath>/.build-verification.md` when the worker set `verification_surface_path` (the #418 inflow-cut). **The orchestrator uses the deterministic path it pre-created in 3b — not the verdict's `verification_surface_path` value** — so the surface body never enters orchestrator context; `pr.sh` emits a structured ERROR if no file exists at that path, so a non-standard worker that wrote elsewhere surfaces immediately rather than silently. Omit the flag when the worker produced no surface; `pr.sh` then falls back through the verdict's `verification_surface_path` → inline `verification_surface` → the acceptance recap.
   → `PR_OPENED {pr_number}` (`--verdict -` reads stdin; `--body-only` prints the assembled body without creating the PR — for inspection, and the dry mode the build tests assert on). The script assembles the body from the verdict's `summary` / `acceptance_results` + the resolved verification surface plus the plan fields, per this contract:
   - The agent's `summary`
   - **If the item has `gh_issue:`** (pass `--gh-issue`), a literal `Closes #<gh_issue>` line — **not** wrapped in backticks / a code span. GitHub silently ignores closing keywords inside code spans (observed in plan #265/#266, where a backticked `Closes` failed to auto-close and the issues had to be closed by hand). This is what auto-closes the issue on merge; the 4d board→Done move is the belt-and-suspenders complement. **This PR body is the ONLY place an issue-closing keyword belongs** — the orchestrator owns issue linkage here; worker commit messages must never carry one (forbidden in 3c, enforced by the 3f step-0 pre-push scan above). **Cross-repo `repo:` honor point:** when the item's `gh_issue:` is tracked in a *different* repo than the one this PR opens against (the plan's home repo, tracking a kernel-classified item's issue — the mirror image of the `repo:` case, since `gh_issue:` normally lives wherever the item was triaged), pass the fully-qualified `owner/repo#N` form instead of the bare number, so `pr.sh` emits `Closes owner/repo#<gh_issue>` — never bare `Closes #N`, which is **same-repo only** and silently fails to reach a tracker in another repo (per the global `CLAUDE.md` § Issue linkage cross-repo-closes caveat). The common, same-repo case is unaffected — bare `Closes #<gh_issue>` as before.
   - **If the item has `also_closes:`** (a list of additional issue numbers; pass `--also-closes`), the script emits **one bare `Closes #M` line per entry**, each on its own line, **alongside** the primary `Closes #<gh_issue>` above — never combined (`Closes #1 and #2` closes only #1; GitHub requires each issue carry its own keyword). Same bare-text rule: no backticks / code span. This is how one PR closes several issues atomically on merge — the opportunistic same-PR fix (a `capture.sh`'d defect fixed alongside the planned work) and the root-cause-collapse atomic close (a survivor's fix PR closing the survivor plus its absorbed symptoms together). `also_closes:` absent emits nothing extra. Any `also_closes:` entry tracked in another repo is qualified the same way as `gh_issue:` above (`owner/repo#M`).
   - Bulleted recap of `acceptance:` items with `passed/evidence`
   - A `## Verification` section = the **resolved** verification surface (per-change-type proof: before/after numbers, per-fixture diffs, config side-by-side, or a no-behavior-change claim + proving diff). `pr.sh` resolves it by precedence — `--verification-surface-file` → the verdict's `verification_surface_path` (the file the worker wrote, #418) → the inline `verification_surface` (back-compat) → the acceptance recap. The recap fallback fires **only** if the worker produced no surface at all — the bare recap alone does not satisfy the project's PR-verification-surface rule. **Surface the degraded case (don't let it fail silently):** when the orchestrator passed no `--verification-surface-file` AND the verdict carries neither `verification_surface_path` nor an inline `verification_surface`, emit a named warning at 3f — `PR #N: no verification surface, acceptance recap only — does not satisfy the PR-verification-surface rule` — and tally such items in the Step 6 summary. This catches a worker that missed the file-write instruction (e.g. a crash-resumed worker built from a stale prompt) *before* CI, while the PR body is still editable, instead of leaving the reviewer to discover it.
   - Back-link: `Tracked in: [[Plans/<date> <project> - <title>#<item.slug>]]` (pass the wikilink target via `--plan-link`)
   - Source ref: `Derived from: <item.source>` (pass via `--source`)
   - Standard `🤖 Generated with [Claude Code]` footer.
3. Capture the PR number, then **immediately** record durable PR identity on the item — `mcp__obsidian__patch_vault_file` to add `  - pr: <N>` and `  - pushed_sha: <worker-sha>` sub-lines — *before* starting the CI watch (3g). The sentinel stays `[~]`, but the item now carries a recoverable pointer to the open PR (Step 1.4 resume re-attaches via `pr:` instead of re-spawning; Step 0.5 reconciles against it). **If the item carried a `  - speculative: true` sentinel** (it was speculatively pre-spawned, per 3b), **clear it now** — remove the sub-line in the same patch. Recording `pr:` is the promotion event; past it the item is an ordinary `[~]` whose `pr:` pointer governs resume, so the speculative sentinel must not linger. **Also emit an issue-touch record for this PR-open** (foundation #916/#919, the issue-touch-stream — skip when the item carries no `gh_issue:`, there being no issue to attribute the touch to): `workflows/scripts/emit-issue-touch.sh --repo <owner>/<repo> --issue <item.gh_issue> --kind pr-open || true`. Telemetry only — `|| true`-isolated so a missing/unwritable raw lake never blocks the PR-open path; its presence here is mechanically enforced by `workflows/scripts/validate-issue-touch-emit.sh` (wired into `scripts/quality-gates.sh`). Claim touches are already covered by `claim.sh`'s own emit (3a) and are never duplicated here.

#### 3g. Monitor CI

Watch CI with the toolkit's CI poll (deterministic spine, epic #253), run via `Bash { run_in_background: true }`:

```bash
workflows/scripts/build/ci-poll.sh <owner>/<repo> <pr>
```

**Headless one-shot runs (`FUNNEL_OPERATOR_ABSENT=1`) — run the poll in the FOREGROUND, never backgrounded (#626).** A `claude -p` headless session has **no re-invoke-on-background-completion loop**, so a backgrounded poll would yield and end the session before CI finishes — the bug that left funnel PRs green-but-never-merged (the full reason, why `FUNNEL_OPERATOR_ABSENT=1` is the marker, and why it is *narrower* than the `--unattended OR FUNNEL_OPERATOR_ABSENT=1` operator-absent predicate — the interactive walk-away `--unattended` case has a working loop and keeps the backgrounded/timed path: [[Decisions/foundation - build headless (claude -p) foreground merge — no re-invoke loop (#626)]]). All three #626 foreground branches (here, 4b timed gate, 4b MERGED-poll) key on `FUNNEL_OPERATOR_ABSENT=1`. When `FUNNEL_OPERATOR_ABSENT=1`, instead run `ci-poll.sh` as an ordinary **blocking** `Bash` call (it loops to a terminal outcome itself) with an explicit bounded `--timeout "$BUILD_HEADLESS_POLL_TIMEOUT"` (under the ~10-min Bash foreground cap): `ci-poll.sh <owner>/<repo> <pr> --timeout "$BUILD_HEADLESS_POLL_TIMEOUT"`. The session itself is un-timeout'd, so if CI is still pending at the bound **re-issue the foreground poll once more before parking** (one retry within 3g, not zero — most CI finishes inside a second window); if it is *still* pending after that retry, park `[m]` with the `pr:` pointer and **stop this item** — the `funnel-merge-pending` hand-off marker (#624) resumes it next tick when CI is long-green. Do **not** yield expecting a re-invoke. (This branch is headless-only; operator-present runs keep the backgrounded two-sweep poll above unchanged.)

The script owns the poll mechanics: it resolves the PR head SHA **once** over REST, polls `commits/<sha>/check-runs` at a **≥30s** interval (gh api calls are core-REST, but there's no reason to poll tighter than CI completes; `--interval <secs>` overrides), and emits ONE structured JSON line from a **closed outcome set** — the orchestrator branches on `.outcome`, never parses prose. **Do not hand-roll the poll, and never use `gh pr checks --watch`** — that is GraphQL-backed and polls the *scarce* Projects-v2-shared GraphQL bucket every few seconds; with parallel fanout that's N concurrent GraphQL pollers and a primary driver of board rate-limit exhaustion (GH #53). CI check state is on the **core REST bucket** (roomy, barely used), so the script polls there and leaves GraphQL for the board ops that have no REST equivalent. Flags: `--sha <sha>` pins the head explicitly (use it on a re-poll right after a force-push, before the PR object catches up); `--timeout <secs>` (default 3600) bounds the watch.

On the script's outcome:
- **`{"outcome":"CI_GREEN",…}`** → 3h.
- **`{"outcome":"CI_FAILED","failed_run_ids":[…],…}`** → the script reports *what* failed; **diagnosis — flake vs real, and what to tell the re-spawn — stays an LLM seat (the orchestrator's judgment), never the script's.** Inspect each failed run (`gh run view <run-id> --log-failed`, ids straight from `failed_run_ids`); re-invoke `Agent` against the **same pre-created `worktreePath`** (the `.build-guard` marker is still in place, so the guard stays armed), instructed to first `git fetch origin <item.branch> && git reset --hard FETCH_HEAD` (build on the pushed work), then "CI failed; fix and commit; do not push." Then the orchestrator **force-pushes** the new SHA — `git push -f origin "$(git -C <worktreePath> rev-parse HEAD)":refs/heads/<item.branch>` — and re-runs `ci-poll.sh` on the new head, **pinning the head with `--sha <the same SHA you just force-pushed>`** (reuse the `git -C <worktreePath> rev-parse HEAD` value computed for the `git push -f`; loop within 3g). **Do NOT let the poll re-resolve the head from the PR API after a force-push** (i.e. never re-read `gh api repos/<owner>/<repo>/pulls/<pr> --jq .head.sha`): GitHub's `pulls/{pr}` object can briefly lag and still return the **stale pre-push head**, whose check-runs are already green from the prior run — so a re-poll on that SHA prints a **false `CI_GREEN`** and advances the item to merge without ever running CI on the fixed code (#254, observed in the #508 retro). The locally-pushed SHA is authoritative by construction; the API round-trip only adds the race. (The *initial* 3g poll above takes no `--sha` and lets the script resolve `head.sha` once — that read is race-free because no force-push has occurred.)
- **`{"outcome":"NO_CI","waited":…,…}`** (temperloop#605) → zero check-runs ever appeared on the head SHA through the script's bounded grace window (never the full poll window) — a repo with no CI configured, not a hang. This is **not a failure and not a TIMEOUT**: emit a legible skip notice — `PR #N: no CI configured on this SHA — skipping the CI gate` — then **→ 3h (all of its steps 0–4 apply unchanged)**, with the skip recorded in the level summary (4a) alongside the parked item so it is never mistaken for a confirmed-green PR. Never re-enter 3g for this outcome and never block the item on it.
- **`{"outcome":"TIMEOUT",…}` / `{"outcome":"ERROR",…}`** (non-zero exit) → the poll never completed with CI genuinely pending or unreachable (zero-check-runs no longer routes here — that's `NO_CI` above). Never treat it as green: surface it, fix the cause (push the head, or re-run the poll), then re-enter 3g.

#### 3h. Park as `[m]` and move on

PR is green — **or** 3g resolved `NO_CI` (temperloop#605: no CI configured on this repo, a legible skip, not a failure). No human gate yet — that fires at the end of the level. Do all of, **in this order** (record durable success *before* destroying recovery material):

0. **Parent-checkout cleanliness assertion — per-worker leak backstop (Option C, the detective layer).** `git status --porcelain` in the parent checkout, run **once per worker** (in the parallel fanout, at the join — see the within-level execution note). It must be **empty**: the guard hook (Option B) should have *prevented* any out-of-worktree write, and a pre-created worktree (Option A) means the worker never had a reason to touch the parent — so a dirty parent here means a leak slipped past both (e.g. the guard was un-armed, or a write path the guard's allow-list permitted). **Attribute and auto-discard by committed paths (bisect-by-committed-paths):** list the dirty paths (`git status --porcelain | awk '{print $2}'`), and for each, check whether it is a subset of the paths the worker already committed to its branch (`git -C <worktreePath> diff --name-only origin/<default>...HEAD`). A dirty path that **matches** a committed path is a confirmed leak of that worker's own edit — `git checkout -- <path>` to discard it (the committed copy on the branch is canonical). A dirty path that does **not** match any committed path is unexplained — surface it to the user rather than blind-discarding. Then tighten the next worker prompt's worktree-path instruction and confirm the worktree's `.build-guard` marker was present (the guard arms from it; a missing marker means 3b didn't run via `worktree.sh create`). Do **not** proceed with a dirty parent tree — Step 0's dirty-tree guard runs only at run start, so an uncaught leak here rides silently into the next item's branch.
1. **Record success first.** `mcp__obsidian__patch_vault_file` on `$1`: `[~] **<title>** ...` → `[m] **<title>** ... — PR #N parked for batch merge` (the `pr:`/`pushed_sha:` sub-lines from 3f are already present). Doing this *before* worktree removal guarantees a crash never leaves the gap "state says `[~]` but the worktree/evidence is gone." **If 3g resolved `NO_CI`** (temperloop#605), also stamp a `  - no_ci: true` sub-line in the same patch — same durable-sentinel pattern as `pr:`/`pushed_sha:`/`speculative: true`. This is what lets Step 1.4's resume revalidation and the 4a summary tell "no checks were ever going to appear" apart from "checks appeared and are stuck/gone" for a parked item with no `statusCheckRollup` to point to.
2. **Board integration ON — best-effort marker clear; the claim is HELD until Done.** The board claim legitimately **stays In Progress** through the park — this is the sanctioned **claim-until-Done** flow (K#275, CLAUDE.md § Park, don't abandon): a parked `[m]` item is not done, so its claim is *held*, and the board half only leaves In Progress via the close→Done cascade on merge (4d) or a deliberate skip. Clearing the local terminal marker here is therefore an **optional convenience, never a required step**: run `"$RELEASE" <gh_issue>` (the entrypoint resolved by the Step 0 probe) to clear the orchestrator window's claim marker (status bar → "No Issue Claimed") if it clears cleanly, but **do NOT fail or block the park on a non-zero release.** `release.sh` is per-window and one-marker-per-window, so in a **multi-claim window** — the documented within-level parallelism, where one session claims several items in a level — the marker holds only the *latest* claim and `release.sh` will refuse a *non-latest* issue (`this window holds a claim for #B, not #A — refusing`). That refusal is **expected and non-fatal**: leave the earlier claim held until Done; `reconcile.sh` and the close→Done cascade catch it. The next item's 3a claim re-sets the marker regardless.
3. **Remove the orchestrator-created worktree LAST** (pure scratch once the SHA is pushed in 3f) at its **deterministic path** from 3b: `workflows/scripts/build/worktree.sh remove <repo-root> <item.slug>` — removes the `.build-guard` marker, the worktree (force semantics built in, covering any uncommitted scratch the worker left behind), and the throwaway `build/<item.slug>` branch; structured outcome `REMOVED` (or `NOT_FOUND` on a re-run). `gh pr merge` at the gate is a remote op from the parent checkout, so no worktree is needed past this point. If the run aborts before this step, the leftover worktree is recoverable via `worktree.sh prune <repo-root>` (and the path is re-derivable from the slug) — strictly safer than losing the `[m]` record.
4. Move to the next item in this level. If this was the last item, fall through to Step 4 (batch merge gate).

## Step 4 — Batch merge gate (per level)

Fires once all items in the current level have reached terminal-for-level state (`[m]`, `[v]`, `[-]`, or aborted). Spike items are already terminal at `[v]` (closed in the per-item kind-fork) and do not enter the gate. For each `[m]` item:

### 4a. Surface the level summary

Print a single block to the user before the question, e.g.:

```
=== Level 0 merge gate: 6 parked PRs (backend: MANAGED · strict) ===
[m] sweep-watchdog-orphan-sleep         PR #201  CI ✓                    MERGEABLE/CLEAN
[m] tier2-baseline-partial-promote      PR #202  CI ✓                    MERGEABLE/CLEAN
[m] b2b-artist-preservation             PR #203  CI ✓                    MERGEABLE/CLEAN
[m] diff-value-render-consistency       PR #204  CI ✓                    MERGEABLE/CLEAN
[m] adapter-first-ci-run                PR #206  CI —  (no CI configured) MERGEABLE/CLEAN
[m] gemini-35-flash-detected-lang       PR #205  CI ✓                    CONFLICTING/DIRTY
# MANAGED+strict ⇒ the 5 CLEAN PRs (incl. #206's no-CI skip) merge serially via gate.sh managed-merge --strict (update-branch + re-CI each after the first); #205 → 4c
```

Mergeability comes from `gh pr view <pr> --json mergeable,mergeStateStatus` per PR — surface **both** fields: `mergeable` (MERGEABLE / CONFLICTING / UNKNOWN) and `mergeStateStatus` (CLEAN / BEHIND / BLOCKED / DIRTY / …). If `mergeable: UNKNOWN`, wait ~3s and re-poll once before classifying it as non-mergeable. Note `mergeStateStatus` can **lag** right after a CI pass — a PR whose parent already *is* `main` HEAD may briefly read `BEHIND`; don't act on a single stale read. **Revalidate liveness too:** include `state,statusCheckRollup` in that read — a `[m]` parked in a prior session whose PR was since closed/merged out-of-band or went red must be surfaced (re-open / skip / re-run), never presented as a stale green tick (cf. Step 1.4).

**A `NO_CI` item (the item's `  - no_ci: true` sentinel, stamped by 3h per temperloop#605) renders distinctly in this summary — `CI —  (no CI configured)`, never `CI ✓`** — so a genuinely-untested-by-CI PR is never visually indistinguishable from a confirmed-green one at the one point a human (or a log reviewer) sees the set before it merges. This distinction matters most on the `FUNNEL_OPERATOR_ABSENT=1` headless immediate-merge path (3g, 4b) — nobody reads the transcript live before that merge fires, so the rendered summary line is the *only* record of "this PR's mergeability was never actually vetted by CI" surviving into the level's audit trail.

**Detect the merge regime here, before any merge** — it decides 4b. Two scripted probes run here, both read-only against `main`, both returning closed JSON the orchestrator branches on (never prose-parsed):

- **`backend` probe (scripted — selects the 4b merge path).** Run `workflows/scripts/build/gate.sh backend <owner>/<repo>`. Closed outcome set: `{"outcome":"NATIVE"}` | `{"outcome":"MANAGED"}` | `{"outcome":"MANAGED","probe_failed":true}` (always exit 0). Mechanics (as built): the `BUILD_MERGE_BACKEND` knob (`build.config.sh`) short-circuits an explicit `native`/`managed` override **without probing**; otherwise the script probes the branch ruleset (`repos/<owner>/<repo>/rules/branches/main`) for a `merge_queue` rule. **A probe failure resolves to `MANAGED`, never `NATIVE`** (fail-safe: MANAGED never silently arms a merge nobody chose), flagged `probe_failed:true` so "no queue" and "couldn't tell" stay distinguishable — **surface `probe_failed:true` in the 4a summary line** (e.g. `backend: MANAGED (probe failed — could not read branch rules; set BUILD_MERGE_BACKEND=native if this repo has a queue)`), never swallow it. `NATIVE` → the 4b native-queue path; `MANAGED` → the 4b managed path.
- **`strict` probe (scripted — picks the managed sub-flag only).** Run `workflows/scripts/build/gate.sh strict <owner>/<repo>` → `{"outcome":"STRICT"}` | `{"outcome":"NON_STRICT"}` (reads `required_status_checks.strict`; a 404 *"Branch not protected"* → `NON_STRICT`). Its **sole remaining role** is selecting `gate.sh managed-merge`'s flag on the MANAGED backend: `STRICT` → `--strict`, `NON_STRICT` → `--non-strict`. On `STRICT` the managed batch merges **serially** (each merge advances `main`; each subsequent PR's `--strict` update-branch re-runs CI) — say so in the summary so the serial cadence (≈ one CI-length wait per PR after the first) isn't a surprise. On the NATIVE backend the probe is informational only (the queue itself enforces the up-to-date guarantee).

**The `backend` axis is the SUCCESSOR of the old 4a `queue` axis** — the regime table is now 1×2:

| backend | 4b merge path |
|---------|----------------|
| NATIVE | Native queue: `gh pr merge <pr> --auto` per PR (no `--queue`/`--merge`/`--delete-branch` — the queue owns strategy + branch lifecycle); the queue's `merge_group` trial branch serializes/batches and updates branches itself. `update-branch` nudge RETIRED. **The live state of both org build repos** (queues stood up after the #330 org migration; see [[Decisions/foundation - build native merge-queue adoption]]). |
| MANAGED | Managed serial loop (temperloop#13 — a repo with no native queue, e.g. a free personal repo that can't provision one): orchestrator-driven per-PR `gate.sh managed-merge <owner>/<repo> <pr> --strict\|--non-strict` calls in plan-item order (flag from the `strict` probe above). See 4b "Merge — MANAGED backend". |

**Migration note (retires the old cell names):** the old 2×2's `queue=ON` cells **C/D** ≡ `NATIVE`; its `queue=OFF` cells **A/B** are **absorbed by `MANAGED`**, with the strict/non-strict split demoted from a path-selecting axis to `managed-merge`'s `--strict|--non-strict` sub-flag — `--non-strict` keeps cell A's cheap immediate-merge cost profile (no update-branch, no CI re-poll), `--strict` replaces cell B's hand-rolled `--auto` + `update-branch`-nudge serialization with scripted per-PR mechanics. There is no third axis over the old 2×2.

(`backend`/`strict` select only the *mechanical* merge path — they are **orthogonal** to the block-vs-timed regime computed next, which decides whether the run blocks for a human first.)

**Compute the block-vs-timed regime here too — it decides whether 4b is modal or timed.** Per [[Decisions/foundation - Timed merge-gate auto-merge (supersedes never-auto-merge-main)]], the selected set is either **risky → hard-block (modal, explicit approval, never timed-auto-merge)** or **clean-disjoint-independent → timed (countdown then auto-merge unless objected)**. The set is **risky** iff **ANY** of the following structural conditions holds — evaluate them mechanically over the selected PRs, no judgment call:
- **(a) Overlapping *hunks*** — two selected PRs touch the **same file AND their changed line ranges actually intersect**. Same-file-touch alone is **NOT** risky: git's 3-way merge auto-resolves disjoint hunks even within one file, so "both touch `foo`" ≠ "will conflict" — this predicate over-approximated conflict as file-collision and forced the modal gate on sets that merge perfectly cleanly (e.g. L1's #317/#318, two disjoint same-file PRs that tripped the gate then merged clean). The check is now two-tier:
  - **File-overlap probe (cheap, advisory).** For each selected PR, `git diff --name-only origin/main..<branch>`; compute the pairwise path intersection. A non-empty intersection is **surfaced as an advisory note** in the 4a summary (e.g. `note: PRs #X and #Y both touch claude/commands/build.md`) but does **NOT** by itself make the set risky — a same-file/disjoint-hunk set stays **timed-eligible** (auto-merges after the window unless objected, or lands via the native queue on a `NATIVE` backend).
  - **Hunk-overlap probe (only on a same-file pair → decides risky).** For each pair that shares a file `F`, test whether their changes to `F` genuinely conflict. Run a trial 3-way merge per shared file — `git merge-tree --write-tree <branchA> <branchB>` (or, on older git, `git merge-tree $(git merge-base origin/main <branchA> <branchB>) <branchA> <branchB>`) and check for a conflict marker / non-zero conflict status; **a detected textual conflict** on any shared file makes the set **risky** (→ modal). If the trial merge is clean, the same-file overlap is **disjoint-hunk** → advisory only, set stays timed-eligible. (A genuine conflict that only surfaces later, at merge time, still routes to **4c** via the `CONFLICTING`/`DIRTY` `mergeStateStatus` in (c) — the queue, on a `NATIVE` backend, is itself the safety net that serializes/re-tests, which is *why* same-file-disjoint no longer needs a blocking tax here.)
- **(b) Flagged PR** — any selected PR carries a `hold` or `risky` **label** (`gh pr view <pr> --json labels`), **or** its plan item has a `notes:` **risk flag** (a `notes:` field on the plan item marking risk).
- **(c) Not-CLEAN mergeStateStatus** — any selected PR's `mergeStateStatus` is **not `CLEAN`** (i.e. `BEHIND` / `BLOCKED` / `DIRTY` / `UNKNOWN`), read from the same `gh pr view --json mergeable,mergeStateStatus` above (after the single re-poll for a lagging `UNKNOWN`/`BEHIND`).

**Read the deferred `## Questions` batch here too — it is drained at this gate (4f).** At gate-entry, re-read `$1` and collect every **unanswered** (`[ ]`) entry in the plan note's `## Questions` section (the `batch-at-gate` decisions deferred during this level — Step 2.5 issue creation, Step 2.6 epic creation, Step 0.5 reconcile auto-fix, Step 4c conflict dispositions). These are surfaced as **ONE batch before merge consent is recorded** (4f), in whichever regime this gate runs — they do **not** add a second interruption. An empty/absent `## Questions` section is the common case (no deferrals this level) → the gate runs exactly as before.

If **none** of (a)/(b)/(c) holds, the set is **clean, disjoint, and independent → timed** — a same-file/disjoint-hunk advisory note (the (a) file-overlap probe firing without a hunk conflict) does **not** flip this; it is printed in the summary and the set stays timed. **Window:** `BUILD_MERGE_GATE_WINDOW` (seconds); **`BUILD_MERGE_GATE_WINDOW=0` forces every set to hard-block** — the always-modal kill-switch that restores the prior always-explicit-approval behavior. **The kill-switch is evaluated here, at gate-entry (4a), so it must be set BEFORE the gate fires to take effect.** Changing `BUILD_MERGE_GATE_WINDOW=0` *mid-window* does **not** preempt a countdown already running — once 4a has selected the timed regime and scheduled the wake, the only mid-window intervention is the objection mechanism (any operator response → modal, per 4b), not the env var. State the selected regime (and, when timed, the window) in the 4a summary so the operator knows whether a countdown is running.

### 4b. Mergeable set — approval (modal or timed), then merge

**Select the approval regime from the 4a block-vs-timed computation.** A **risky** set (4a (a)/(b)/(c), or `BUILD_MERGE_GATE_WINDOW=0`) takes the **hard-block modal** path; a **clean, disjoint, independent** set takes the **timed** path. Both paths converge on the identical merge mechanics below — the regime changes only *when/whether the run blocks for a human*, never the merge incantation. Per [[Decisions/foundation - Timed merge-gate auto-merge (supersedes never-auto-merge-main)]].

**Surface the deferred `## Questions` batch first (4f) — modal path.** Before the merge-approval question, present every unanswered `## Questions` entry collected in 4a as **one batch** alongside the merge gate (resolve them per 4f, recording each answer/default back into the section). Showing them here — at the modal gate the operator is already attending — is what makes the deferral free: the operator answers the deferred decisions and the merge in one sitting, never mid-level.

**Approval — risky set (hard-block, modal).** Require explicit approval; **never** timed-auto-merge a risky set. Surface via `decision_sink_ask(<the risky set + why it blocked>, <one selectable option per mergeable PR (label = slug + PR # + title; default = all selected, user deselects any to hold) + an **Abort the run** option that short-circuits everything else>, blocking-now)` with `multiSelect: true` — operator-present → modal `AskUserQuestion`, the **identical multi-select approval prompt** as before; operator-absent → async decision-issue posted on the **plan note's epic issue** (not any single item's issue) with the risky-set summary and the three options (Merge all / Hold some — specify which / Abort), `decision` label applied, operator assigned, tick stops for this level. **The tool caps at 4 options** — when more than 4 PRs are parked, you can't list one per PR. Instead print the per-PR summary as text (4a) and ask a single single-select `decision_sink_ask(...)`: **Merge all** / **Hold some — I'll name them** / **Abort the run**. The seam carries only the ask; the deferred `## Questions` (4f) drain that is shown alongside it and the merge incantation both stay **outside** the call (and the 4f drain precedes the merge, unchanged). Surface *why* the set hard-blocked (the offending predicate — overlapping *hunks* (a real same-file conflict detected by the 4a trial merge, **not** mere same-file overlap, which is advisory) / flagged PR / not-CLEAN, or the `=0` kill-switch) so the operator knows the gate isn't timed.

**Approval — clean-disjoint-independent set (timed auto-merge).** Present the green set (the 4a summary) plus the countdown, then auto-merge the whole set via the merge mechanics below **after the window elapses, UNLESS an operator objection arrives first OR a wake-time re-validation finds the set no longer clean**:
- **Headless one-shot runs (`FUNNEL_OPERATOR_ABSENT=1`, per the 3g definition) skip the objection window entirely and merge immediately (#626).** Load-bearing reason — no re-invoke loop → a `ScheduleWakeup` never returns to this plan's gate → no timed window can run; and the structural risky-set hard-block, not a per-merge window, is the real supervision (a risky set never reaches this timed branch): [[Decisions/foundation - build headless (claude -p) foreground merge — no re-invoke loop (#626)]]. So when `FUNNEL_OPERATOR_ABSENT=1`: do **not** `ScheduleWakeup`, do **not** announce a window, and **do NOT run the objection scan** (check 2 below) — **no window was opened, so there is no objection period and no `<announce-ts>` anchor; any epic-issue `OBJECT` comment that exists is stale from a prior window and MUST NOT be read here** (reading it would silently park every future headless merge on this plan — the failure this branch must avoid). Run **only check 1 synchronously, now**: re-poll every PR `OPEN`+`CLEAN` (a fast `gh pr view`, no wait). If all pass, apply any unanswered `## Questions` (4f) defaults inline, then proceed **straight to the merge incantation** for the re-confirmed-clean set. If any PR is no longer `OPEN`+`CLEAN`, fall back to the modal hard-block path exactly as the timed path does (operator-absent → async decision-queue on the epic issue, park, stop). State `immediate-merge (headless, no window)` in the 4a summary. The bullets that follow — the `ScheduleWakeup` window, the re-homed announce, **and the operator-absent objection receptacle** — serve **loop-capable runs only** (operator-present, and the interactive walk-away `--unattended` case, both of which have a working re-invoke loop); under #626 a `FUNNEL_OPERATOR_ABSENT=1` run is headless-no-loop and takes this immediate-merge branch instead, so the cron objection-receptacle bullet is **superseded for the funnel** by this branch.
- **Reuse the existing approval-poll / `ScheduleWakeup` mechanism** — the same self-paced wake loop `assess` Step 6 uses to watch a plan's `status`. Schedule a wake `BUILD_MERGE_GATE_WINDOW` seconds out; **encode the absolute deadline timestamp AND the announce timestamp in the `ScheduleWakeup` `prompt` itself** (not just working memory) so they survive context summarization across the window, exactly as the Step-6 poll encodes its `ASSESS_POLL_BUDGET` deadline. Announce the window and the set up front, **including any unanswered `## Questions` entries collected in 4a** so the operator can act on them within the window: *"Merging N PRs in 5 min unless you object — reply to cancel. P deferred questions will take their defaults at the deadline unless you answer: <one-line per entry with its default>."*
- **Re-homed objection receptacle (operator-absent / cron runs).** On an operator-present run the in-session turn stream is the objection surface — any reply cancels the timer. On an **operator-absent / cron run** that in-session surface does not exist (there are no operator turns to scan), so "no turns found" would silently mean "no objection" and the gate would always auto-merge — a structural degradation. **Fix:** at announce time, when the run is operator-absent, post a comment on the **plan note's epic issue** (the same issue the risky-set modal path already uses) that announces the merge window and names the `OBJECT` keyword:
  ```
  Merge window open — Level <k> (<N> PRs): <PR list>.
  Merging in ~<window>s unless you reply OBJECT on this issue within the window.
  Deadline: <ISO-8601-deadline>.
  ```
  Encode the **epic issue number** in the `ScheduleWakeup` `prompt` alongside the deadline and announce timestamps, so it survives context summarization. The objection check at wake-time (check 2 below) then reads comments on that issue, not in-session turns. **The epic issue is the per-window-level objection receptacle on cron runs, not a new decision issue** — no `decision` label is applied (it is the merge-window announcement, not a pending decision). Operator replies with `OBJECT` on this issue to cancel. If the epic issue is unavailable (board integration OFF or no `epic:` in the plan frontmatter), fall through to the **modal hard-block path above** instead of running the timed path (the timed path requires a durable objection surface that exists across ticks; without a GH issue it has none).
- **Window-elapse applies each `## Questions` entry's default (4f, timed path).** A deferred entry still `[ ]` when the window elapses **takes its `default`** — the same auto-proceed the timed gate already applies to the merge itself (absence of objection = consent). Resolve each such entry per 4f (tick `[x]`, record `→ default taken: <value>`) **before** executing the merge. This is what lets a fully-unattended timed gate proceed end-to-end with no silent stall — every deferred decision has a value to proceed on.
- **Objection cancels — and routes the deferred questions into the modal fallback.** **Any operator response within the window** cancels the auto-merge and falls back to the **modal hard-block path above** (the `decision_sink_ask(...)` risky-set approval), where the operator selects/holds/aborts **and** answers the deferred `## Questions` batch (4f, modal path) rather than letting them default. The post-announce objection scan (the durable check below, check 2) stays **outside** the seam and **precedes** the fallback ask and the merge incantation, unchanged. (No response is the walk-away case the timed gate exists to serve — there the defaults are taken.)
- **Wake-time re-validation (BLOCKER guard — required before any merge).** When the wake fires at the deadline, **BEFORE executing any merge**, the orchestrator must re-confirm the set is still safe — the 4a risk predicate (esp. `mergeStateStatus`) was computed once at gate-entry and a PR can silently flip `CLEAN → BEHIND/BLOCKED/DIRTY` during the window (e.g. another merge advanced `main`). Two checks, both required:
  1. **Re-poll every PR in the presented set:** `gh pr view <pr> --json state,mergeable,mergeStateStatus`. **Proceed with the timed auto-merge only if ALL PRs still read `state == OPEN` + `mergeStateStatus == CLEAN`.** If **any** PR is no longer `OPEN`+CLEAN, the timed consent is void: fall the **whole set** back to the **modal hard-block path above** (the `decision_sink_ask(...)` risky-set approval) — except a PR that has developed a genuine content conflict (`CONFLICTING`/`DIRTY`) routes to **4c** while the rest re-enter the modal gate. (`mergeStateStatus` lag still applies — re-poll once after ~3s before treating a lone `BEHIND`/`UNKNOWN` as a real flip, per 4a.)
  2. **Re-check for a post-announce objection — surface depends on operator-presence (re-homed, not inherited):**
     - **Operator-present path (unchanged):** scan for any operator turn that arrived since the **announce timestamp** encoded in the wake prompt. Do **not** rely solely on implicit harness turn-ordering to have surfaced an objection — if any operator turn is present since the announce, treat it as an objection: cancel the auto-merge and fall back to the modal path.
     - **Operator-absent path (cron/tick — re-homed onto the epic issue):** scan comments on the **plan note's epic issue** (whose number is encoded in the wake prompt — see the announce step above) posted after the **announce timestamp**. If any comment body contains the word `OBJECT` (case-insensitive, anywhere in the line), treat it as an objection: cancel the auto-merge and fall back to the modal hard-block path (`decision_sink_ask(...)` on the epic issue, as the risky-set path already does — operator-absent backend). Implementation: `gh issue view <epic> -R "$REPO" --json comments --jq '[.comments[] | select(.createdAt > "<announce-ts>") | .body] | map(test("OBJECT";"i")) | any'`. A `true` result cancels. The epic issue comment is the durable, cross-tick objection surface that a ScheduleWakeup-alone cannot provide on a cron run.
  **"Absence of objection = consent" applies only to a set re-confirmed clean at merge time.** Only when both checks pass — all PRs OPEN+CLEAN and no post-announce objection (via session turns or epic-issue comment, per path above) — does the wake proceed. **Serialization order at wake-time (preserved, load-bearing — none of these is inside any seam call; the timed walk-away path issues no `decision_sink_ask`):** (1) the post-announce objection scan (check 2 above) → (2) apply each unanswered `## Questions` entry's default (4f, timed path: tick `[x]`, record `→ default taken: <value>`, take the deferred action) → (3) the merge incantation below. So when both checks pass: **apply each unanswered `## Questions` entry's default first** — then merge the presented set via the mechanics below. (If check 2 found a post-announce objection, the deferred batch routes into the modal fallback per the objection bullet above, where the operator answers it via the `decision_sink_ask(...)` risky-set approval — defaults are taken only on the clean walk-away path.)
- **Strict `main` still requires MERGED-confirmation on the timed path.** The timed/modal distinction does **not** relax the #130 premature-close guard: the timed path still requires poll-until-`state == MERGED` confirmation before any 4d Done-move — on a `NATIVE` backend via the "Merge — NATIVE backend" poll predicate (a `--auto` enqueue is never landed), on a `MANAGED` backend via `gate.sh managed-merge`'s internal confirm poll (its `MERGED` outcome *is* that confirmation) — a timed-gate merge is never treated as landed merely because the wake fired and the merge was queued/issued.
- This **only** removes *blocking* friction on a set that is provably safe **both at announce and at wake time**; it does **not** change *which* PRs merge (the whole presented green set) or *how* (the unchanged incantation below).

**Select the merge path from the 4a `backend` probe first — the backend is resolved at run time by `gate.sh backend <owner>/<repo>`, never hard-coded per repo.** `NATIVE` ⇒ the **"Merge — NATIVE backend"** path immediately below. `MANAGED` ⇒ the **"Merge — MANAGED backend"** path after it. Because the probe reads the repo's *current* branch ruleset each run, **a repo's backend can change** (a native merge queue can be stood up or torn down), so always take the path from this run's probe rather than assuming one from the repo's identity. Today the **native-queue path is the live path on the org build repos and on temperloop** — all currently probe `NATIVE`; in particular **temperloop probes `NATIVE`** (its `main` has a native merge queue), so despite being the free-tier kernel repo it is **NOT** the MANAGED case. The managed path is the live path only on a repo with no native queue (one that can't provision or has disabled its queue) and is the fallback when the backend probe fails.

**Merge — NATIVE backend (`NATIVE` — native merge queue; formerly "queue ON", cells C/D — unchanged in behavior):** for **every** selected PR, run **`gh pr merge <pr> --auto`** — the queue-ON incantation from Step 0 (**no** `--queue`/`--merge`/`--delete-branch`: `--queue` is not a real gh-2.93 flag, and a queue branch owns the merge strategy + branch deletion). `--auto` arms auto-merge, which enqueues the PR once its required checks pass. The queue serializes/batches via its own `merge_group` trial branch (an integration branch GitHub builds + tests as a combined candidate, then merges FIFO once branch-protection checks pass), and **updates branches itself** — so the per-PR `update-branch` nudge loop (the #83 workaround, now folded into `gate.sh managed-merge --strict`'s own mechanics on the MANAGED path) is **RETIRED for queue repos**. Then drive the queue-aware poll/stall path:

1. **Poll predicate.** After each `--auto`, poll `gh pr view <pr> --json state,mergeStateStatus,mergedAt`. **gh 2.93 exposes NO merge-queue-entry field** — `mergeQueueEntry` is not a valid `--json` field (it returns nothing), and `autoMergeRequest` reads **null even while a PR is queued** (it tracks classic auto-merge, not the merge queue). So queue *position/state is not observable via the gh CLI*; classify on `state`/`mergeStateStatus` plus the timeout ceiling (step 2) as the stall/dequeue backstop:
   - **In-queue / processing (keep waiting):** `state == OPEN` AND `mergeStateStatus` NOT in `DIRTY`/`CONFLICTING` (it routinely reads `CLEAN`/`BLOCKED`/`UNKNOWN` while queued — **do not treat that as failure on its own**). A dequeue that leaves the PR `OPEN` is indistinguishable from "still queued" **from the `gh pr view` fields alone**, so the **`BUILD_QUEUE_TIMEOUT` ceiling still bounds this wait** — but a stall is **no longer opaque**: run **`gate.sh diagnose-queue <owner>/<repo> <pr>`** (temperloop#1150), which classifies it from the two channels `gh pr view` cannot see — the GraphQL `mergeQueueEntry` membership signal (still-in-queue) + the `merge_group` run history (why it left) — returning `QUEUED` / `MERGE_GROUP_FAILED` / `DEQUEUED` / `MERGED`. Use its verdict to route recovery (step 3). **Cadence** (each call spends a GraphQL + REST request — do **not** run it every poll tick, per the shared-budget concern the board adapter guards): invoke it **at most twice** per PR — once when the in-queue wait first crosses the stall-suspicion point (about half the `BUILD_QUEUE_TIMEOUT` ceiling — so a `MERGE_GROUP_FAILED` is caught early rather than after the full wait), and once at the ceiling before step-3 recovery.
   - **Merged (success → 4d):** `state == MERGED` (`mergedAt` set). **Only this triggers the 4d Done-treatment** — preserves the #130 guard (`--auto` only *enqueues*; "added to queue" / "auto-merge enabled" is NOT landed).
   - **Failed (→ recovery):** `state == OPEN` AND `mergeStateStatus == DIRTY`/`CONFLICTING` → a real content conflict → route that one PR to **4c**, as today. OR `state == CLOSED` without `mergedAt` (abandoned/dequeued-and-closed) → recovery (step 3). OR `gate.sh diagnose-queue` returns **`MERGE_GROUP_FAILED`** (the queue's `merge_group` CI concluded failure — a semantic conflict or a transiently-broken main, with the failed run id) → route straight to **4c** without waiting out the ceiling. OR `gate.sh diagnose-queue` returns **`DEQUEUED`** (left the queue with no failing group run) → recovery (step 3), the same recovery target as a `CLOSED`-without-merge state.
2. **Timeout ceiling — `BUILD_QUEUE_TIMEOUT` (seconds)** per selected PR's time in queue. **Encode the absolute deadline timestamp in the poll's `ScheduleWakeup` `prompt`** (same pattern as the 4b timed-gate window and assess Step 6), so it survives context summarization across the wait. **Headless one-shot runs (`FUNNEL_OPERATOR_ABSENT=1`) poll for `MERGED` in the FOREGROUND, not via `ScheduleWakeup` (#626)** — a blocking `gh pr view` poll loop **at a ≥30s interval** (the same floor `ci-poll.sh` enforces in 3g — no reason to poll tighter than the queue lands) bounded by `"$BUILD_HEADLESS_POLL_TIMEOUT"` per Bash call (under the ~10-min cap; chain another foreground poll if still queued and the session budget allows). On `state == MERGED` → 4d in-session. If the bound elapses with the PR still `OPEN`+enqueued (`--auto` armed, not `DIRTY`/`CONFLICTING`), do **not** dequeue and do **not** yield for a wake — leave it enqueued and **stop**: the `funnel-merge-pending` marker (#624) resumes next tick, re-checks, finds `MERGED`, and clears the marker. (The step-3 classify-and-recover flow still applies on a *confirmed* stall/conflict, just driven by the foreground bound rather than a backgrounded wake.)
3. **Recovery on stall** (timeout elapsed, or a confirmed persistent dequeue-without-merge). **Classify first** — `gate.sh diagnose-queue <owner>/<repo> <pr>` (temperloop#1150) turns the stall into a structured verdict; branch on it:
   - **`MERGE_GROUP_FAILED`** (the queue's `merge_group` CI concluded failure — a real group conflict / broken main, carrying the failed run id) → **dequeue** (`gh pr merge --disable-auto <pr>`) and route straight to **4c**: a `managed-merge` retry would just re-hit the same group failure, so skip it.
   - **`DEQUEUED`** (the entry dropped during queue churn — no failing group run) → **re-arm once** via `gate.sh queue <owner>/<repo> <pr>` (idempotent `--auto`); if it stalls again next pass, fall through to the managed fallback below.
   - **`QUEUED`** (still legitimately enqueued) → it is not actually stalled; keep waiting unless the `BUILD_QUEUE_TIMEOUT` ceiling is genuinely hit, in which case take the managed fallback below.
   - **`MERGED`** → it landed after all → treat as success (go to 4d).
   - On an `ERROR`/unclassifiable verdict, or the `QUEUED`-past-ceiling case, fall through to the mechanics that recover any stall regardless of cause:
     1. **Dequeue** the PR — `gh pr merge --disable-auto <pr>` (removes it from the queue / disables the pending auto-merge).
     2. **Fall back to the managed per-PR mechanics** for that PR: `gate.sh managed-merge <owner>/<repo> <pr> --strict` (the scripted successor of the old cell-B `--auto` + `update-branch`-nudge fallback — same behavior: make the branch current, revalidate CI on the updated head, land, confirm `MERGED`).
     3. If the managed fallback also can't land it (real conflict — `CONFLICTING`, exit 3), route to **4c** exactly as today.
4. **Unattended default = classify-and-recover, recorded not silent.** On the `--unattended`/cron arm, the step-3 recovery is taken automatically (never spin forever) AND recorded: append one `### open` entry to the pending-decisions surface (`Pipeline/pending decisions.md` vs the legacy `Context/pipeline - pending decisions.md` — target pinned by the append-target resolution rule, `claude/commands/check-in.md`; in the knowledge store) via `mcp__obsidian-builtin__vault_append` (this is a `batch-at-ritual` auto-disposition — `claude/CLAUDE.md` § Unattended pending-decisions surface — so the next `check-in` reviews it). **The record must name the actual `diagnose-queue` verdict and the branch step 3 took** — the `Default taken:` line is the durable surface `check-in` trusts, so a fixed action would misreport the three branches that don't dequeue-and-fallback:
   ```markdown
   ### <YYYY-MM-DD HH:MM> · build Step 4b queue-stall · <host>:<sess8>
   - **Decision:** native merge queue stalled on PR #<N> (<slug>) at level <k> after BUILD_QUEUE_TIMEOUT — `diagnose-queue` → <QUEUED|MERGE_GROUP_FAILED|DEQUEUED|MERGED|ERROR>
   - **Default taken:** <the branch actually taken — MERGE_GROUP_FAILED → "dequeued, routed to 4c (merge_group run #<id> failed)"; DEQUEUED → "re-armed via gate.sh queue"; MERGED → "confirmed MERGED, no recovery needed"; QUEUED-past-ceiling / ERROR → "dequeued (--disable-auto) + managed-merge --strict fallback">
   - **Disposition:** auto-taken (unattended run; no live operator)
   - **Status:** open
   ```
   On an **attended** run, surface the stall as a modal `AskUserQuestion` whose **context names the `diagnose-queue` verdict** (so the operator sees the cause, not just "stalled"), offering the verdict-appropriate options — e.g. on `MERGE_GROUP_FAILED`: route-to-4c / leave-queued / abort; on `DEQUEUED`: re-arm / dequeue-and-fallback / abort — instead of writing the surface.
5. **MERGED-confirmation gates 4d (load-bearing — #130).** Every arm above gates the 4d Done-move on confirmed `state == MERGED` (`mergedAt` set) — including a dequeue-and-fallback that ultimately lands the PR via the strict-serial path. A still-queued PR stays `[m]` (its item keeps In Progress) until a later pass confirms `MERGED`. **Never** treat "added to queue" or "auto-merge enabled" as landed (the exact #130 bug).

**Merge — MANAGED backend (`MANAGED` — no native queue; absorbs the old `queue=OFF` cells A/B, temperloop#13):** a serial per-PR loop the **orchestrator owns** — the loop, the plan-item processing order, and the stop/continue policy are orchestrator-side; `gate.sh managed-merge` stays strictly **per-PR mechanics** (a set-loop inside gate.sh would move merge-order policy into the deterministic spine). Also the **queue-stall fallback** target from "Merge — NATIVE backend" step 3.

1. **Record consent durably FIRST — the gate-log consent line.** The loop's first action, immediately after consent exists (modal approval given, timed window elapsed with no objection, or the headless immediate-merge re-poll passed) and **before the first `managed-merge` call**, is to append one consent line to the plan note's **`## Merge gate log`** section (schema: `~/.claude/plan-schema.md` § Orchestrator-written `## Merge gate log` section) via `mcp__obsidian-builtin__vault_append`: level, ISO-8601 timestamp, consent mode (`modal-approved` | `timed-elapsed` | `headless-immediate`), and the **exact consented PR list**. This line is what makes **resume-without-re-consent** mechanical (state table below): a crash mid-loop resumes the remaining consented PRs without re-asking. Consent pins that exact PR list — a PR outside it (or re-opened under a new number) is NOT covered and takes a fresh gate.
2. **The serial loop.** For each consented PR **in plan-item order** (the within-level source order from Step 1.4), run `workflows/scripts/build/gate.sh managed-merge <owner>/<repo> <pr> --strict|--non-strict` — the flag from the 4a `strict` probe (`STRICT` → `--strict`, `NON_STRICT` → `--non-strict`). Per-PR mechanics (as built): `--strict` (the default) runs `gh pr update-branch` → resolves the **new** head SHA → SHA-pinned CI re-poll on that updated head (REST check-runs, never `gh pr checks --watch`) → on green, merges via the Step 0 canonical incantation (`--merge --delete-branch`, **not** `--auto` — mergeability is already established) → polls to **confirmed `state == MERGED`**; a red re-poll **ejects with no merge attempted**. `--non-strict` skips the update-branch + re-poll entirely (cell A's immediate-merge cost profile — no CI-cost regression) and goes straight to merge + confirm. On `--strict`, budget ≈ one CI-length wait per PR after the first (each merge advances `main`; the next PR's update-branch re-runs CI) — the old cell-B serial cadence, now scripted.
3. **Branch on the closed outcome set** (`.outcome` of the one JSON line; never parse prose). Persist each PR's disposition to the plan note **as it resolves** (`[x]` / stays `[m]` / `[-]`), so an interrupted loop reconstructs from sentinels + the consent line:
   - **`MERGED` (exit 0) → 4d.** The success outcome **is** the internal poll-until-`state == MERGED` (`mergedAt` set) confirmation — so the **#130 confirmed-MERGED guard applies verbatim on the managed path**: the 4d Done-treatment keys on this confirmed `MERGED` outcome, never on "the merge call returned" (and never on anything short of `state == MERGED`; a PR closed-without-merge can never read as merged).
   - **`EJECTED` (exit 5 — CI red on the updated head; no merge was attempted; `failed_run_ids` in the payload) → its OWN disposition set, and the loop CONTINUES with the remaining consented PRs** (an eject never halts the level's queue). This is **not** a 4c case — 4c stays real-content-conflicts-only; an eject is a CI failure freshly surfaced by folding `main` into the head. Three dispositions:
     - **Re-spawn worker to fix the CI failure** — re-invoke `Agent` against the item's pre-created `worktreePath` (re-arm the `.build-guard` marker per the 4c re-spawn mechanics if 3h removed it), handing it the `failed_run_ids`; worker fixes + commits, orchestrator re-pushes, re-watches CI (3g), item re-parks `[m]` for the next gate pass.
     - **Skip permanently** — mark `[-]` + reason.
     - **Leave for manual triage (default)** — stays `[m]`; append `- ejected at level-k managed merge: CI red after update-branch (runs <failed_run_ids>)`.
     The disposition is a **`batch-at-gate`** decision: on an attended modal gate ask it inline (the operator is already present); otherwise append one `## Questions` entry (`` `step: 4b` `pr: <N>` — EJECTED PR #<N> (<slug>) disposition? **default: leave for manual triage** ``, with an `auto-proceed:` sub-line) and proceed on the default — non-destructive, exactly like 4c's conflict default.
   - **`MERGE_REJECTED` (exit 6 — the platform itself refused the `gh pr merge` call) → re-probe the backend, then dispose.** The likeliest cause is a **queue-armed repo mis-probed as MANAGED** (a `probe_failed:true` verdict, or a stale `BUILD_MERGE_BACKEND=managed` override) — a queue-required branch rejects a direct merge. Re-run `gate.sh backend <owner>/<repo>`: if it now reads `NATIVE` (or the operator confirms a queue), route the PR and the rest of the consented set through the **NATIVE** path above (the consent line still covers them — same PR list, same level). If it still reads `MANAGED`, the residual disposition is a **`batch-at-gate`** decision, not a `blocking-now` halt — leaving the PR `[m]` is a safe, non-destructive default (nothing merges; nothing is lost): on an attended gate ask inline; otherwise append a full `## Questions` entry (`` `step: 4b` `pr: <N>` — MERGE_REJECTED PR #<N> (<slug>): platform refused the merge (<error>). Disposition? **default: leave `[m]` for manual triage** ``, with an `auto-proceed:` sub-line) and proceed on the default; the loop continues with the rest.
   - **`CONFLICTING` (exit 3 — a real content conflict, or the post-merge confirm poll hit `CONFLICTING`/`DIRTY`) → route that one PR to 4c**, as today; the loop continues with the rest.
   - **`TIMEOUT` (exit 4 — a poll deadline passed) → surface it; the PR stays `[m]`** (keeps In Progress) for a later pass to re-check; the loop continues with the rest. Two poll loops can raise it (temperloop#23): the **SHA-pinned CI re-poll** running out its deadline with checks still pending (`--strict` arm; **no merge attempted** — the head is not yet green), or the **post-merge confirm poll** running out its deadline (the merge call was issued but `state == MERGED` wasn't confirmed in the window — the #130 guard keeps it `[m]`, not merged). Either way nothing is treated as landed. On an unattended run, also record it (`batch-at-gate`): append a `## Questions` entry (`` `step: 4b` `pr: <N>` — managed-merge poll timed out (CI re-poll or confirm poll). **default: leave `[m]` — a later pass re-checks** ``, with an `auto-proceed:` sub-line) so the stall is legible, not just parked.
   - **`ERROR` (exit 1 — a gh/git call inside `managed-merge` failed: the update-branch call, the head-SHA resolve, or the CI re-poll query itself erroring) → surface the error verbatim; never treat it as green.** A CI re-poll that merely runs out its deadline is **not** ERROR — it is `TIMEOUT`/exit 4 above (temperloop#23; ERROR is reserved for a failed call, per gate.sh's header exit-code contract). The PR stays `[m]` (nothing was merged); the loop continues with the rest, and the unattended run records it exactly like `TIMEOUT` above (same `## Questions` shape, default: leave `[m]`).
4. **No queue/resume label — sentinels + consent line + live probe only.** Resume state on the managed path rides the plan-note sentinels (`[m]`/`[~]`), the `## Merge gate log` consent line, and a live `gh pr view` probe — **no new label anywhere** (a `tl:queued`-style PR label was considered and subtracted at design review: dual-state drift between label and plan note). `FUNNEL_MERGE_PENDING_LABEL` (`funnel-merge-pending`, `build.config.sh`) remains the **funnel-plane** cross-tick marker on the board item (3g/#624) — a different plane, cross-referenced here, never duplicated as a build-plane merge-state label.
5. **Resume-without-re-consent — the pinned state table.** On resume (Step 1.4 `[m]`/`[~]` revalidation, or Step 0.5 reconcile), route each item of a partially-merged level by: **consent line present for this level × sentinel × live PR state**:

   | Consent line covers this PR | Sentinel | Live PR state | Action |
   |---|---|---|---|
   | yes | `[m]` | `MERGED` (`mergedAt` set) | **Mark merged** — flip to `[x]` and run the 4d treatment (the crash happened after this PR landed). |
   | yes | `[m]` | `OPEN`, CI green — or pending/red (a crash mid-`managed-merge`'s own update-branch/re-poll window; a parked `[m]` was green *before* the loop touched it) | **Resume the managed loop** — re-issue `gate.sh managed-merge` for this and the remaining consented PRs, in plan-item order, **without re-asking consent**. Re-issuing is idempotent: it redoes the update-branch + re-poll itself, and a genuinely-red head simply resolves to `EJECTED` and its disposition set. |
   | no (line absent, or PR not in its list) | `[m]` | `OPEN` | **Fresh gate** — no consent was recorded for it; the level's remainder re-enters 4a/4b normally. |
   | yes | `[~]` | any | **Fresh gate after re-park** — the item regressed out of `[m]` post-consent (e.g. an EJECTED re-spawn); it re-earns `[m]` via 3g and takes the *next* gate; the old consent does not carry to a re-pushed head. |
   | any | `[m]`/`[~]` | `CLOSED` without `mergedAt` | **Escalate** — PR closed-unmerged is a `blocking-now` halt (re-open / re-run / skip `[-]`), never inferred merged (#130 class). |

**The 4b merge mechanics are unchanged by the timed gate** — the **incantation is exactly Step 0's** (NATIVE uses the smaller `gh pr merge <pr> --auto`; MANAGED merges inside `gate.sh managed-merge` via `gh pr merge <pr> --merge --delete-branch`, with `--strict` folding an update-branch + CI re-poll in front). The timed/modal regime decides only *whether the run blocks for a human first*; once the set is approved (explicitly in the modal path, or by absence-of-objection in the timed path), the merge runs identically. On the MANAGED `--strict` arm, budget ≈ one CI-length wait per PR after the first (each merge advances `main`); the old hand-driven `update-branch` babysitting is now inside `managed-merge`'s own mechanics. The integration-branch **merge train** (4b-train) is **retired** — absorbed by this managed loop (see 4b-train's retirement note); on a NATIVE repo the queue's `merge_group` does the batching itself.

### 4b-train. Integration-branch merge train (RETIRED — absorbed by the MANAGED backend)

**RETIRED (temperloop#13) — do not run.** The train's only live condition (strict `main`, queue absent, ≥3 disjoint PRs) no longer exists as a merge path: on a `NATIVE` repo the queue's `merge_group` trial branch *is* the integration branch, and on a `MANAGED` repo the scripted serial loop (`gate.sh managed-merge`, 4b "Merge — MANAGED backend") is the train's replacement. Rationale for retiring it (one scripted path over a third unscripted merge axis; the ≥3-disjoint-PR serial-CI trade accepted): [[Decisions/foundation - build native merge-queue adoption]] §4. The heading is retained so historical cross-references resolve.

### 4c. Non-mergeable set — per-PR question

This is for **real content conflicts** only — `CONFLICTING`/`DIRTY` (or `UNKNOWN` after re-poll). It is **not** for `BEHIND`: a `MERGEABLE/BEHIND` PR has no conflict, just a stale base, and is resolved in 4b — on the NATIVE backend the queue updates branches itself; on the MANAGED backend `gate.sh managed-merge --strict` folds the `update-branch` + re-CI into its per-PR mechanics — **not** a manual rebase. (Nor is it for a managed **`EJECTED`** — CI red after update-branch is a CI failure with its own disposition set in 4b, not a content conflict.) Only escalate to 4c if a PR develops an actual content conflict (`CONFLICTING`/`DIRTY`) that updating the branch can't clear.

For each PR flagged `CONFLICTING` (or `UNKNOWN` after re-poll), the disposition is a **`batch-at-gate`** decision — **append one `## Questions` entry per conflicting PR** rather than issuing a per-PR interrupting `AskUserQuestion`, so the conflict dispositions fold into the **single batched gate question** (4f) alongside any other deferred entries. Entry: `` `step: 4c` `pr: <N>` — Conflicting PR #<N> (<slug>) disposition? **default: leave open for manual rebase** `` with an `auto-proceed:` sub-line (unanswered → leave the PR open, mark `[m]`, append `- open PR #N awaiting manual merge`; nothing destructive is done). The three resolutions an operator may pick at the gate:

- **Leave open for manual rebase** (the default) — leave the PR open / `[m]`.
- **Skip permanently** (mark `[-]`).
- **Re-spawn worker to rebase** — re-invoke `Agent` against the item's pre-created `worktreePath` (its `.build-guard` marker persists with the worktree, so the guard stays armed; if 3h already removed it, re-create it per 3b first — `worktree.sh create` re-drops the marker), instructed to `git fetch origin`, `git reset --hard origin/<item.branch>` (the PR tip), then `git rebase origin/main`, resolve conflicts, commit (do NOT push). The orchestrator then **force-pushes** the rebased SHA (`git push -f origin "$(git -C <worktreePath> rev-parse HEAD)":refs/heads/<item.branch>`), re-watches CI, and re-checks mergeability. If newly mergeable, merge inline.

The conservative default (leave open) is non-destructive — an unanswered conflict at gate resolution never skips or rewrites a PR; it just parks it for a human, exactly as if no one was watching.

### 4f. Drain the deferred `## Questions` batch (one batch, before merge consent)

This is where the `batch-at-gate` decisions that were **deferred** during the level (appended to the plan note's `## Questions` section instead of interrupting — Step 2.5 issue creation, Step 2.6 epic creation, Step 0.5 reconcile auto-fix, Step 4c conflict dispositions, Step 4b managed-merge EJECTED dispositions from a *prior* gate pass) are surfaced and resolved as **ONE batch, before merge consent is recorded**. It runs in BOTH gate regimes, off the unanswered set collected in 4a:

- **Modal regime** (risky set, or `BUILD_MERGE_GATE_WINDOW=0`, or an objection arrived): present every unanswered `## Questions` entry **alongside the merge `decision_sink_ask(...)`** (4b modal path) — the operator answers the deferred decisions and the merge in one sitting. Group the entries into a single question (e.g. one `multiSelect`, or a per-entry option set when ≤4); each entry's stated `default` is the pre-selected/recommended choice.
- **Timed regime** (clean walk-away, window elapsed with no objection): each unanswered entry **takes its `default`** when the window elapses (4b timed path) — no interruption, consistent with the timed gate's absence-of-objection consent. The announce message (4b) lists the deferred entries + their defaults up front, so an attending operator can answer them within the window before they default.

**Resolve each entry** by ticking `[ ]` → `[x]` in the `## Questions` section and appending `→ answered: <choice>` (modal) or `→ default taken: <value>` (timed) via `mcp__obsidian__patch_vault_file`, then performing the entry's deferred action keyed off its `step:` (e.g. `step: 2.5` skip-creation retracts the just-created issues; `step: 4c` `pr:` re-spawns the rebase worker; `step: 0.5` runs the safe auto-fixes). A resolved entry is never re-asked on resume.

**No silent stall — load-bearing.** Every `## Questions` entry carries a `default` (the plan-schema invariant), so an entry unanswered at gate resolution **always** has a value to proceed on — the timed regime takes it on window-elapse, the modal regime pre-selects it. An unattended timed gate therefore drains the whole batch to defaults and proceeds end-to-end; it never hangs waiting on a non-blocking question. (This is precisely why `blocking-now` decisions are **excluded** from `## Questions` — they have no safe default, so they must keep their own interrupting halt in 3d/3a/4b-risky, never defer here.)

### 4d. Update plan-note sentinels, move merged items to Done, and close the epic if drained

After the gate completes, **for each item whose PR is confirmed `state == MERGED` (`mergedAt` set) — not merely `--auto`-queued —** patch `$1` (a still-queued/unconfirmed item stays `[m]` + In Progress, handled by the "Left open" arm below). **This applies identically on the queue path:** `gh pr merge <pr> --auto` only *arms auto-merge / adds the PR to the queue*; the 4d Done-treatment fires only on confirmed `state == MERGED`, never on "added to queue" / "auto-merge enabled" (the queue extension of the #130 guard — see the 4b "Merge — NATIVE backend" poll predicate and the callout below; the MANAGED path satisfies the same guard inside `gate.sh managed-merge`, whose `MERGED` outcome is itself the confirmed poll).
- Merged: `[m]` → `[x] **<title>** ... — merged in #N (YYYY-MM-DD)`
- Left open: `[m]` stays `[m]`; append `- open PR #N awaiting manual merge`
- Skipped at gate: `[m]` → `[-]` + append `- skipped at level-k merge gate <date>: <reason>`

**Post-merge worktree+branch convergence (in-lane, idempotent).** For **each item just confirmed `MERGED`** above, reclaim its worktree + throwaway local branch **now**, in the acting session's OWN lane: `workflows/scripts/build/worktree.sh remove <repo-root> <item.slug>` (the same `<repo-root>` resolved in 3b — the plan's own checkout, or the `repo:` cross-repo root — **never** a foreign canonical checkout, per the working-tree-ownership rule). This closes the **post-merge-convergence gap**: cleanup happens deterministically at the moment of merge rather than relying solely on the next session-start sweep. It is **idempotent by construction, not by a new guard** — `worktree.sh` `cmd_remove` already returns `REMOVED` when a worktree or `build/<slug>` branch existed and `NOT_FOUND` when neither did (see 3h step 3 and its own header contract), so this call is a **safe no-op on re-run and safe when the happy-path 3h already removed the worktree** (its `NOT_FOUND` arm *is* the already-gone case — no double-remove error). The two paths converge here: the **conversational path** removed the worktree pre-gate at 3h (this step then reads `NOT_FOUND`), while the **`--workflow` path** left it intact (the workflow removes no worktrees — see the Step 3 dual-path note), so this is where a workflow-driven item's worktree is actually reclaimed. Left-open and skipped items are untouched — their worktrees, if any, persist for the re-drive/manual path. *(Backstop: the **deploy-mini session-start sweep** (F#653) — which prunes merged locals in every clean-on-main checkout at session start — plus `worktree.sh prune <repo-root>` remain the **crash-path** net for a session that died before reaching this step; this step is the deterministic in-lane convergence, the sweep is the safety net, not the primary mechanism.)*

**Also emit an issue-touch record for each item just confirmed `MERGED`** (foundation #916/#919, the issue-touch-stream — skip when the item carries no `gh_issue:`): `workflows/scripts/emit-issue-touch.sh --repo <owner>/<repo> --issue <item.gh_issue> --kind merge || true`. Telemetry only, same `|| true`-isolated contract as the 3f PR-open emit above, and covered by the same `validate-issue-touch-emit.sh` presence-lint. Left-open and skipped items do **not** get a `merge` touch — only items that reached confirmed `MERGED` in this pass.

**Board integration ON** — build does **NOT** issue a redundant explicit per-item `board_set_status … Done` write. The merge's `Closes #<gh_issue>` closes the issue, and the close→Done cascade automation (GH #340) then moves the card to **Done** on its own — one line (the PR body's `Closes`) does both the close and the board move. A read-then-skip conditional was considered and **rejected**: the cascade is async, so at the moment this step runs (right after confirming `MERGED`) the card usually still reads In Progress and the conditional would write Done anyway — near-zero budget saving plus an added read. Dropping the explicit write also **removes the #130 premature-Done-move risk surface** (there is no longer a hand-timed Done move that could fire before the real merge — see the callout below) and saves a write on the shared 5,000-pt/hr Projects-v2 GraphQL budget. **`reconcile.sh` is the safety net** for the rare card the cascade misses (Step 0.5 reconcile flags a merged/`[x]` item still showing In Progress). Left-open and skipped items keep their current board status (In Progress). See [[Decisions/foundation - Drop redundant board Done-write (rely on close-to-Done cascade)]].

> **Never let anything move the card to Done before the PR is confirmed `MERGED`.** On strict `main` the 4b merge is async (`--auto`), and on a native-queue repo it is *also* async via `--auto` (which enqueues the PR; the gate "completing" means the PRs are *enqueued in the `merge_group`*, not landed — the queue merges FIFO minutes later): in both async cases the gate "completing" means the selected PRs are *queued*, not landed — and a Done-move at queue time fires the board's **Done→close-issue automation** against **unmerged code** (observed in #130: issue #490 was set Done / closed at 00:51 while PR #510 was only queued, and the PR didn't actually merge until 01:37 — a 46-min window where the issue was closed against unmerged code, and would have stayed wrongly closed had auto-merge been cancelled). The close→Done cascade is **inherently safe against this**: it fires only when the issue actually closes, which (via `Closes #N`) only happens on a real merge — so relying on the cascade removes the hand-timed Done write that #130 mis-fired. Never re-introduce a hand-timed Done-move gated on "the gate completed" rather than confirmed `MERGED`.

**If the plan has `epic:`,** closing a merged child auto-ticks the epic's sub-issue list when linked via the typed sub-issues API; for the task-list fallback (Step 2.6), tick `- [ ] #<child>` → `- [x]` in the epic body and post a one-line `Level k: merged #…` rollup comment on the epic.

#### 4d-epic. Epic auto-close (board ON + plan has `epic:`)

After the per-item Done-moves above, check whether the epic still has open children and close it if not. This is what gives the epic a **death** — and it belongs here because a merge (or a spike's `[v]` close) is the only thing that closes a child.

1. **Count open children — data-driven, not "plan finished."** `gh api repos/<owner>/<repo>/issues/<epic>/sub_issues --jq '[.[] | select(.state=="open")] | length'`. (Sub-issues API unavailable → fall back to the epic-body task-list: count unticked `- [ ] #<n>` lines, or query each listed child's state.)
2. **If the count is 0 — account for activation obligations before closing** (temperloop#317; see `Decisions/temperloop - Activation-completeness contract`). "All children closed" proves the epic's work *merged*, not that it *activated* — the same merged/activated split 3e.6's class-A gate draws for a single item, here applied at the epic. Run both sub-steps against the **pending-activations ledger** (`Pipeline/pending activations.md`, falling back to the legacy `Context/pipeline - pending activations.md` — path fallback convention, `claude/commands/check-in.md`; record grammar per `claude/commands/check-in.md` § Pending-activations ledger) before touching step 3:

   a. **Emit-on-close (idempotent).** For each item in **this plan note's** `## Items` list that reached a terminal `[x]`/`[v]` state and carries an `activation:` block with `class: B` or `class: C` — sourced straight off that item's own `activation:` block — append one ledger record per the check-in.md grammar via `mcp__obsidian-builtin__vault_append`. **Class-A obligations are excluded outright**: a class-A block is proven synchronously at 3e.6 and is never ledgered, so it plays no part here. First `vault_read` the ledger and skip emitting for any item whose record already exists (the idempotency check — see the heading convention below). Name each new record's heading `<epic>-<item-slug>` — e.g. `### 317-my-item — <one-line what-must-go-live>` — so it is identifiable as belonging to *this* epic (2b relies on that prefix). Populate the record from the item's own `activation:` block: `class:` verbatim; `proof:`, `locus:`, and the class-appropriate `watermark:` (class B) / `soak-until:` (class C) copied verbatim whenever the item's block states them. Plan-schema does not yet require those sub-fields on a B/C block (temperloop#317 machinery is still landing) — when one is missing, state your best inference from the item's `notes:`/`files:`/`repo:` context and prefix it `(inferred)` rather than leaving the field blank, so an under-specified block still produces a reviewable record instead of blocking the emit. Every freshly emitted record starts `status: open`.
   b. **Refuse-close-while-undischarged.** Re-use the ledger read from 2a (or read fresh) and check for **any** `### <epic>-*` heading — this prefix is exactly how "a record for this epic" is identified, whether it was emitted just now in 2a or by a prior run of this or another plan targeting the same epic — whose `status:` still reads `open`. If **any** such record is open, **do not close the epic**: skip step 3 entirely this pass, leave the epic open, and post one flag comment on the epic issue (mirrors the existing sub-issue rollup-comment pattern below) — `gh issue comment <epic> -R <repo> --body "Activation-incomplete: N record(s) still open on the pending-activations ledger — close deferred."` — skipping the comment if the epic's most recent comment already carries that marker, so a resumed run doesn't repost it every pass. Also note it in the Step 6 run summary: `epic #<epic> children drained but N activation record(s) still open — close deferred`. This is a **flag, not a stall**: only `/check-in` and `/tidy`'s env-hygiene probe ever flip a record's `status:` to `discharged` (check-in.md's ledger section), so the epic closes on a later 4d-epic pass — a resumed run, a later plan against the same epic, or the "every merge gate" re-check below — once every `<epic>-*` record reads `discharged`. (4d-funnel/4d-retro/4d-archive all key off "the epic just transitioned open→closed in this run," so a close deferred here correctly holds them dormant too — no separate change needed there.)
   c. **No-regression fallthrough.** An epic with open-children == 0 and **no** `<epic>-*` activation record at all — no item declared a B/C obligation, or the plan is grandfathered/exempt per plan-schema rule 14 — has nothing for 2a to emit and nothing for 2b to find, so it falls straight through to step 3 exactly as before this section existed. This is the **no-regression invariant**: an epic carrying zero activation records closes on the unchanged open-children==0 path.
   d. **Refuse-close-while-body-acceptance-open** (temperloop#458). "All children closed" also does not prove the epic's **acceptance** ran: acceptance / verification legs written as **prose checkboxes in the epic body** — rather than materialized as their own sub-issues — are invisible to the open-children count and would vanish silently on this close (the ssmobile#328 incident: four hands-on e2e scenarios carried only in the epic body, one merge from disappearing). So before closing, read the epic body — `gh issue view <epic> -R <repo> --json body -q .body` — and count **unchecked** acceptance boxes: `- [ ]` lines that sit under a heading whose text matches `accept` or `verif` (case-insensitive), **excluding** bare child-reference checkboxes (`- [ ] #<n>` or `- [ ] <owner>/<repo>#<n>` — those are tracked by the open-children count, not body prose). If **any** remain, **do not close the epic**: skip step 3 this pass, leave it open, and post one flag comment on the epic issue — `gh issue comment <epic> -R <repo> --body "Acceptance-incomplete: N unchecked acceptance/verification checkbox(es) remain in the epic body — close deferred. Tick them, or break each work-bearing leg out into its own sub-issue, then the epic closes on the next 4d-epic pass. (Deliberate override: close by hand, or re-run with the box(es) ticked.)"` — skipping the comment if the epic's most recent comment already carries the `Acceptance-incomplete:` marker (same anti-repost guard as 2b). Also note it in the Step 6 run summary: `epic #<epic> children drained but N body acceptance checkbox(es) still open — close deferred`. Like 2b this is a **flag, not a stall** and holds 4d-funnel/4d-retro/4d-archive dormant (no open→closed transition this run); the epic closes on a later pass once every body acceptance box is ticked (or each work-bearing leg has been broken out into its own now-closed sub-issue). The **explicit operator override** is to tick the boxes, or close the epic by hand — never a silent auto-close.
3. **If no `<epic>-*` record is open AND no unchecked body acceptance/verification checkbox remains** (2b and 2d both passed, or were a no-op per 2c): close the epic — `gh issue close <epic> -R <repo> --comment "All sub-issues closed; epic complete."`. Do **NOT** issue a redundant explicit `board_set_status … Done` for the epic card: `gh issue close <epic>` triggers the same close→Done cascade (GH #340) that moves the epic's board item to Done on its own, with `reconcile.sh` as the safety net for a cascade miss (Step 0.5 reconcile flags an open-but-drained or mis-statused epic). **If open children > 0, or 2b/2d deferred the close:** do nothing further (the epic stays open).
4. **Idempotent:** an already-closed epic is a no-op (`gh issue close` on a closed issue is harmless); a ledger record already present for an item (2a) is never re-emitted; a `discharged` record never reverts to blocking a later pass.

**Why data-driven, not plan-status-driven** (per [[Decisions/foundation - Triage stage and the logical-technical pipeline split]]):
- A plan may cover only a **subset** of an epic's children (a focused batch). Closing the epic when *this plan* finishes would prematurely close it while other children are still open. Counting open children closes it only when the epic is *actually* done — whether the last child was closed by this run, another run, a triage cull, or by hand.
- Run this check at **every merge gate** after the Done-moves (and once more at end of run), and in the spike kind-fork after a `[v]` close — so the epic closes as soon as its true last child closes, in whichever step that happens. The same repeated-check cadence is what lets a 2b- or 2d-deferred close resolve later without a dedicated re-poll: the next time this step runs and finds open-children == 0 again, it re-reads the ledger (2b) and re-scans the epic body (2d) and closes only if every `<epic>-*` record has since been discharged **and** every body acceptance box is now ticked.

#### 4d-funnel. Surface the funnel un-freeze at epic close (foundation #864)

Fires when 4d-epic just transitioned the epic open→closed in *this* run **AND** the plan note carries a `## Funnel freeze` section recording a freeze this run (or a prior run of this plan) took at Step 1.7 that is **not yet resolved** (no resolution line appended). The freeze's standing obligation is that its un-freeze is **surfaced — never silent** — the moment the interference window closes:

- **Operator-present:** `AskUserQuestion` — **Re-enable the funnel now** (flip the schedule note `enabled: no` → `yes` via full-file `vault_write`, same contract as the Step 1.7 flip) vs **Leave frozen** (the operator re-enables later by hand: schedule note `enabled: yes`, plus re-arming any deploy-host LaunchAgent/cron wake they tore down manually). Append the chosen resolution as one line to the plan's `## Funnel freeze` section (`vault_append`), so the section reads as resolved.
- **Operator-absent:** do **not** auto re-enable — never resume autonomous spend with no operator present (the same fail-closed philosophy as the schedule gate). Append one `### open` entry to the pending-decisions surface (`Pipeline/pending decisions.md` vs the legacy `Context/pipeline - pending decisions.md` — target pinned by the append-target resolution rule, `claude/commands/check-in.md`; batch-at-ritual format; default taken: **leave frozen**) for `check-in` to review, and append a `left frozen (operator-absent epic close <date>)` line to the plan's `## Funnel freeze` section.
- **Runs that end with the freeze still in place** — including partial runs where the epic stays open and 4d-funnel never fires — MUST state it in the Step 6 summary (`funnel remains FROZEN — frozen by this run at Step 1.7`). The freeze may outlive the run, but never invisibly.

#### 4d-archive. Archive the done plan note into the repo (board ON + plan has `epic:`)

Fires on the **same trigger as 4d-retro** — only when 4d-epic just transitioned the epic open→closed in *this* run (the step-2 close above ran). It may sit just **before or after** 4d-retro; both fire on the identical open→closed transition. Skip entirely when the epic did not close in this run, when board integration is OFF, or when the plan has no `epic:` frontmatter.

This is the **"machines to git"** half of the vault/git boundary (epic #252). While the epic is open the plan note lives **only in the vault** `Plans/<note>.md` as the live, mutable cross-session ledger (status flips, item ticks). At epic close — when the plan has reached `status: done` and the work is sealed — build copies that done plan note into the repo so the **durable record rides git history** instead of depending on the un-versioned vault. The vault copy stays the live copy; the repo copy is an immutable snapshot.

**Mechanism — copy-to-repo-dir via the shared protected-main lander (#408).** The snapshot lands in the repo `Plans-archive/` directory so it rides normal **git history** (diffable, blame-able, survives vault loss) rather than living as a GitHub issue attachment (opaque, un-diffable, lost if the issue is deleted). Run the deterministic-spine script — do **not** inline a bare `git commit`:

   ```bash
   workflows/scripts/build/archive-plan.sh "$PLAN_SRC" "<epic>" "<repo-root>"
   ```

- **`$PLAN_SRC`** — the run's plan note, the vault path `Plans/<note>.md` this run executed (the same `Plans/<date> <project> - <title>` referenced in 4d-retro's Links). It is `status: done` by the time 4d fires (Step 5 set it, or it was already done on a resume).
- **`<epic>`** — the epic number whose close triggered this archive (for the commit message).
- **`<repo-root>`** — the repo build operates on (where `Plans-archive/` lives). The script copies `$PLAN_SRC` to `Plans-archive/$(basename "$PLAN_SRC")` there.

**Why a script, not an inline commit (the #408 fix).** A bare `git commit` to the orchestrator's local `main` can never reach `origin` on a protected branch — since the #330 merge-queue ruleset a direct push is rejected (GH013), the *same* strand #404 fixed for the session-transcript archive. `archive-plan.sh` drives the shared kernel `workflows/scripts/lib/land-on-protected-main.sh`, which lands the snapshot **durably and idempotently**: on a protected `main` it builds the commit in a throwaway worktree off `origin/main` (never touching the orchestrator's checkout), pushes the stable `chore/plan-archive` branch, opens/adopts one PR, and arms `gh pr merge --auto` so the queue lands it; on an unprotected branch or no remote it commits in place. Re-runs converge (stable branch + diff-against-origin short-circuit) — no duplicate snapshot, no empty commit.

**Handle the one status line the script prints (never treat any as a failure that aborts the run):**
- `plan-archived: <rev>` — the snapshot is on the default branch (pushed / no-remote / already on origin). Done.
- `plan-archive-pr-queued: <pr>` — `main` is protected; the snapshot rode a branch + PR that is **enqueued** and lands async. This is **success** — note it in the run summary as `plan archive PR #<pr> queued`. Unlike a `/tidy` transcript stub, the immutable snapshot has **no stub-deletion coupling**, so nothing is gated on the merge; a later epic-close run is a harmless no-op once it merges.
- `plan-archive-skipped: <why>` — not landed (not a repo, push rejected, etc.); non-fatal, log `<why>` and continue.

**Does NOT touch `/tidy`.** This is a mechanical build archive step, **not** a Live/Drain extraction rule — it adds no live/drain pair, so no `tidy.md` Step-3 backstop is required and none is added.

#### 4d-retro. Mint the per-epic retro tracker (board ON + plan has `epic:`)

**Mint the per-epic retro tracker.** Fires **only when 4d-epic just transitioned the epic open→closed in *this* run** — i.e. this run's merges / `[v]` verdict-captures closed the epic's last open child and the step-2 close above ran. It fires from **BOTH** the Step-4 merge-gate path and the spike kind-fork path — the trigger is identical: 4d-epic just transitioned the epic open→closed in this run. This closes the contract-decomposition loop: every epic build completes **mints** its own retro tracker — a cheap, durable record of the epic's build health, filed at close so the decomposition learnings get a durable home instead of depending on someone remembering to open one.

The tracker is a **mint, not a questionnaire**: it records *what happened* (merge friction + build-health signals) and hands the *judgment* — was the threshold right, did the seam hold — to the overlay `/retro` judge, which later picks up `retro-pending` trackers. The four decomposition-retro questions and the handoff-defect taxonomy that used to live here have **moved out of the kernel** into that judge's sixth axis; the mint deliberately carries no judgment questions.

**Gated by `RETRO_MINT_ENABLED`** (`workflows/scripts/build/build.config.sh`, already in scope from Step 0, default on): when it is **off** (`0` / empty), this entire step no-ops — no idempotency probe, no tracker, **zero footprint**. Skip entirely when `RETRO_MINT_ENABLED` is off, when board integration is OFF, or when the plan has no `epic:` frontmatter.

**Does NOT fire for:** an epic that was already closed **before** this run (4d-epic was a no-op); a **partial-coverage** plan whose epic stays open (4d-epic didn't close it — open-children > 0); or a triage-cull / by-hand close that happened **outside** build. The trigger is specifically *build drove this epic to completion in this run*, not "an epic is closed."

**DOES fire for the reconcile-driven close.** When Step 0.5 reconcile auto-fixes a *missed* 4d-epic close (an epic left open with zero open children), that close is still 4d-epic driving the epic open→closed in this run — so 4d-retro **should** run after it; the body-marker idempotency probe (step 1) safely no-ops if the retro was already filed on a prior run.

**Synchronous at epic-close — not deferred to `/tidy`.** Mint the tracker **the moment** 4d-epic closes the epic, in this same step — the mint is the *primary* filing, *not* a `/tidy` extraction rule. Rationale (the architecture-review alternative, rejected): a drain-time extraction would lose the retro if the session ends before the next drain; minting synchronously guarantees the tracker exists the instant the epic dies. `/tidy` still carries a **report-only** backstop for this mint (the registered Live/Drain pair — `tidy.md` Step 3 § `Retro mint backstop`), but it is a periodic *net* that only reports a missed mint, never the filing path.

1. **Idempotent — probe before create (body-marker, not title), corpus-rg-first.** Mirror Step 2.5's body-marker pattern (same helper, same resolve-once `$PROBE_LIB` seam — re-source it here if this is a fresh bash block): probe for the unique machine-readable marker line `Retro-for-epic: #<epic>` embedded in the retro issue **body**, so a resumed / re-run build never files a duplicate: `issue_marker_probe "$repo" "Retro-for-epic: #<epic>"` — searches the rendered issue corpus first (zero `gh` calls when fresh), falling back to the identical live `gh issue list -R "$repo" --search "Retro-for-epic: #<epic> in:body" --state all --json number` whenever the corpus is absent or stale-beyond-limit (or `PROBE_LIB` doesn't exist in this checkout — same graceful degrade as 2.5). A match (`jq -r '.[0].number // empty'` off the returned array) → already filed; do nothing. **Probe the body, not the title** — a `#<epic>` token inside an `in:title` search is parsed by GitHub as an issue-reference and can silently return empty even when the issue exists (→ a duplicate retro on every resume); `in:body` is literal text search, not a reference token, and this helper's corpus path never reads title at all (see issue-marker-probe.sh's header on the in:title trap), so the same protection holds on both the corpus and the gh-fallback path.
2. **Gather the mint's signals and decide its labels — no judgment, only what the run observed.** From this run's own record of the epic's build, collect:
   - **Merge friction** — conflicts encountered (count + which PRs) and each conflict's resolution-cost bucket (`trivial-inline` resolved in place / `rebase-respawn` needed a rebase round-trip / `real-rework` a contract or edge had to change); `0/none` if clean. (This is the content the `## Merge friction` body section renders.)
   - **Build-health signals**, each recorded as a `(value, gate-state)` pair — where gate-state is **`passed`** (the gate actually ran this epic and verified this value) or **`did-not-run`** (the gate never executed this run, so a clean value is *unverified*, not confirmed). This clean-verified vs clean-unverified distinction is load-bearing — never render a bare `0` that conflates "the gate ran and saw zero" with "the gate never ran":
     - **CI retries** — total CI-retry count across the epic's PRs; gate `did-not-run` on a spike-only epic (no `code` item ⇒ no PR ⇒ CI never ran).
     - **worker ejections** — count of workers ejected-and-respawned during the build; gate = worker execution (ran iff any item built).
     - **design forks** — count of items that returned a `design-fork` verdict this run.
     - **merge gate** — whether the epic's PRs actually cleared a merge gate (`passed`), or never reached one (`did-not-run`, e.g. a spike-only epic).
   - **State label** — resolve whether an overlay `/retro` judge is installed to pick this tracker up, via the shared `command_declared` predicate (`workflows/scripts/lib/command_declared.sh`; resolve-once like `$PROBE_LIB`: `CMD_DECL_LIB="$(git rev-parse --show-toplevel 2>/dev/null || echo .)/workflows/scripts/lib/command_declared.sh"; [ -f "$CMD_DECL_LIB" ] && source "$CMD_DECL_LIB"`). Then: **`retro-pending`** iff `command_declared retro` is true (a judge exists — the tracker is *pending* judgment); else **`retro-info`** (a **terminal** label — no judge is installed, so nothing will ever pick it up; the tracker stays a durable record only). If `command_declared` can't be sourced in this checkout, treat it as false ⇒ `retro-info`. **Never apply a `spike` label** — the mint is not a spike.
   - **Urgency** — the mint is *past the urgency bar* iff **any** worker ejection (≥1), **any** design fork (≥1), or **CI retries ≥ `RETRO_URGENT_CI_RETRIES`** (`build.config.sh`). Past the bar ⇒ the tracker **also** carries `retro-urgent`, in addition to its state label.
3. **Ensure the tracker's labels exist first, then file exactly one tracker into Backlog via the board `capture` entrypoint.** `capture.sh` passes a non-work-class `--label` value straight through to `gh issue create`, which **aborts with no issue created** if the label doesn't exist on the target repo — and a fresh-history kernel/adopter repo won't have the `retro-*` labels yet, so an un-provisioned mint would fail on every epic close with no tracker and no `Retro-for-epic:` marker, defeating the step-1 idempotency probe (it would find nothing and re-fail identically forever). Provision them idempotently first, using the same helper `capture.sh` uses for its own work-class labels — `_board_issues_ensure_label` is in scope from `$BOARD_LIB`; its inner `gh label create` is `|| true`, so this is a harmless no-op wherever the labels already exist:

   ```bash
   _board_issues_ensure_label "$repo" "retro-pending" "fbca04"
   _board_issues_ensure_label "$repo" "retro-info"    "c5def5"
   _board_issues_ensure_label "$repo" "retro-urgent"  "d93f0b"
   ```

   Then file the tracker (the same adapter front door 2.5/`capture.sh` use — lands in Backlog, no `--milestone`), carrying the state label from step 2:

   ```bash
   "$(dirname "$BOARD_LIB")/../capture.sh" "Process retro: epic #<epic>" \
     --body "<body>" --board "$BOARD" --label "<retro-pending|retro-info>"
   ```

   Then, **only when the mint is past the urgency bar** (step 2), add the urgency label to the issue `capture` just created: `gh issue edit <n> -R "$repo" --add-label retro-urgent` (`<n>` = the number `capture` printed). If that add-label call fails (a transient `gh` error), **do not silently drop the escalation** — the tracker still exists with its state label, but its urgency is lost and the step-1 probe will short-circuit ("already filed") on every later run, so nothing re-applies it: log the failure and fold `retro-urgent add FAILED on #<n>` into the Step 6 summary so a human can re-apply the one label, rather than leaving it invisibly missing. Both labels ride the same tracker.
4. **Tracker body** — the machine-readable marker (**unchanged** — the idempotency probe in step 1 keys on it), a one-line reader gloss carrying the honest state label, the merge-friction and build-health stamps, then the links. **No retro questions, no handoff-defect taxonomy** — the mint records *what happened*; the overlay `/retro` judge asks the decomposition questions later:

   ```markdown
   Retro-for-epic: #<epic>

   Process retro (mint) for epic #<epic> (closed by /build on <date>). State: <retro-pending|retro-info>.
   This is a durable **mint** — the build-health record for this epic, filed at close: it records what

### 4e-quota. 5-hour quota gate (after each level)

After Step 4 fully completes for this level (all merges done, 4d sentinels + board updated) and **before advancing to the next level**, check the 5-hour usage quota — so a long run pauses at this clean, state-consistent boundary instead of risking being cut off mid-merge in the next level. **Skip when no levels remain** (fall through to Step 5; there is nothing to protect). **Skip when `$quotaGate` is empty** (Step 0 found no gate script — a non-vendoring repo): the optional gate is a no-op, advance normally.

Run the decision script resolved in Step 0 (its config is already sourced, so the threshold is in scope):
```
"$quotaGate"
```
Branch on the one-line verdict JSON:
- **`{"action":"proceed",…}`** or **`{"action":"unavailable",…}`** → continue to **4e-keystone** (which then falls through to 4e Advance). **`unavailable` fails open** — a missing / stale / unparseable quota signal (or an account with no 5h window) never pauses a run.
- **`{"action":"pause","wait_secs":S,"reset_ts":T,"remaining_pct":N}`** → the remaining 5h quota is below `BUILD_QUOTA_PAUSE_PCT`. **Pause, then auto-resume in-session:**
  1. Record a one-line marker in the plan note's `## Run status` — `paused for 5h quota at <HH:MM>, remaining <N>%, resuming ~<reset HH:MM>` — so the pause reads as intentional, not a dead session.
  2. Launch the wait as a **background** Bash command — `sleep <S>` with `run_in_background: true`. Foreground `sleep` is blocked in the harness; the backgrounded process re-invokes this session when it exits. End the turn announcing the resume time.
  3. **On the wake re-invocation, re-run `quota-gate.sh`.** `proceed`/`unavailable` → continue to 4e-keystone (then Advance). Still `pause` (the window hadn't rolled, or a no-`reset_ts` short re-check) → repeat step 2. Then continue to 4e-keystone.

This gate is **path-agnostic** — it runs at the orchestrator's level boundary under **both** the conversational and `--workflow` paths (the orchestrator owns the boundary, and the merge gate, in both). All four knobs (`BUILD_QUOTA_PAUSE_PCT` / `_CACHE` / `_WAIT_BUFFER` / `_MAX_AGE`) live in `workflows/scripts/build/build.config.sh`.

### 4e-keystone. Keystone-spike review halt (temperloop#526)

**This gate is path-agnostic** — like 4e-quota, it runs at the orchestrator's level boundary under **both** the conversational and `--workflow` paths, and it is **plan-note-driven** (it re-reads `$1` and keys off sentinels, never an in-memory flag — so nothing has to cross the orchestrator↔workflow return boundary).

**Before advancing, resolve any keystone-spike halt.** Re-read the plan note. If the level **just processed** contains an item carrying `keystone: true` (§ plan-schema `keystone:`) that is now `[v]`, **and** non-terminal items (`[ ]`/`[~]`/`[m]`) remain elsewhere in the plan, the run **STOPS HERE instead of advancing**: a keystone verdict reshapes downstream contracts, so the operator reviews it before any dependent level builds. This is a `blocking-now` gate with **no safe default** — `--unattended` controls the run *start*, never a mid-run gate, so it does **not** bypass this halt (§ Operating principles: "`--unattended` controls the *start*, not the *gates*").

**Resume is self-limiting — no double-halt.** The check keys on the level *just processed this run*. On a resumed RUN 2 the keystone spike is **already `[v]` at run start**, so Step 1.4 excludes its (terminal) level from the level list — that level is never re-processed, the kind-fork never re-runs, and this check never re-fires for it. RUN 2 processes only the dependent levels, whose just-processed set contains no keystone `[v]`, so it advances/completes normally. Re-running `/build` **is** the operator's approval to proceed; there is no separate approval flag to flip. (If the verdict says the plan itself should change, the operator edits/abandons the plan rather than re-running.)

On the halt:
1. **Record the boundary in `## Run scope`** (plan-schema § `## Run scope`) via `vault_append`: `- RUN <n> (spike) · <ISO-ts> · keystone spike \`<slug>\` (level <k>) [v] — HALTED before levels <k+1>–<N>; verdict: <verdict-note wikilink> · review then re-run /build to proceed`.
2. **Leave `status: executing`** (do **not** demote to `done`) so a later run resumes at the next level — the parked-not-finished state. **Skip Step 5 entirely** for this stop (it is guarded on "all items reach terminal state", which is false here). **Leave the epic In Progress with its claim HELD** — do **not** run Step 5's epic park-back: the plan is mid-flight awaiting RUN 2, not finished, so this is the sanctioned "Claim held until Done" state (`claude/CLAUDE.kernel.md` § Task workflow), and RUN 2 resumes under the same claim. (`/tidy`'s stale-claim sweep is the backstop for a keystone halt the operator genuinely abandons.)
3. **Surface the halt by regime:**
   - **Operator-present:** print the keystone summary — the spike slug, the verdict-note path to review, and the held levels — and stop. No `AskUserQuestion` is needed (the operator's *action* is to review then re-run; there is nothing to answer inline).
   - **Operator-absent (`--unattended` / `FUNNEL_OPERATOR_ABSENT=1`), board ON + plan has `epic:`:** **park and post, never proceed.** Post a `decision`-labelled verdict-review issue on the plan's **epic** (not any item's issue), body naming the keystone spike, the verdict-note wikilink, and the held levels, with the instruction "review the verdict, then re-run `/build <plan>` to build the dependent levels" — assign the operator (same `$ASSIGNEE` resolution as the async-decision backend, § Operating principles), stop the tick. Add the posted issue number to the `## Run scope` line (`· decision issue: #N`). This mirrors the 4b risky-merge async-decision pattern — a `blocking-now` decision parks and reports; silence is never consent. **Do not** also write the pending-decisions surface: that surface is `batch-at-ritual`-only (safe-default, auto-proceed), and a no-default `blocking-now` block written there would mis-signal its urgency to `/check-in`; the labelled+assigned epic issue is the correct durable tracking surface (as the async-decision backend already is, and it writes no pending-decisions entry either).
   - **Operator-absent, board OFF or no `epic:` (no epic issue to post to):** fall back exactly as 4b does when the epic issue is unavailable — there is no async receptacle, so raise a **loud terminal halt**: the `## Run scope` line (step 1) is the durable record, and the run stops with the keystone summary printed prominently in the Step 6 summary. A later human re-run past the review is still required.
4. **When the run later resumes past this boundary** (RUN 2, the keystone spike already `[v]` at start and this level's dependents about to build), append the matching resume line to `## Run scope`: `- RUN <n+1> (build) · <ISO-ts> · resumed after operator review — building levels <k+1>–<N>`.

If the level just processed contains **no** keystone `[v]` (or no non-terminal work remains), fall through to Advance unchanged.

### 4e. Advance

If more levels remain: loop back to Step 3 with the next level (which now has its `depends-on` deps merged into main, so workers will see them in their base). A non-speculative worker spawned here branches its worktree (3b) from the **freshly-advanced** `origin/<default>` — its base is current by construction, so it skips the 3f step-0.5 currency check.

- **Reattach any in-flight speculative workers (opt-in optimization on).** If the orchestrator pre-spawned eligible items of this next level during *this* level's merge-wait (per "Speculative next-level execution"), those workers already ran 3a–3e against the pre-merge base. Do **not** re-spawn them — instead resume each at the **3f step-0.5 base-currency check**, now that this level has merged and advanced `origin/<default>`: `BASE_CURRENT` items promote straight to push-by-SHA; `BASE_STALE` items rebase-then-reverify (or discard-and-respawn). Items of the next level that were **not** speculation-eligible (a `depends-on` into the just-merged level) spawn fresh here as usual, on the current base.

If no more levels: fall through to Step 5. (Any speculative workers pre-spawned for a level that turns out not to exist were gated by eligibility against a real next level, so this case does not arise — speculation only pre-spawns items that are actually in the plan's level list.)

## Step 5 — Demote plan status

When all items reach terminal state, patch frontmatter:
- All items `[x]` or `[v]` (terminal-complete) → `status: done`
- Any `[m]` or `[-]` remaining → `status: executing` (next run can resume from the parked / skipped items)

**Epic closure is independent of plan status.** It is the data-driven 4d-epic check (open-children == 0), *not* a function of this plan reaching `done`. A plan that covers only some of its epic's children will correctly reach `status: done` while the **epic stays open** (other children remain) — the epic closes later when its real last child closes, via 4d-epic in this or another run. Never close an epic from Step 5.

**Epic park-back (board ON + plan has `epic:`) — the inverse of the run-start claim-stamp (3a-3).** *Closing* a drained epic is 4d's job; *parking* a still-open one is Step 5's, and it is what stops the epic being stranded In Progress under a now-ending run (the "ghost In-Progress epic" — observed on epics left In Progress for days under dead sessions, cleaned up by hand). After the plan-status patch above, if the epic is **still open**:
- **Park it iff this run owns it.** Read the epic's `Host/Session` via the adapter (`board_resolve_item "$BOARD" <epic#>`). If the stamp **equals this run's** stamp AND no child is in flight (this run reached terminal state, so none is), **move the epic → `Ready` and clear the stamp**: `eid="$(board_item_id <epic#>)"; board_set_status "$eid" "$BOARD_OPT_READY"; board_stamp "$eid" "$BOARD_FIELD_HOSTSESSION" ""`. **Check both calls' exit codes** — the same unchecked-return gap as the 3a stamp (foundation#1220): if either exits non-zero, **warn loudly naming the epic and the failure** (`WARNING: epic #<epic#> park-back failed: <error>`) and **name it in the Step 6 summary** (the Board line's `park-back:` slot) — a failed park-back is never silent; the epic stays In Progress under this run's stamp until `/tidy`'s stale-claim sweep (the backstop below) or a manual park clears it. This is `park-don't-abandon` applied to the epic — **not a close** (the epic re-enters In Progress + re-stamps when its next child is claimed, by this or a later run).
- **Never clear an epic stamped by a *different live* session** — that is a concurrent run legitimately owning it (the cross-session parallelism contract-scoped decomposition enables), not drift. Only un-own what this run stamped.
- **Backstop:** a stamp a *crashed* run never cleared is swept cross-session by `/tidy`'s stale-claim sweep (foundation #85) — Step 5's park-back only covers the clean end-of-run path.

## Failure modes

- **Plan note edited mid-run.** Re-read `$1` at the start of each item, not just once.
- **Worktree leaks.** `git worktree list` shows them. The orchestrator-created per-item worktrees live at the deterministic path `<repo>.wt/<item.slug>` on branch `build/<item.slug>`; `workflows/scripts/build/worktree.sh remove <repo-root> <slug>` reclaims one (worktree + branch + marker). 3d/3h do this per item; a crashed run can leave stragglers to sweep with `worktree.sh prune <repo-root>` (removes merged-into-`origin/<default>` worktrees only, skipping dirty ones unless `--force`; the path is re-derivable from each item's slug).
- **Branch already exists.** Handled in 3b — surface and ask user to pick alternative or use existing.
- **CI flake vs real failure.** If logs show no real failure, surface to user before re-spawning the worker.
- **Strict branch protection serializes the whole batch.** When `main` requires up-to-date branches (`required_status_checks.strict: true`), only the *first* PR at the gate merges with a plain `gh pr merge`; every other parked PR flips `MERGEABLE/CLEAN` → `MERGEABLE/BEHIND` and errors `Required status check "<ctx>" is expected (mergePullRequest)`. This is **not** a conflict — even fully disjoint-file PRs serialize this way; it's just slow. On a `NATIVE` repo the queue serializes and updates branches itself; on a `MANAGED` repo the 4a `strict` probe reading `STRICT` selects `gate.sh managed-merge --strict`, whose per-PR mechanics fold the `update-branch` + CI re-poll in before each merge (the scripted successor of the #83 hand-nudge — observed pre-script: a 7-PR level where only PR #1 merged until each was `update-branch`ed and re-CI'd, and a PR that sat `BEHIND` 25 min under `--auto` until nudged). Do **not** misroute `BEHIND` to 4c.
- **Cross-item merge conflict at batch gate.** A genuine `CONFLICTING`/`DIRTY` — distinct from the `BEHIND` case above — happens when items in a level touch the same file *and the hunks overlap* (git's 3-way merge auto-resolves disjoint hunks, so same-file ≠ conflict). Mitigation: (a) the plan's `## Sequencing notes` should flag likely collisions; (b) Step 4c handles `CONFLICTING` PRs explicitly (leave-open / skip / rebase); also an `update-branch` in 4b that surfaces a conflict escalates there. Authors who want zero risk should encode the ordering as `depends-on` so the items land in separate levels.
- **`mergeable: UNKNOWN` doesn't resolve.** Re-poll once after ~3s in 4a; if still unknown, treat as `CONFLICTING` and route through 4c.
- **`mergeStateStatus` reads `BEHIND` on an up-to-date branch.** GitHub lags right after a CI pass — a PR whose parent already is `main` HEAD can momentarily show `BEHIND`. Under the NATIVE `--auto` queue a *transient* stale read self-corrects once checks are green. A `BEHIND` that **persists** past the prior merge, though, is the real-stale-base case from #83 — on the MANAGED backend it is cleared mechanically by `gate.sh managed-merge --strict`'s built-in `update-branch` (or, one-off, `gate.sh nudge`) rather than hand-polling indefinitely.
- **User aborts mid-batch-gate.** Selected-but-not-yet-merged PRs stay `[m]`; non-selected stay `[m]`; the plan demotes to `status: executing` so the next session resumes at Step 1 with the parked set surfaced.
- **Worker can't satisfy an acceptance bullet.** Workers must return `done` only when all bullets pass. If genuinely unsatisfiable, worker should return `blocked` with the bullet as a question, not lie.
- **Worker ignores return contract.** No JSON block → treat as `failed: "no structured return"`. User decides whether to retry with a stricter prompt.
- **Board claim fails or is contended.** A non-zero claim (the `$CLAIM` entrypoint — missing `project` scope, issue not on the board, network) or a pre-check showing the issue already In Progress under a *different* `Host/Session` **halts that item** (retry/skip/abort) — build never silently works an unclaimed or someone-else's item. Fix scope with `gh auth refresh -s project`.
- **Issue creation must not double-run on resume.** Step 2.5 creates issues only for items *missing* `gh_issue:` and writes the number back into the plan note immediately, so a resumed run sees the existing `gh_issue:` and creates nothing twice. If a created issue never made it onto the board, the 3a claim fails (halt) rather than proceeding untracked.
- **`release.sh` is local-only, and the claim is held until Done.** It clears the tmux marker, not the board column. Parked `[m]` items legitimately stay In Progress until the merge gate moves merged ones to Done (4d); items left open at the gate stay In Progress by design (work isn't finished). The board only ever leaves In Progress via 4d (merge → Done) or a deliberate skip. This is the **claim-until-Done** sanctioned flow (K#275): the marker clear at 3h step 2 is optional and best-effort, so a `release.sh` **refusal on a non-latest claim in a multi-claim window** (`this window holds a claim for #B, not #A`) is expected — the earlier claim stays held and is cleared by the close→Done cascade / `reconcile.sh`, never by forcing a per-window marker that points elsewhere. Never fail a park on a release refusal.
- **Epic doesn't close though all children merged.** The 4d-epic close reads the live open-children count via the sub-issues API. If that API is unavailable, fall back to the epic-body task-list (all `- [x]`) as the close signal; if even that is ambiguous, leave the epic open and flag it in the Step 6 summary for a manual close. **Never close an epic while any child is still open** — that is the failure the data-driven count exists to prevent.
- **Split member must not share one issue across its items (#73).** When `/assess` splits one coarse sub-issue #N into several items, every item must end up with its **own** `gh_issue:` — never a copied `gh_issue: #N` on all of them. The mechanism: `/assess` leaves `gh_issue:` unset + stamps `split_from: #N`; Step 2.5 mints a fresh issue per split item (1:1 restored); Step 2.6 step 3 closes the coarse #N once all its leaves are minted. If items ever share a `gh_issue:`, the first PR's `Closes #N` (3f) closes the issue while the rest of the chain is unmerged and the board moves #N to Done prematurely — the exact failure this guards against (observed on stageFind epic #247: #258→3 items, #262→2, hand-worked around by minting #440-444).
- **Partial-coverage plan must not close the epic early.** A plan covering only a subset of an epic's children reaches `status: done` with the epic still open by design. The 4d-epic check (open-children == 0, not "plan done") is what prevents a premature close; Step 5 never closes the epic. Step 0.5 reconciles the inverse drift (an epic left open after its true last child closed out-of-band).
